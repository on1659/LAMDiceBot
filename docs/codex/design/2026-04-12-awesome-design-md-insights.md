# awesome-design-md 인사이트 정리

날짜: 2026-04-12
대상 저장소: `https://github.com/VoltAgent/awesome-design-md`

## 목적

`awesome-design-md`가 LAMDiceBot에 실질적인 도움이 되는지 확인하고, 바로 가져올 수 있는 운영 방식과 시각 레퍼런스를 분리해서 정리한다.

## 한 줄 결론

이 저장소의 핵심 가치는 특정 브랜드 스타일을 그대로 가져오는 것보다, 프로젝트 루트에 `DESIGN.md`를 두고 AI가 따라야 할 디자인 규칙을 한 문서로 고정하는 방식에 있다.

## 저장소에서 본 핵심 포인트

- `DESIGN.md`는 AI가 읽는 디자인 시스템 문서 역할을 한다.
- 문서 한 장에 색상, 타이포그래피, 컴포넌트, 레이아웃, 반응형, 금지 패턴, 프롬프트 가이드를 함께 담는다.
- 레퍼런스 사이트를 직접 복제하기보다 "이 프로젝트는 어떤 톤과 규칙을 가져야 하는가"를 고정하는 데 강하다.

## LAMDiceBot 기준으로 유효한 이유

현재 프로젝트는 이미 다음 자산을 가지고 있다.

- [css/theme.css](../../../css/theme.css)
- [docs/renewal/design-unification-summary.md](../../../docs/renewal/design-unification-summary.md)
- [index.html](../../../index.html)
- 각 게임별 HTML 파일과 `js/shared/*` 공용 UI 모듈

즉, 디자인 자산이 없는 상태가 아니라 규칙이 여러 파일에 흩어져 있는 상태에 가깝다. 그래서 외부 스타일을 새로 들여오는 것보다, 기존 규칙을 `DESIGN.md` 형식으로 재정리하는 편이 효과가 크다.

## 바로 적용할 만한 인사이트

### 1. `DESIGN.md` 도입

가장 먼저 시도할 만한 작업이다.

포함하면 좋은 항목:

- 공용 색상 토큰과 게임별 accent 규칙
- 버튼, 상태 배지, 패널, 랭킹, 튜토리얼 오버레이 규칙
- 메인 랜딩과 게임 화면의 역할 차이
- 문서/가이드 페이지의 읽기 우선 규칙
- "하지 말아야 할 것" 목록

예시 금지 규칙:

- 게임 화면에 게임별 accent를 무시한 임의의 새 메인 컬러를 추가하지 않기
- 공용 상태 UI에 각 화면마다 다른 의미 색을 다시 정의하지 않기
- 문서 화면에 게임 화면 수준의 강한 그라디언트와 유리 질감을 남용하지 않기

### 2. 화면군별 톤 분리

현재 프로젝트는 한 서비스 안에 성격이 다른 화면이 섞여 있다.

- 메인 랜딩: 첫인상, 진입 CTA, 브랜드 인지
- 게임 화면: 빠른 이해, 상태 전달, 실시간 상호작용
- 문서/가이드: 긴 글 가독성, SEO, 검색 유입 대응
- 운영/통계 화면: 데이터 밀도, 스캔 속도

`awesome-design-md` 방식의 장점은 이 차이를 문서화해서 AI가 임의로 섞지 않게 만든다는 점이다.

### 3. Do / Don't 명문화

외부 레퍼런스 문서들에서 가장 유용했던 부분이다.

LAMDiceBot용으로 정리하면 좋을 항목:

- Do: 게임별 대표색은 유지하되 공용 컴포넌트 구조는 통일하기
- Do: 메인 랜딩은 감정선이 있어도 게임 화면은 정보 우선으로 유지하기
- Do: 가이드 페이지는 본문 가독성을 최우선으로 두기
- Don't: devtool 스타일 다크 테마를 게임 전체에 일괄 적용하기
- Don't: CTA, 상태, 경고 색 의미를 화면마다 바꾸기
- Don't: SEO 페이지에 과도한 애니메이션을 넣기

## 참고하기 좋은 레퍼런스

### 1. Notion / Mintlify 계열

추천 대상:

- `pages/*.html`
- `docs/GameGuide/*`
- FAQ, 가이드, 정책, 분석 문서형 화면

가져올 포인트:

- 긴 글 기준의 안정적인 여백
- 얇은 경계선과 읽기 우선 타이포
- 과하지 않은 강조색 사용

주의:

- 게임 랜딩과 같은 수준의 시각적 에너지는 부족할 수 있다.

### 2. PostHog 계열

추천 대상:

- 통계, 관리자, 랭킹, 데이터가 많은 패널형 UI

가져올 포인트:

- 밀도 높은 정보 배치
- 너무 딱딱하지 않은 개발자 친화 톤
- 운영 화면에도 개성을 남기는 방법

주의:

- 그대로 들여오면 지금의 밝은 게임 경험과 톤 차이가 커질 수 있다.

### 3. Framer / Vercel 계열

추천 대상:

- 메인 랜딩의 히어로 구성
- 카드 진입 애니메이션
- CTA 우선순위 설계

가져올 포인트:

- 헤드라인 밀도 조절
- 카드와 버튼의 리듬감
- 제품 소개 구간의 시각적 집중도

주의:

- 전체 devtool 다크 테마화는 LAMDiceBot과 맞지 않을 가능성이 높다.
- 부분 차용이 적합하다.

## 우리 프로젝트에 대한 판단

현재 기준으로는 특정 레퍼런스 하나를 채택하는 것보다 다음 순서가 더 적절하다.

1. 기존 `theme.css`와 디자인 통합 문서를 바탕으로 `DESIGN.md` 초안 작성
2. 메인 랜딩, 게임 화면, 문서 화면, 운영 화면을 서로 다른 화면군으로 정의
3. 각 화면군마다 허용되는 색상, 타이포, 모션 강도를 적어두기
4. 이후 필요한 경우 개별 레퍼런스 스타일을 부분 차용

## 추천 액션

- `DESIGN.md` 초안 만들기
- 메인 랜딩 전용 규칙과 게임 화면 전용 규칙 분리
- 문서형 페이지용 타이포와 간격 규칙 별도 정의
- `index.html`과 각 게임 HTML이 공용 규칙을 어디까지 공유해야 하는지 명시

## 초안에 넣으면 좋은 항목

- 브랜드 개요
- 공용 색상 토큰
- 게임별 색상 역할
- 타이포 규칙
- 버튼, 상태, 패널, 카드 규칙
- 모션 원칙
- 반응형 원칙
- Do / Don't
- AI용 작업 지침 예시

## 메모

이 정리는 "어떤 사이트가 예쁘냐"보다 "우리 프로젝트가 AI와 함께 작업해도 일관성을 잃지 않게 하려면 무엇이 필요한가"에 초점을 둔다.

따라서 다음 단계는 레퍼런스 수집 확대보다 LAMDiceBot 전용 `DESIGN.md` 초안 작성이 더 적합하다.
