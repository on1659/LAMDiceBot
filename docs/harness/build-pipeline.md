# /build 파이프라인 설계

> 코드 구현 파이프라인: Coder → Reviewer → QA → Main 최종 확인

---

## 흐름도

```
사용자: /build "주문 시스템에 자동완성 추가해줘"
  │
  ▼
┌─────────────────────────────────────────┐
│  Stage 1: Main Orchestrator             │
│  참조: skill-pd.md, guidelines.md       │
│        skill-ui.md, skill-ux.md          │
│                                         │
│  - 요구사항 분석                          │
│  - 영향 범위 파악 (어떤 게임/파일에 영향?) │
│  - 화면 대응 계획 수립:                   │
│    ☐ 모바일(375px) 레이아웃 어떻게 배치?  │
│    ☐ 태블릿(768px) 레이아웃 변화점?       │
│    ☐ 데스크톱(1920px) 기본 레이아웃?      │
│    ☐ 터치/마우스 인터랙션 차이가 있는가?   │
│    ☐ 모바일에서 생략/축소할 요소가 있는가? │
│  - 작업 계획 수립 (수정 파일, 순서)        │
│  - Quality Gate 기준 설정                │
│  - Coder에게 지시서 전달                  │
│    (모바일/PC 레이아웃 명세 포함)          │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│  Stage 2: Coder Agent                   │
│  참조: skill-backend.md, skill-frontend.md │
│        rules/backend.md, rules/frontend.md │
│  실행: worktree 격리 환경                  │
│                                         │
│  - 테스트 먼저 작성 (TDD — hook 강제)     │
│  - 모바일 퍼스트로 구현                    │
│    ☐ CSS: 모바일 기본 → @media로 확장     │
│    ☐ 터치 + 마우스 이벤트 동시 대응        │
│    ☐ 뷰포트별 레이아웃 Main 지시서 따름    │
│  - 계획대로 코드 구현                     │
│  - 기존 패턴 준수                         │
│  - 변경 사항 diff 반환                    │
│                                         │
│  [Hook] tdd-guard.sh → 테스트 없으면 차단  │
│  [Hook] security-guard.sh → 보안 위반 차단 │
│  [Hook] fairness-guard.sh → 난수 위반 차단 │
│  [Hook] css-var-guard.sh → 색상 하드코딩 차단│
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│  Stage 3: Reviewer Agent                │
│  참조: skill-backend.md, skill-frontend.md │
│        skill-ui.md, skill-ux.md          │
│        rules/guidelines.md               │
│                                         │
│  - 코드 품질 리뷰                        │
│  - 패턴 준수 확인                         │
│    ☐ Socket: checkRateLimit() 호출       │
│    ☐ DB: 파라미터화 쿼리                  │
│    ☐ CSS: 변수 사용                      │
│    ☐ API: 상대 경로                      │
│  - UI/UX 관점 확인                       │
│    ☐ 게임별 시각 아이덴티티 일관성         │
│    ☐ 사용자 플로우 자연스러움             │
│    ☐ 접근성 기준 충족                    │
│  - 모바일 UI/UX 검증                     │
│    ☐ 반응형 브레이크포인트 동작 (~480px, ~768px) │
│    ☐ 터치 타겟 최소 44x44px              │
│    ☐ 소프트 키보드 시 레이아웃 깨짐 없음   │
│    ☐ 가로/세로 모드 양쪽 대응             │
│    ☐ 모바일 Safari/Chrome 호환성         │
│    ☐ 게임 애니메이션 모바일 성능           │
│  - 판정: approve / request-changes       │
│                                         │
│  request-changes → Coder로 루프백 (최대 3회)│
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│  Stage 4: QA Agent                      │
│  참조: skill-qa.md, skill-backend.md     │
│        rules/guidelines.md               │
│                                         │
│  - Tier 1: Happy Path 테스트 실행        │
│  - Tier 2: 엣지케이스 검증               │

│    ☐ 동시 참여 경쟁 조건                 │
│    ☐ Rate Limit 초과                    │
│  - 공정성 검증                           │

│    ☐ 클라이언트 조작 경로 차단 확인       │
│  - 멀티플레이어 동기화 검증               │
│    ☐ 상태 일관성                        │
│    ☐ 재연결 시 복원                     │
│  - 모바일 환경 테스트                     │
│    ☐ 모바일 브라우저 Socket.IO 연결 안정성 │
│    ☐ 백그라운드 전환 후 복귀 시 상태 복원  │
│    ☐ 터치 이벤트로 게임 조작 정상 동작     │
│    ☐ 모바일 네트워크 전환(Wi-Fi↔LTE) 시 재연결│

│      (375x667, 667x375, 430x932,         │
│       768x1024, 1920x1080)               │
│    ☐ 모바일 게임 플로우 실행 (클릭/입력)   │
│    ☐ 게임별 애니메이션 모바일 렌더링 확인   │
│    ☐ WebKit(Safari) 엔진 크로스 브라우저   │
│  - 판정: pass / fail                    │
│                                         │
│  fail → Coder로 루프백 (최대 3회)         │
└──────────────────┬──────────────────────┘
                   ▼
┌─────────────────────────────────────────┐
│  Stage 5: Main Orchestrator (최종)       │
│  참조: skill-pd.md                       │
│                                         │
│  - 전체 결과 종합                        │
│  - Quality Gate 확인                     │
│    ☐ 기존 게임에 영향 없는가             │
│    ☐ Rate Limiting 적용됐는가            │
│    ☐ 모바일/데스크톱 양쪽 동작하는가      │
│    ☐ 공정성 유지되는가                   │
│  - 배포 Gate 확인                        │
│    ☐ 테스트 통과                        │
│    ☐ DB 하위 호환성                     │
│    ☐ AdSense 유지                      │
│  - 변경 요약 작성                        │
│  - 사용자에게 보고                       │
└──────────────────┬──────────────────────┘
                   ▼
사용자: 변경 내용 확인 → 승인 / 수정 요청
```

---

## 루프백 정책

| 단계 | 조건 | 동작 | 최대 횟수 |
|------|------|------|----------|
| Reviewer → Coder | request-changes | 리뷰 피드백을 Coder에게 전달, 수정 후 재리뷰 | 3회 |
| QA → Coder | fail | 실패 원인을 Coder에게 전달, 수정 후 재검증 | 3회 |
| 최대 횟수 초과 | 3회 실패 | Main이 현재까지 결과 + 미해결 이슈를 사용자에게 보고 | - |

---

## Coder Agent 실행 환경

```
실행 방식: Agent tool (isolation: worktree)

- git worktree로 격리된 복사본에서 작업
- main 브랜치에 영향 없음
- 구현 완료 후 diff만 반환
- 최종 승인 시 Main이 머지 결정
```

## Stage별 출력 형식

### Main 지시서 출력 (→ Coder에게 전달)
```
## 작업 지시서
- **요청**: (원래 요청)
- **수정 대상 파일**: (경로 나열)
- **화면 대응 명세**:
  - 모바일(375px): (레이아웃, 숨김 요소, 터치 인터랙션)
  - 태블릿(768px): (레이아웃 변화점)
  - 데스크톱(1920px): (기본 레이아웃)
  - 인터랙션 차이: (터치 vs 마우스)
- **구현 순서**: (파일별 작업 순서)
- **Quality Gate**: (통과 기준)
```

### Coder 출력
```
## 구현 완료
- **수정 파일**: (경로 나열)
- **추가 파일**: (경로 나열)
- **테스트 파일**: (경로 나열)
- **화면 대응 결과**:
  - 모바일: (구현 방식 — CSS 미디어쿼리, 레이아웃)
  - 데스크톱: (구현 방식)
  - 터치/마우스: (이벤트 처리 방식)
- **변경 요약**: (무엇을 왜 바꿨는지)
```

### Reviewer 출력
```
## 코드 리뷰
- **판정**: approve / request-changes
- **품질**: (패턴 준수 여부)
- **보안**: (체크리스트 통과 여부)
- **UI/UX**: (일관성, 접근성)
- **반응형**: (모바일/PC 대응 코드 적절한가)
- **수정 요청**: (있을 경우 구체적 지적)
```

### QA 출력
```
## QA 검증
- **판정**: pass / fail
- **테스트 결과**: (Tier별 통과 여부)
- **공정성**: (검증 결과)
- **엣지케이스**: (발견된 이슈)
- **회귀 리스크**: (영향받는 기능)
```

### Main 최종 출력
```
## 빌드 완료 보고
- **요청**: (원래 요청)
- **변경 요약**: (무엇이 바뀌었는지)
- **수정 파일**: (전체 목록)
- **리뷰 결과**: (Reviewer 판정)
- **QA 결과**: (QA 판정)
- **배포 리스크**: (main 푸시 안전 여부)
- **다음 단계**: (사용자가 해야 할 것)
```
