01 한눈에 보기 — 결론

긴 토론의 착지점부터 먼저.

한 줄 결론 세 후보 Ⓐ 4.8 단독 · Ⓑ 4.7+하네스 · Ⓒ 4.8+하네스 중 정답은 Ⓒ다. 단 "Ⓑ에서 모델만 4.8로 갈아끼우기"가 아니라 "4.8 + 재보정된 하네스" — 하네스는 비용 절약 장치가 아니라 "위험 라우팅 + 멈춤 장치"라 모델이 좋아져도 사라지지 않지만, 4.8의 셀프리뷰 개선으로 STANDARD 의식은 중복이 되므로 실행 의식은 줄이고 검증은 "가정 오류 위험"에 비례하게 재배분한다.

02 핵심 비교: 세 후보

4.8 단독 · 4.7+하네스 · 4.8+하네스 — 두 축(모델 품질 × 하네스 유무)이 만드는 세 점.

Ⓐ 4.8 단독 (쌩)

// 좋은 모델, 규율 없음

  • 의식 0 → 가장 빠른 루프
  • 판단력·지시준수 최상
  • 1M 컨텍스트 통째 보유
  • 토큰 최소
  • 독립검증·도메인 가드·안전레일 없음
단순 작업엔 최적 · 위험 작업엔 무방비

Ⓑ 4.7 + 하네스 (구 세팅)

// 규율 풀세트, 약한 모델

  • 독립 리뷰/QA = 가정 오류 검출
  • 결정론적 훅 = 모델 무관 강제
  • 공정성·크로스게임 가드
  • 약한 모델 → 의식이 "필수"라 무거움
  • 셀프리뷰 약해 독립리뷰 의존 큼
안전하지만 느리고 비쌈 (의식이 정당)

Ⓒ 4.8 + 하네스 (권장)

// 좋은 모델 + 규율

  • Ⓐ의 속도·판단 + Ⓑ의 안전·가드

3자 비교가 드러내는 두 가지 ① Ⓐ vs Ⓑ는 거짓 양자택일 — 우위 항목이 거의 안 겹친다(모델 성능 vs 정책·안전). 다른 층이라 합칠 수 있다 = Ⓒ.
② Ⓑ → Ⓒ의 함정 — 모델만 4.8로 올리고 하네스를 그대로 두면, 4.8의 개선된 셀프리뷰 때문에 STANDARD의 독립리뷰 의식이 중복(낭비)이 된다. 그래서 Ⓒ는 "하네스 유지"가 아니라 "하네스 재보정"이어야 한다.
결론 Ⓒ 4.8 + (재보정된) 하네스. Ⓐ의 속도·판단을 SIMPLE/STANDARD에서 살리고, Ⓑ의 독립검증·도메인 가드를 COMPLEX-계약에 집중한다. "버리기"도 "그대로 두기"도 아닌 재배분이다. 구체적 다이얼은 15장 참조.

03 모델 ID 감사 결과

"하네스가 4.7 기준인데 4.8에서 괜찮나?"의 실측 확인.

결론 에이전트 frontmatter(scout/coder/reviewer/qa)와 settings.json에 모델 ID 하드코딩 0건. 서브에이전트는 전부 메인 모델(4.8) 상속. 하네스 동작이 특정 버전에 묶여 깨질 위험 없음.

발견 — 깨지진 않지만 옛 버전 박힌 곳 (수정 완료)

파일전후
.claude/commands/meeting-multi.mdClaude Sonnet 4.5Claude Opus (버전 제거)
.claude/commands/summitdocs.mdClaude Opus 4.5Claude Opus (버전 제거)
왜 버전을 지웠나 Co-Authored-By는 git에 박히는 순수 텍스트 — 자동으로 버전을 못 찾는다. Opus 4.5는 틀리고 다음 버전에 또 stale. Claude Opus(버전 없음)는 영원히 안 틀린다. 덜 구체적일 뿐.

문제 아님 — 등급 이름만 쓰는 곳

나머지 Opus/Sonnet 참조는 전부 버전 숫자 없는 등급 이름이라 현재 모델로 자동 해석됨. (meeting-codex, meeting-team, docs.md 등) skill-*.md의 ### mid (4-7년차)는 경력 표기 — 오탐.

04 트리아지 3단계란?

하네스의 핵심 — 요청을 받으면 먼저 실행 수준을 판정한다.

단계조건동작
SIMPLE1~2파일, UI/공정성/DB/소켓 무관직접 수정 (훅 가드만 동작)
STANDARDUI 소규모 또는 3파일 미만Scout → Coder → Reviewer
COMPLEX3파일+, 새 기능, DB/소켓 변경, 공정성 영향+ QA

매 턴 첫 줄을 [트리아지: …]로 강제하는 UserPromptSubmit 훅과, Edit/Write 직전에 형식을 재검증하는 check-triage.sh가 이 시스템을 떠받친다.

05 셀프리뷰 vs 독립리뷰 — 토론의 개념적 핵심

"4.8 셀프리뷰가 좋아졌는데 독립리뷰가 왜 필요해?"에 대한 답.

"셀프리뷰"라는 한 단어가 두 가지 다른 오류를 섞고 있다. 쪼개면 답이 깔끔해진다.

✓ 실행 오류 (Execution)

  • 오타, off-by-one
  • null 체크 누락
  • 엣지케이스 빠뜨림
  • API 시그니처 불일치

셀프리뷰로 잘 잡힘. 4.8에서 확실히 개선.

✗ 가정 오류 (Assumption)

  • 소켓 이벤트 계약 오해
  • "updateRoomsList가 전체 브로드캐스트한다" 식 잘못된 멘탈모델
  • 공유모듈 동작 가정 틀림

셀프리뷰로 구조적으로 약함. 모델 개선과 무관.

왜 가정 오류는 똑똑해져도 안 잡히나 같은 컨텍스트에서 검토하면 코드 짤 때 세운 잘못된 전제를 그대로 들고 본다. 전제가 틀린 걸 의심하려면 그 추론을 안 본 신선한 눈이 필요한데, 이건 IQ가 아니라 정보 격리 문제라 모델 성능으로 안 닫힌다.

06 4.8 셀프리뷰 개선의 진실

개선은 진짜다. 단, 어디에 작동하느냐가 중요.

오류 종류셀프리뷰로 잡힘?4.8에서 개선?
실행 오류✓ 잘 잡음✓ 확실히 좋아짐
가정 오류✗ 약함⚠ 별로 안 변함

그래서 계산이 이렇게 바뀐다. 셀프리뷰가 좋아질수록 독립리뷰와의 격차는 좁아진다. 하지만 닫히지는 않는다:

  • STANDARD: 격차가 충분히 좁아 셀프리뷰로 대체 OK
  • COMPLEX: 남은 격차가 작아도 × 비대칭 리스크(실서버 공정성/desync)라 여전히 정당화. "90%→93%"의 3%가 실서버 버그면 비싸다

07 교차모델(Codex)의 독립 가치

셀프리뷰 개선과 완전히 무관한 축.

사각지대는 IQ로 못 메운다 4.8이 아무리 자기검토를 잘해도 4.8 훈련의 사각지대는 공유한다. Codex 교차리뷰는 그 사각지대 밖에서 보는 것이라, 4.8 셀프리뷰가 100점이어도 가치가 남는다. 같은 4.8 두 번보다 4.8 + Codex가 잡는 게 다르다. → 4.8 시대에 오히려 가치 상승.

08 Dynamic Workflows vs 커스텀 하네스

"이미 Workflows 있는데 이걸로 가는 게 낫지 않나?"

둘은 같은 게 아니라 다른 층이다.

커스텀 하네스 (.claude/)Dynamic Workflows (네이티브)
정체정책/규율 층 — 언제 에스컬레이트, 불변조건, 도메인실행/오케스트레이션 엔진 — 팬아웃·반증·스케일
발동매 턴 자동 (훅 강제)명시적 opt-in, 백그라운드, 토큰 다수
강점가볍고 항상 켜짐, 공정성·크로스게임병렬·adversarial verify·재개·구조화 출력
착지 하네스는 "두뇌(정책·불변조건·훅)"로 남기고, COMPLEX-계약의 "실행 근육"만 네이티브 Workflows로 갈아끼운다. 새 오케스트레이션 코드를 재구현하지 말고, 트리아지가 네이티브 도구를 호출만 하게. 하네스는 더 얇아지고 실행은 더 강해진다.
Workflows가 대체 못 하는 것 ① 트리아지 결정 자체 (도구일 뿐, 매 턴 발동하는 정책 아님) · ② 결정론적 훅 (셸 강제) · ③ SIMPLE/STANDARD의 가벼운 규율 (2파일에 워크플로우는 과함)

천장(품질 상한)은 진짜 최고다 — 부정 금지. "여러 워크플로우 순차 + 적대적 검증"은 틀리면 재앙인 작업에서 최고 확신을 준다. 문제는 언제나 쓰는 게 낭비라는 것.

켜는 순간 — 행사용 편성

  • 대형 기능 출시 직전 하드닝 (새 게임 추가 등)
  • 주기적 심층 감사 (공정성/소켓 전수 점검, /cso)
  • 위험한 마이그레이션 (DB 스키마, 공유모듈 대수술)
  • "이번 건 절대 틀리면 안 됨" 핵심 재작성

11 비용 무관 세계 재평가

"토큰 비용 신경 안 쓴다면?" — 가장 큰 반대 논거를 제거하면.

솔직히 인정: 비용이 반대 논리의 상당 부분을 떠받치고 있었다. 빼면 다이얼이 워크플로우 쪽으로 이동한다. 하지만 "전부 워크플로우"가 정답이 되진 않는다 — 구속 조건이 다른 걸로 바뀔 뿐:

#비용 빼도 남는 제약요지
1지연시간 ≠ 비용2파일 수정이 직접 20초 vs 워크플로우 8분이면, 공짜여도 개발 루프가 느려짐
2인간 주의력이 진짜 희소자원잡을 게 없어도 N개 차원의 "이슈 0건" 리포트를 사람이 훑어야 함
3코드베이스 철학과 충돌 ⚠️guidelines.md의 "최소·외과적"과 울트라코드의 "최대 철저"가 정면 충돌. 컴퓨트↑ → 골드플레이팅 유혹↑
4실서버는 검토가능성이 확신사람이 따라갈 수 있는 외과적 변경이, 쏟아진 대형 변경보다 신뢰하기 쉬움
반전 — 항상-오케스트레이션이 골드플레이팅을 더 잘 막을 수도 제약 ③이 함정이지만, 워크플로우 안에 "surgical-change 위반 비평가"를 한 차원으로 넣으면 "요청 범위를 넘었나? 안 망가진 걸 리팩터했나?"를 자동 검사할 수 있다. 게이트 모델에선 사람이 해야 하는 일이다.
비용 무관 시 다이얼 이동 변화는 천장이 아니라 바닥(언제 발동)에서 일어난다. STANDARD·COMPLEX-실행이 워크플로우로 상향. 단 SIMPLE 직접은 유지 — 한계효용이 진짜 0이라 비용과 무관.

12 단계를 없애도 되나?

"그럼 SIMPLE/STANDARD/COMPLEX 없애는 거?"

아니다. 트리아지는 비용 절약 장치가 아니라 두 가지를 한 라벨에 묶고 있다:

축무엇비용 빠지면?
① 검증 강도에이전트 몇 개, 얼마나 철저히"항상 최대" — 이 축은 무너짐
② 작업 형태직접 vs 오케스트레이션, 외과적 vs 전수그대로 남음 — 비용 무관

비용 질문은 ①만 때린다. ②는 지연·주의력·골드플레이팅·검토가능성이 떠받치므로 살아있다.

폐지가 아니라 통합 비용이 STANDARD와 COMPLEX-실행을 갈라주던 것이므로, 빠지면 둘이 합쳐진다. 3단계 → 2단계(직접 / 오케스트레이션)로 줄이는 게 정직한 귀결. 0단계(완전 폐지)는 잘못.

완전 폐지가 나쁜 구체적 이유

  • 결정론적 훅이 트리아지 선언 위에 지어져 있음 → 단계 개념을 지우면 훅 강제 레이어를 통째로 재작성
  • 트리아지 한 줄 = 공짜 "멈춤 장치". 토큰 0원인데 "어, 이거 공정성 건드리네"를 사전에 잡음

13 모델 X vs 모델 Y — 진짜 크럭스

"몇 단계냐"가 아니라 "라우팅 결정이 어디 사는가"가 본질.

모델 X — 게이트 (현재)

// 작업 앞에서 사람이 선언

  • 싸고 투명함
  • 멈춤 장치가 사람 눈에 보임
  • 사람 오버라이드 가능
  • 단점: 분류를 정보 적을 때 미리 해야 함
VS

모델 Y — 자가-트리아지

// 오케스트레이터가 내부 분기

  • 사소 → short-circuit 인라인
  • 위험 → 다차원 + 반증
  • 계약 → + 교차모델 + guidelines 비평가
  • 단점: 진입 지연, 멈춤 장치 숨음, 훅 재작성
왜 중요한가 모델 Y면 "전부 워크플로우"가 말이 된다 — 사소한 건 워크플로우가 알아서 short-circuit하니까. 즉 "SIMPLE은 직접"이 사라지는 게 아니라 오케스트레이터 안으로 들어가는 것. "완전 폐지"가 비용 없는 세계에선 생각보다 덜 미친 선택 — 단 폐지가 아니라 라우팅 이전이다.

14 정확도는 어디가 높은가

"X vs Y, 정확도는?"

핵심 정확도의 천장은 같다. 같은 작업에 같은 검증 깊이를 적용하면 출력 정확도는 동일. 정확도는 아키텍처가 아니라 "검증 깊이"가 결정한다. 진짜 차이는 라우팅 오류(놓침) 하나뿐.
분류 시점정보량놓침
X (게이트)작업 시작 전적음 (아직 안 파봄)"SIMPLE인 줄 알았는데 소켓 계약" 통과 ⚠
Y (자가)빠른 스캔 후많음 (훑고 결정)도중 위험 감지 시 자동 에스컬레이트

→ Y가 "겉보기 사소하지만 실은 계약-위험"을 덜 놓친다. 그게 바로 가정 오류 클래스. 이 한 가지로 Y의 실효 정확도가 살짝 높다.

단 — 조건부 Y의 자가-분류기가 잘 만들어져야 함. 못 만들면 위험한 걸 short-circuit 해 X보다 나빠질 수 있다. X는 사람 오버라이드가 있고, Y는 그걸 일부 제거한다.
작업 종류정확도 차이 (추정)
계약-위험 작업Y가 놓침 ~몇%p 적음 (자동 에스컬레이션)
사소한 작업둘 다 4.8 직접 ~97~98%, 검증 더 부어도 98~99% → 차이 거의 0

15 최종 다이얼 권고

비용 신경 여부에 따라 두 버전.

다이얼 A — 비용 신경 O (권장 기본)

단계처리
SIMPLE→ 쌩 4.8 (직접) · 항상
STANDARD→ 4.8 + 셀프리뷰 1패스
COMPLEX-실행→ 4.8 + QA

울트라코드→ 전부 최대 · 출시/감사/대수술 때만 토글

다이얼 B — 비용 신경 X

단계처리 (바닥이 상향됨)
SIMPLE→ 직접 · 그대로 (지연·노이즈 때문에 공짜여도 직접)
STANDARD→ + 경량 워크플로우 (병렬 셀프 + verify 1패스)
COMPLEX-실행→ + 워크플로우 (다차원+반증)
COMPLEX-계약→ 울트라코드급 (순차 다중 워크플로우 + 교차모델)

가정-위험 트리거 (단계 무관, 닿으면 자동 승격)

  • socket/*의 emit/on 이벤트 계약 변경
  • 공정성 경로 (서버 결과 결정, 클라 Math.random)
  • js/shared/*, utils/room-helpers.js 등 크로스게임 공유 모듈
  • DB 스키마 / db/* 쓰기 경로

절대 안 건드리는 것 (frozen — 모델 좋아져도 가치 불변)

  • 결정론적 훅 (트리아지/Edit 가드)
  • 공정성 가드 (클라 Math.random 0회 검증)
  • impl 문서 강제 (3파일+/DB)
  • MEMORY.md 기록 규율
  • main = 실서버 안전레일

16 미결 질문

결정을 끝내려면 답해야 할 것.

  1. 모델 X vs Y 중 어느 쪽인가? (단계 개수는 그다음 문제) — 현재 "더 논의" 상태
  2. "비용 안 씀"이 진심인가, 사고실험인가? 이게 X/Y를 가르는 결정타
  3. Y의 자가-분류기를 신뢰할 만큼 잘 짤 수 있나? 이 코드베이스에서 "계약-위험" 자동 판별 난이도가 X/Y를 거의 끝냄
  4. 실측: 현재 워킹트리 변경(socket/* db/* js/*)으로 순차 vs 워크플로우 리뷰를 1회씩 돌려 잡은 이슈·시간·토큰 직접 비교할 것인가?
