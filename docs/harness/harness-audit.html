하네스 감사 — Harness Audit

LAMDiceBot의 이더(Ether) 하네스를 6가지 룰로 점검하는 슬래시 커맨드. /harness-audit 명령으로 수동 실행, SessionStart 카운터로 10세션마다 자동 알림.
v1.0 2026-05-04 분리형(A) 구조 Memory-Audit과 짝

개요

하네스 폴더(.claude/)는 시간이 지나면서 stale worktree, 끊긴 참조, 안 쓰이는 에이전트, stale lock 파일이 쌓인다. 하네스 감사는 이런 누적을 6가지 룰로 자동 검출하고, 발견 즉시 삭제하지 않고 보고서로 제시해 사용자가 결정하게 한다.

Memory-Audit과의 관계
하네스 감사와 메모리 감사는 분리된 슬래시 커맨드다 (구조 옵션 A). 룰의 종류가 다르기 때문 — 메모리는 stale/중복/secret, 하네스는 worktree/orphan/reference. SessionStart 카운터(.claude/hooks/audit-counter.sh)는 둘 모두에 안내 메시지를 보낸다.

6가지 룰

1
Stale Worktree
정리 우선순위 높음
.claude/worktrees/에 남은 옛 에이전트 worktree를 검출하고, dirty 여부를 보고한다.
무엇
git worktree list의 외부 worktree 중 main 브랜치 외 모든 항목.
왜
에이전트 작업이 끝난 뒤 worktree를 정리하지 않으면 디스크와 git 메타데이터에 누적된다. dirty 상태로 남으면 어떤 작업이었는지 헷갈려서 자료 손실의 위험이 있다.
어떻게
# 1) 모든 worktree 나열
git worktree list

# 2) 외부 worktree 각각의 dirty 검사
for wt in .claude/worktrees/*; do
  cd "$wt" && git status --porcelain
done
매칭 시
  • Clean worktree → 즉시 git worktree remove <path>
  • Dirty worktree → 사용자 보고 후 결정 (보존 / stash / force remove)
2026-05-04 발견 사례
7개 worktree 중 3개 clean(자동 제거 완료), 4개 dirty: agent-aa9e925a (자동 주문 잔재 추정), agent-acdb6a3a (경마 스프라이트), bold-shirley-a7d227 (공정성 테스트), mystifying-wu (.claude/agents/ 수정). dirty 4개는 보존 후 사용자 판단으로 처리.
2
Stale Lock
긴급
.claude/*.lock 파일이 release 안 된 채 남아있으면 다른 세션의 작업이 막힌다.
무엇
.claude/scheduled_tasks.lock 등 .lock 확장자 파일. JSON 본문에 acquiredAt 타임스탬프 있는 경우 경과 시간 비교.
왜
세션이 비정상 종료되면 락이 해제되지 않는다. 24시간 이상 지속되면 거의 확실히 stale. 새 세션이 같은 자원을 요구할 때 충돌하거나 무한 대기에 빠질 수 있다.
어떻게
# lock 파일에서 acquiredAt 추출 후 24h 비교
acquired=$(jq -r .acquiredAt .claude/scheduled_tasks.lock)
now=$(date +%s%3N)
diff=$(( now - acquired ))
[ $diff -gt 86400000 ] && echo "STALE"
매칭 시
  • 24h+ stale → 자동 삭제 안전 (단, pid가 살아있으면 보존)
  • 1h~24h → 보고만, 사용자 판단
2026-05-04 발견 사례
scheduled_tasks.lock — 8일 전(2026-04-26) 획득된 락이 release 안 됨. 정리 완료.
3
Orphan Files (고아 파일)
우선순위 낮음
agents/, rules/, skills/, commands/에 있지만 어디서도 참조 안 되는 .md 파일.
무엇
파일명을 추출 후 다른 모든 마크다운 파일과 코드 파일에서 grep. 0회 매칭이면 고아.
왜
하네스를 리팩토링하다 보면 더 안 쓰이는 에이전트나 룰 파일이 남는다. 자동으로 삭제하면 위험하지만, 후보 목록은 유용하다.
어떻게
for f in .claude/agents/*.md .claude/rules/*.md; do
  name=$(basename "$f" .md)
  refs=$(grep -rln "$name\.md\|$name'" \
    --include="*.md" .claude/ docs/ CLAUDE.md \
    | grep -v "$f" | wc -l)
  [ $refs -eq 0 ] && echo "ORPHAN: $f"
done
매칭 시
자동 삭제 금지. 사용자에게 후보 목록만 제시. 일부는 의도적으로 자료/참고용으로 남겨둔 것일 수 있다.
4
Broken References (참조 깨짐)
긴급
CLAUDE.md, harness.md, workflow.md가 가리키는 파일이 실제로 존재하는가.
무엇
마크다운 링크 [text](path) 또는 코드 인용 `path/file.md`의 모든 경로. 로컬 상대 경로만 검사 (외부 URL 제외).
왜
파일 이동/이름 변경 후 참조가 끊기면 다음 세션에서 Claude가 잘못된 경로를 안내하거나 파일을 못 찾는다. 하네스 신뢰성 직접 영향.
어떻게
# 마크다운 링크에서 경로 추출 후 존재 검증
grep -oE '\]\([^)]+\.md\)' CLAUDE.md \
  .claude/rules/*.md \
  | sed 's/.*(\(.*\))/\1/' \
  | while read p; do
      # 템플릿 placeholder({...}) 제외
      [[ "$p" =~ \{.*\} ]] && continue
      [ ! -f "$p" ] && echo "BROKEN: $p"
    done
매칭 시
  • 파일 이름 비슷한 게 있음 → rename 후보 제시
  • 완전히 사라짐 → 참조 줄을 제거하거나 새 경로로 갱신
5
Hook 무결성
검증
settings.json에 등록된 hook 스크립트가 실제로 존재하고 실행 권한이 있는가.
무엇
.claude/settings.json의 hooks.*.hooks[].command 필드에서 $CLAUDE_PROJECT_DIR/.claude/hooks/*.sh 패턴 추출.
왜
hook이 등록만 되어 있고 파일이 없거나 chmod +x 안 되어 있으면 매 도구 호출마다 조용히 실패. 가드(공정성 검사 등)가 작동 안 하는데 사용자는 모름.
어떻게
jq -r '.hooks[][].hooks[].command' \
  .claude/settings.json \
  | grep -oE '\.claude/hooks/[^"]+\.sh' \
  | sort -u \
  | while read h; do
      [ ! -x "$h" ] && echo "BROKEN HOOK: $h"
    done
매칭 시
  • 파일 없음 → settings.json에서 등록 제거 또는 hook 작성
  • 실행 권한 없음 → chmod +x
6
Stale Agent
우선순위 낮음
.claude/agents/*.md가 60일 이상 수정 안 된 경우. (rules/, skills/는 안정적이라 stale OK)
무엇
에이전트 정의 파일의 마지막 수정 시각이 60일 이상 지난 것.
왜
에이전트 정의는 워크플로우와 함께 진화한다. 60일+ 변경 없는 에이전트는 실제로 호출되고 있지 않거나, 호출되더라도 다른 에이전트와 룰이 어긋날 가능성. 룰 파일은 의도적으로 안정적이라 stale 대상 아님.
어떻게
find .claude/agents -name "*.md" \
  -mtime +60 \
  -exec stat -c "%y %n" {} \;
매칭 시
자동 삭제 금지. 후보 목록 + 마지막 수정일 보고만. 사용자가 "아직 쓰는가?" 판단.

실행 흐름

수동 실행: /harness-audit
  1. 6개 룰을 순차 적용 (병렬 가능한 것은 병렬)
  2. 발견된 후보를 표 형식 보고서로 제시
  3. 각 후보에 권장 액션 명시 (제거 / 보존 / 보고만 / 수동 검토)
  4. 사용자 승인 → 일괄 처리. 자동 삭제 금지.
  5. 처리 후 before/after 통계 보고
자동 알림: SessionStart 카운터

.claude/hooks/audit-counter.sh가 매 세션 시작 시 카운터를 +1한다. 10의 배수에 도달하면 알림 메시지를 inject: 

🔔 감사 주기 도달 (세션 N회).
   /memory-audit + /harness-audit 점검을 권장합니다.

카운터 위치: ~/.claude/projects/d--Work-LAMDiceBot/memory/.audit-counter

안전 가드

자동 삭제 금지 — 모든 destructive 액션은 사용자 승인 필요
룰 1(dirty worktree), 3(orphan), 6(stale agent)은 보고만 한다. 룰 1(clean worktree)과 2(24h+ lock)만 자동 처리 안전. 룰 4(broken ref), 5(hook 무결성)은 자동 수정이 의도와 다를 수 있어 사용자 확인 필수.
Memory-Audit과 충돌 없음
하네스 감사는 .claude/만 본다. 메모리 폴더(~/.claude/projects/.../memory/)는 건드리지 않는다. 반대로 /memory-audit은 메모리만 보고 .claude/는 건드리지 않는다.
