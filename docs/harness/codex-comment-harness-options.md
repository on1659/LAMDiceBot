# Codex 코멘트 — 하네스 최소/맥시멈 버전 제안

> 대상 문서: `docs/harness/*`
>
> 작성일: 2026-04-13

---

## 목적

현재 `docs/harness` 문서는 방향이 좋고 구조도 잘 잡혀 있다.  
다만 지금 저장소 상태에서 바로 돌릴 수 있는 버전과, 이상적으로 확장한 버전을 분리해서 보는 편이 현실적이다.

이 문서는 두 가지 버전을 제안한다.

1. **최소 하네스 버전**
   - 지금 저장소에 바로 붙일 수 있는 현실형
   - 과도한 차단을 피하고, 실제 팀 생산성을 우선
2. **맥시멈 하네스 버전**
   - 문서가 현재 지향하는 완성형
   - 멀티에이전트 + Hook 강제 + 모바일 QA까지 포함

---

## 현재 문서 총평

좋은 점:

- `/build`와 `/meeting` 목적이 명확하게 분리되어 있다
- 역할별 참조 스킬/규칙 매핑이 구체적이다
- 모바일/PC 대응을 초반부터 파이프라인에 포함한 점이 좋다
- Hook를 단순 가이드가 아니라 **게이트**로 보려는 방향이 맞다

주의할 점:

- 문서가 약속하는 “물리적 차단”과 실제 Hook 등록 범위가 아직 맞지 않는다
- 테스트 강제를 지금 수준으로 바로 걸면 일상 작업이 많이 막힐 가능성이 높다
- `Math.random()` 차단 범위가 너무 넓어 UI 연출 코드까지 막을 수 있다
- `/meeting`의 형식 강제 Hook는 설계는 되어 있지만 실제 등록 예시는 빠져 있다

---

## 버전 1: 최소 하네스

### 목표

“지금 바로 굴릴 수 있는 버전”을 만든다.  
핵심은 **차단보다 일관된 작업 흐름 확보**다.

### 권장 범위

- `/build` 파이프라인만 먼저 운영
- `/meeting`은 문서 템플릿 수준으로만 운영
- Hook는 **block 최소화 + warn 중심**
- 모바일 QA는 필수지만 브라우저 자동화는 선택적 실행

### 포함 요소

#### 1. Stage 구성

`/build`는 아래 3단계만 사용:

1. **이더(Ether)**  
   - 요청 분류
   - SIMPLE / STANDARD / COMPLEX 판단
   - 수정 파일 / 영향 범위 / 모바일 영향 정리

2. **Coder**
   - 실제 구현
   - 기존 규칙/패턴 준수
   - 테스트 추가 가능하면 같이 추가

3. **Reviewer**
   - 패턴 준수
   - UI/UX / 모바일 리스크
   - 회귀 위험 체크

필요 시만 QA 추가:

- 새 기능
- Socket 이벤트 추가
- DB 스키마 변경
- 게임 결과 로직 변경
- 모바일 레이아웃 큰 변경

#### 2. Hook 구성

최소 버전에서는 아래만 권장:

- `security-guard.sh`
  - Socket 핸들러에 `checkRateLimit()` 누락 시 **warn**
- `css-var-guard.sh`
  - 색상 하드코딩 감지 시 **warn**
- `mobile-guard.sh`
  - viewport / 고정 폭 / 작은 터치 타겟 감지 시 **warn**

#### 3. Hook에서 당장 빼는 것

- `tdd-guard.sh`의 **block**
  - 현재 테스트 자산이 충분하지 않아 실무를 과하게 막을 가능성이 높음
- `fairness-guard.sh`의 전역 `Math.random()` 차단
  - UI 연출, 코멘트, 장식 효과와 게임 결과 랜덤을 구분 못함
- `format-guard.sh`
  - `/meeting` 운영을 정착시킨 뒤 2차로 붙이는 편이 현실적

### 최소 버전의 실제 차단 기준

block는 딱 아래만 추천:

- 클라이언트 코드에서 **게임 결과를 결정하는 난수**를 직접 생성한 흔적
- 서버 난수/결과 계산 로직을 클라이언트로 옮기는 변경
- 민감 정보 하드코딩

나머지는 모두 warn으로 시작하는 편이 좋다.

### 최소 버전 운영 방식

#### SIMPLE

- 이더가 직접 처리
- Hook warn만 확인

#### STANDARD

- 이더 → Coder → Reviewer

#### COMPLEX

- 이더 → Coder → Reviewer → QA

### 최소 버전 장점

- 지금 저장소 구조와 충돌이 적다
- 기존 테스트 부족 문제를 우회할 수 있다
- 팀이 하네스에 적응하기 쉽다
- 문서만 있고 안 쓰이는 설계가 되지 않을 가능성이 높다

### 최소 버전 한계

- 강제력은 약하다
- `/meeting`의 형식 일관성은 아직 문서 의존
- 품질 게이트가 “차단”보다 “리마인더”에 가깝다

## 버전 2: 맥시멈 하네스

### 목표

현재 `docs/harness` 문서가 지향하는 완성형에 가깝게 간다.  
핵심은 **멀티에이전트 분업 + Hook 강제 + 품질 문턱 자동화**다.

### 포함 요소

#### 1. `/build` 풀 파이프라인

1. 이더
2. Coder
3. Reviewer
4. QA
5. 이더 최종 게이트

#### 2. `/meeting` 풀 파이프라인

1. 이더 진행
2. Phase 1 기획 검토 병렬
3. Phase 2 구현 검토 병렬
4. Phase 3 디자인 검토 병렬
5. Phase 4 QA 검토
6. Phase 5 PD 최종 판단

#### 3. Hook 풀 세트

- `tdd-guard.sh`
- `security-guard.sh`
- `fairness-guard.sh`
- `css-var-guard.sh`
- `mobile-guard.sh`
- `format-guard.sh`

#### 4. 모바일 QA 자동화

- Playwright MCP 필수화
- 기본 뷰포트 5종 점검
- WebKit(Safari) 포함
- 게임 플로우 스모크 테스트 자동화

### 맥시멈 버전이 성립하려면 필요한 선행조건

#### 1. 테스트 자산 확충

최소한 아래 수준은 필요:

- `socket/*.js` 핵심 이벤트별 테스트
- `routes/*.js` 주요 API 스모크 테스트
- `db/*.js` 핵심 쿼리/폴백 경로 테스트
- 게임별 Happy Path 자동화

#### 2. 공정성 규칙 정교화

`Math.random()` 자체를 막는 게 아니라 아래처럼 분리 필요:

- 허용:
  - UI 파티클
  - 장식 효과
  - 코멘트 랜덤 노출
  - 시각적 변주
- 차단:
  - 승패 결정
  - 순위 결정
  - 서버 판정을 대체하는 랜덤

#### 3. Hook 대상 도구 확장

문서가 말하는 “물리적 차단”을 구현하려면 `Write`만으로 부족하다.

실제 편집 경로 기준으로 아래까지 고려 필요:

- `Write`
- `Edit`
- `MultiEdit`
- `apply_patch` 계열
- 필요 시 shell 기반 파일 수정도 감시

#### 4. 역할별 출력 형식 표준화

`/meeting`은 각 역할이 정해진 템플릿으로 답하도록 맞춰야 한다.

예:

- 현우: `대상 사용자`, `MoSCoW`, `성공 지표`
- 소연: `KPI`, `ICE`
- 태준: `수정 파일`, `DB 영향`, `Socket 영향`
- 미래: `수정 파일`, `CSS 변수`, `모바일 대응`
- 윤서: `리스크 등급`, `공정성`, `DoD`
- 지민: `Go/No-Go`, `범위`, `배포 리스크`

### 맥시멈 버전 장점

- 팀 작업 표준화가 강하게 걸린다
- 품질 편차를 줄이기 좋다
- 회귀/보안/공정성 리스크를 조기에 차단한다
- 멀티에이전트 협업 품질이 올라간다

### 맥시멈 버전 한계

- 초기 구축 비용이 크다
- 지금 저장소 테스트 체계로는 바로 적용하기 어렵다
- Hook가 과도하게 엄격하면 생산성 역효과가 날 수 있다

## 추천 결론

현재 LAMDiceBot에는 **최소 하네스 → 맥시멈 하네스 순차 도입**이 맞다.

### 지금 당장 추천하는 운영안

#### Phase A. 최소 하네스 적용

- `/build` 우선 운영
- Hook 3종 warn 중심
  - security
  - css-var
  - mobile
- COMPLEX 작업에만 QA 붙이기

#### Phase B. 테스트 확충

- `socket`, `routes`, `db` 핵심 테스트 보강
- Playwright 스모크 시나리오 축적

#### Phase C. 강제력 확대

- `tdd-guard`를 일부 경로에 block로 승격
- `fairness-guard`를 결과 로직 한정 차단으로 정교화
- `/meeting`의 `format-guard` 실제 등록

#### Phase D. 맥시멈 하네스 전환

- 풀 멀티에이전트 운영
- `/meeting` 형식 강제
- 모바일/WebKit 자동 QA 상시화

## Codex 추천 액션 아이템

### 1순위

- `docs/harness` 문서 안에 “최소 하네스 운영안”을 정식으로 반영
- Hook 설명에서 `block`와 `warn`의 현실 기준을 분리

### 2순위

- `fairness-guard`를 “게임 결과 랜덤 차단” 중심으로 재설계
- `tdd-guard`를 전역 block 대신 단계적 도입으로 수정

### 3순위

- `/meeting` 템플릿 실사용 후 `format-guard` 붙이기
- Playwright MCP를 `/build` QA의 정식 체크리스트로 고정

---

## 한 줄 결론

지금은 **최소 하네스**가 맞고, 맥시멈 하네스는 **테스트 자산과 Hook 정교화가 끝난 뒤** 올리는 것이 가장 안전하다.
