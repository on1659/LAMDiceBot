# Codex 코멘트 — 하네스 문서 다듬는 방향

> 주의: 이 문서는 기존 `docs/harness/*` 내용을 직접 수정하지 않고,
> 앞으로 **어떤 방향으로 다듬어야 하는지**만 정리한 메모다.
>
> 작성일: 2026-04-13

---

## 목적

현재 `docs/harness` 문서는 구조와 아이디어가 좋다.  
다만 지금 상태에서 바로 “문서를 다시 쓰는 것”보다, 먼저 **어떤 순서와 기준으로 다듬을지**를 합의하는 편이 안전하다.

이 문서는 그 방향만 제안한다.

---

## 기본 원칙

### 1. 문서의 약속과 실제 동작을 먼저 맞춘다

가장 먼저 손봐야 할 기준은 “멋진 설계”가 아니라 **문서가 약속하는 것과 실제 구현/운영 가능성이 맞는가**다.

예:

- 문서에 “물리적 차단”이라고 쓰면 실제 Hook도 그 수준이어야 함
- 문서에 “테스트 없으면 block”이라고 쓰면 실제 테스트 자산이 그 전제를 받아야 함
- 문서에 “모바일 QA 자동화”라고 쓰면 실제로 어떤 도구와 어떤 시나리오로 하는지 있어야 함

### 2. 완성형보다 운영 가능한 버전을 먼저 고정한다

지금은 하네스의 이상형을 확장하는 것보다, **현재 저장소에서 실제로 굴릴 수 있는 운영 버전**을 먼저 정하는 것이 중요하다.

### 3. 차단 규칙은 최소한으로 시작한다

초기 하네스는 팀 생산성을 해치지 않는 방향이 우선이다.  
처음부터 block를 많이 두면 문서는 화려해도 실제로는 다들 우회하게 된다.

### 4. `/build`와 `/meeting`을 같은 속도로 발전시키지 않는다

두 파이프라인의 성격이 다르므로, 다듬는 우선순위도 분리하는 편이 좋다.

- `/build`: 실제 구현 파이프라인이라 운영 우선
- `/meeting`: 형식 표준화 문서라 2차 정교화 대상

## 다듬는 방향 요약

### 방향 A. “이상형 설계”에서 “운영 버전 정의”로 이동

현재 문서는 완성형 설명이 강하다.  
앞으로는 각 문서에 아래 구분이 더 명확해지는 쪽이 좋다.

- 지금 바로 적용 가능한 것
- 준비가 더 필요한 것
- 장기 목표로 남길 것

### 방향 B. Hook 설명을 “block / warn / future”로 분리

지금 Hook 문서는 전반적으로 강한 규칙을 전제한다.  
앞으로는 Hook마다 아래 셋 중 어디에 속하는지 나눠 적는 방향이 좋다.

- **block**: 지금 당장 강제 가능한 것
- **warn**: 아직 강제하면 과한 것
- **future**: 테스트 자산/도구 정비 후 올릴 것

### 방향 C. 테스트 현실을 반영한 TDD 규칙으로 낮춘다

현재 프로젝트 테스트 자산은 일부만 갖춰져 있으므로,  
`tdd-guard`는 “전역 강제”보다 “핵심 경로 우선 적용” 방향으로 설명이 정리되어야 한다.

### 방향 D. 공정성 규칙을 더 정교하게 쪼갠다

`Math.random()` 전체 차단 같은 넓은 규칙보다는,
**게임 결과를 결정하는 랜덤**만 막는 방향으로 정의가 좁혀져야 한다.

### 방향 E. 모바일 QA는 체크리스트와 자동화를 분리한다

모바일 대응은 중요하지만, 모든 걸 자동화 전제로 적으면 부담이 커진다.  
그래서 앞으로는 아래 두 층으로 나누는 방향이 좋다.

- 수동 체크리스트
- Playwright/WebKit 자동화

## 문서별 다듬는 방향

### 1. `README.md`

다듬는 방향:

- 이 하네스의 현재 상태를 한 줄로 명시
  - 예: `설계 단계`, `부분 운영 가능`, `완성형 아님`
- “목표 구조”와 “현재 운영 구조”를 구분
- 최소 하네스 / 확장 하네스 링크를 분리

핵심 목표:

- 처음 읽는 사람이 “지금 당장 뭘 쓸 수 있는지” 바로 이해하도록 만들기

### 2. `build-pipeline.md`

다듬는 방향:

- 지금 당장 쓸 수 있는 `/build` 흐름을 먼저 명시
- SIMPLE / STANDARD / COMPLEX 분류를 더 운영 중심으로 단순화
- QA를 붙이는 조건을 더 현실적으로 적기
- Hook 의존 부분은 “실제 강제”와 “권장 확인”을 나눠 적기

핵심 목표:

- `/build`가 문서상 파이프라인이 아니라 실제 실행 루틴처럼 보이게 만들기

### 3. `meeting-pipeline.md`

다듬는 방향:

- 현재는 이상적인 회의 구조가 잘 적혀 있으므로, 바로 축소하지 말고
  **실사용 버전과 완성형 버전**을 나누는 쪽으로 다듬는 것이 좋다
- Phase별 필수 산출물만 먼저 고정
- `format-guard` 없이도 사용할 수 있는 템플릿 버전을 먼저 정리

핵심 목표:

- `/meeting`이 “좋은 문서”에서 “실제로 반복 가능한 회의 포맷”으로 내려오게 만들기

### 4. `hooks-spec.md`

다듬는 방향:

- 가장 먼저 손봐야 할 문서
- Hook마다 아래 항목을 추가하는 방향이 좋다
  - 현재 적용 가능 여부
  - 권장 기본 모드 (`warn`/`block`)
  - 과차단 위험
  - 선행조건
- `matcher: Write`만 있는 예시 같은 부분은
  “현재 예시”와 “실전 적용 시 고려사항”을 분리해서 적기

핵심 목표:

- 문서상 강제력과 실제 작동 가능성을 맞추기

### 5. `agent-mapping.md`

다듬는 방향:

- 지금도 잘 되어 있으므로 큰 구조를 바꿀 필요는 없다
- 다만 “어떤 단계에서 어떤 파일을 반드시 읽는지”를 더 구체화하면 좋다
- `/build`와 `/meeting`의 참조 우선순위를 분리해 적는 방향이 좋다

핵심 목표:

- 에이전트별 역할 경계를 더 명확히 하기

## 우선순위 로드맵

### 1단계

- `hooks-spec.md` 현실화 방향 정리
- `/build` 최소 운영 흐름 명확화
- `README.md`에 현재 상태/목표 상태 구분 추가

### 2단계

- `/meeting` 실사용 템플릿 정리
- `format-guard`는 아직 문서상 권장으로만 두고, 강제는 나중으로 미룸
- 모바일 QA 체크리스트와 자동화 항목 분리

### 3단계

- 테스트 자산이 충분해진 뒤 `tdd-guard`를 점진 강화
- 공정성 Hook를 결과 로직 기준으로 정교화
- `/meeting` 형식 강제 Hook 도입

## 하지 말아야 할 다듬기

### 1. 처음부터 문서 전체를 축소하지 않는다

현재 문서는 완성형 설계로서 가치가 있다.  
바로 단순화만 하면 장기 비전이 사라질 수 있다.

### 2. Hook를 전부 block로 바꾸지 않는다

지금 저장소는 테스트와 자동화가 아직 그 수준을 받쳐주지 못한다.

### 3. `/meeting` 형식 강제를 먼저 붙이지 않는다

실제 회의 포맷이 몇 번 굴러본 뒤에 붙이는 것이 더 안전하다.

### 4. 공정성 규칙을 UI 코드까지 모두 잡지 않는다

연출용 랜덤과 승패 결정 랜덤을 분리하는 방향이 필요하다.

## 추천 결론

기존 `docs/harness`를 당장 갈아엎는 대신, 아래 방향으로 다듬는 것이 좋다.

1. **Hook 현실화**
2. **`/build` 최소 운영 버전 고정**
3. **`/meeting` 템플릿 운영 후 정교화**
4. **테스트 자산 확충 후 강제력 확대**
