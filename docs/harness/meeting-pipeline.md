# /meeting 파이프라인 설계

> 팀 회의 파이프라인: 기획 → 구현 검토 → 디자인 검토 → QA → PD 최종 판단

---

## Phase 병렬 실행 전략

같은 Phase 내 에이전트는 독립적이므로 **병렬 실행**하여 속도를 높인다.

```
Phase 1: 현우 ──┐
                ├─ 동시 실행 → 결과 수집 → Phase 2로
        소연 ──┘

Phase 2: 태준 ──┐
                ├─ 동시 실행 → 결과 수집 → Phase 3로
        미래 ──┘

Phase 3: 다은 ──┐
                ├─ 동시 실행 → 결과 수집 → Phase 4로
        승호 ──┘

Phase 4: 윤서 ── 단독 실행 (Phase 2 결과 참조) → Phase 5로

Phase 5: 지민 ── 단독 실행 (Phase 1~4 전체 참조) → 회의록
```

## Hook: 의견 형식 강제

각 에이전트의 출력이 해당 역할의 **의견 형식**을 포함하는지 검증한다.

| 에이전트 | 필수 포함 항목 |
|---------|--------------|
| 현우 (리서치) | 대상 사용자, MoSCoW, 성공 지표 |
| 소연 (전략) | KPI 연결, ICE 점수, 수익 영향 |
| 태준 (BE) | 수정 파일, DB 영향, Socket 영향, 예상 공수 |
| 미래 (FE) | 수정 파일, CSS 변수, 모바일 대응, 예상 공수 |
| 다은 (UI) | CSS 변수, 레이아웃, 상태 피드백, **모바일 UI 검증** |
| 승호 (UX) | 사용자 플로우, 마찰 포인트, 접근성, **모바일 UX 검증** |
| 윤서 (QA) | 리스크 등급, 테스트 시나리오, 공정성 영향, **Playwright 테스트 제안**, DoD |
| 지민 (PD) | Go/No-Go, 범위 판단, 배포 리스크, 담당자 |

형식 누락 시 해당 에이전트를 재실행한다.

## /build 와의 연계

회의에서 Go 판정이 난 안건은 `/build`로 바로 실행할 수 있다.

```
/meeting "경마 베팅 기능"
  → Go 판정 + 액션 아이템 확정

/build "경마 베팅 기능 — 회의 결정사항 기반 구현"
  → 이더가 회의록 참조하여 Coder에게 지시
```
