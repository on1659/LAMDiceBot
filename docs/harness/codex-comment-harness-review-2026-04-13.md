# Codex Review: Harness Design (2026-04-13)

> `docs/harness/` 멀티에이전트 오케스트레이션 하네스 설계 문서 리뷰

---

## 결론

전체 방향은 좋다.

- Scout 추가로 `추측 기반 지시`에서 `정찰 기반 지시`로 바뀐 점이 특히 좋아졌다.
- Hook 운영 등급을 `block / warn / future`로 나눈 판단도 현재 저장소 현실과 잘 맞는다.
- 페르소나 제거, 행동 지시 중심 전환, 모바일/PC를 계획 단계부터 포함한 점도 적절하다.

다만 지금 상태는 **구조 설계는 충분히 설득력 있지만, 세부 문서 동기화가 아직 덜 끝난 상태**로 보는 것이 맞다.

핵심 보완 포인트는 3가지다.

1. Hook 예시와 운영 등급 설명을 완전히 일치시킬 것
2. Playwright 시나리오를 현재 프로젝트 런타임에 맞출 것
3. Scout 보고서에 "깨지면 안 되는 계약" 항목을 추가할 것

---

## 주요 발견 사항

### 1. Hook 강제 범위를 아직 과하게 약속하고 있음

`README.md`에서는 하네스가 "매 tool call마다 실행"되어 우회가 없다고 설명하고 있다.

하지만 `hooks-spec.md`의 현재 설정 예시는 `matcher: "Write"`만 등록하는 형태다.

즉 현재 문서만 기준으로 구현하면:

- 모든 수정 경로를 막는 것은 아님
- 특정 쓰기 경로만 대상으로 삼는 구현이 될 가능성이 큼

이 지점은 설계 자체의 문제라기보다, **README의 보장 문구가 실제 예시보다 앞서가고 있는 상태**다.

### 2. Hook 등급 분리는 반영됐지만 세부 예시는 아직 일부 예전 상태가 남아 있음

현재 방향은 아래처럼 정리된 것이 맞다.

- `security-guard`: `block`
- `fairness-guard`, `css-var-guard`, `mobile-guard`: `warn`
- `tdd-guard`, `format-guard`: `future`

하지만 세부 문서에는 아직 옛 표현이 남아 있다.

- `build-pipeline.md`에서는 `tdd-guard`, `fairness-guard`, `css-var-guard`가 지금 차단되는 것처럼 읽힌다.
- `hooks-spec.md`의 `security-guard` 샘플은 설명상 `block`인데 실제 예시 반환은 `allow`다.
- `Hook 테스트 방법` 예시에서는 `fairness`와 `security`의 기대 결과가 서로 뒤집혀 있다.

즉 설계 방향은 맞지만, **운영 등급에 대한 세부 예시 정리가 아직 완전히 끝나지 않았다.**

### 3. Playwright 시나리오가 현재 앱 런타임과 맞지 않음

`playwright-mcp.md`의 예시는 `http://localhost:5173/...` 기준으로 작성되어 있다.

하지만 현재 저장소의 실제 런타임은:

- `npm start`
- `node server.js`
- Express 라우트 기반 진입 (`/game`, `/roulette`, `/horse-race`, `/admin`)

즉 지금 문서를 그대로 따라 구현하면 가장 먼저 부딪힐 문제는 **테스트 대상 URL과 서버 기동 전제가 현재 앱 구조와 안 맞는다는 점**이다.

### 4. Scout 핸드오프는 좋아졌지만 정보 손실 가능성은 아직 남아 있음

현재 Scout는 아래를 잘 잡고 있다.

- 수정 대상 파일
- 참조 파일
- 기존 패턴
- 의존성
- 영향 범위
- Socket 이벤트 관계

여기까지는 충분히 좋아졌다.

다만 아직 별도 항목으로 강제되지 않은 것이 있다.

- 깨지면 안 되는 라우트
- 유지해야 하는 이벤트 이름
- 기존 UX에서 바뀌면 안 되는 상호작용
- 운영/배포 계약
- SEO/리다이렉트/AdSense 관련 불변조건

이런 것들은 Ether가 요약하는 과정에서 빠질 수 있다.

따라서 Scout 보고서에는 `불변조건 / must-preserve contracts` 항목이 추가되는 편이 안전하다.

## 질문별 답변

### 1. 파이프라인 흐름에 빠진 단계나 모순이 있는가

큰 흐름 자체는 자연스럽다.

- Ether triage
- Scout 정찰
- Ether 지시서 작성
- Coder 구현
- Reviewer 검토
- QA 검증
- Ether 최종 판단

이 구조는 `/build` 기준으로 충분히 작동 가능하다.

빠진 단계라기보다는 아래 한 칸이 보강되면 더 좋아진다.

- Scout 출력에 `불변조건 / 절대 깨지면 안 되는 계약` 항목 추가

현재의 주요 모순은 구조보다는 **Hook 설명과 예시의 불일치**, 그리고 **Playwright 런타임 전제 불일치** 쪽이다.

### 2. Hook 운영 등급(block/warn/future) 분류가 적절한가

적절하다.

현재 저장소 기준 추천은 그대로 유지해도 된다.

- `security-guard`만 `block`
- `fairness/css/mobile`은 `warn`
- `tdd/format`은 `future`

이 분류는 이전 피드백을 잘 반영한 결과로 보인다.

### 3. Scout → Ether → Coder 핸드오프에서 정보 손실 가능성

이전보다 크게 좋아졌다.

특히 "Scout가 먼저 보고, Ether가 그 보고서를 바탕으로 지시서를 작성한다"는 구조는 매우 적절하다.

다만 아래 정보는 손실될 가능성이 있다.

- 기존 시스템이 반드시 유지해야 하는 계약
- 수정은 작아 보여도 실제 운영상 민감한 파일
- 라우트/이벤트/SEO/배포 관련 암묵 규칙

따라서 Scout 출력 형식에 아래를 추가하는 것이 좋다.

- `불변조건`
- `수정 금지 또는 주의 대상`
- `검증 시 반드시 확인할 회귀 포인트`

### 4. 실제 구현 시 가장 먼저 부딪힐 문제점

우선순위로 보면 아래 4개다.

1. Hook matcher와 실제 차단/경고 반환값을 문서와 맞추는 일
2. Playwright 기준 URL과 서버 실행 방식을 현재 앱 구조에 맞추는 일
3. 기존 `.claude` 명령/스킬 구조와 하네스를 공존시키는 전환 방식 결정
4. `tdd-guard`를 future에서 실제 운영으로 올릴 만큼 테스트 자산이 충분하지 않다는 점

### 5. 실제 `.claude/skills/` + `.claude/agents/`로 옮길 때 주의할 점

현재 저장소에는 이미 아래가 있다.

- `.claude/skills/skill-*.md`
- `.claude/rules/*.md`
- `.claude/commands/*.md`
- `.claude/hooks/*`

반면 `.claude/agents/`는 아직 없다.

따라서 구현 시 주의할 점은 다음과 같다.

- 기존 `/meeting`, `/qa`, `/review` 명령과 충돌하지 않게 단계적으로 넣을 것
- Scout는 읽기 전용으로 명확히 고정할 것
- Ether 지시서의 입력/출력 포맷을 고정할 것
- 이름만 추가하지 말고, 실제 참조 파일과 실행 책임 범위를 명시할 것
- 문서에만 존재하는 가정(예: 모든 tool call hook 강제)을 실제 설정으로 뒷받침할 수 있는지 먼저 확인할 것

### 6. 이전 피드백 반영이 제대로 됐는가

대체로 잘 반영됐다.

반영이 잘 된 부분:

- 운영 버전 분리
- Hook 등급 분리
- TDD 현실 반영
- 공정성 정교화
- 모바일 수동/자동 분리
- Scout 추가
- 페르소나 제거
- 행동 지시 중심 전환

부족한 부분:

- 세부 예시가 최신 등급 체계를 완전히 따라가지 못함
- Playwright 문서가 현재 런타임 구조를 기준으로 다시 맞춰져야 함
- Scout 출력 형식에 불변조건 슬롯이 아직 없음

## 지금 바로 보완하면 좋은 순서

### 1순위

- `hooks-spec.md`와 `build-pipeline.md`의 Hook 설명/예시/기대 결과를 최신 운영 등급 기준으로 완전히 맞춘다.

### 2순위

- `playwright-mcp.md`의 테스트 URL, 서버 시작 방식, 기본 시나리오를 현재 Express 라우트 구조에 맞춘다.

### 3순위

- Scout 보고서 출력 형식에 `불변조건`, `회귀 체크포인트`, `주의 대상 계약` 항목을 추가한다.

### 4순위

- 이후 실제 `.claude/skills/`와 `.claude/agents/` 구조로 옮길 때, 기존 명령 체계와의 공존 전략을 먼저 정한 뒤 점진 적용한다.

---

## 최종 판단

이 설계는 이전보다 확실히 좋아졌다.

지금 가장 중요한 건 구조를 다시 뒤엎는 것이 아니라, **이미 좋아진 설계를 실제 실행 가능한 문서로 다듬는 마무리 작업**이다.

특히 아래 세 가지를 우선 고치면 실전 전환 가능성이 크게 올라간다.

- Hook 설명과 예시 동기화
- Playwright 런타임 전제 정렬
- Scout 불변조건 보고 강화
