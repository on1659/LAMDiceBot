---

## 개요

LAMDiceBot은 `meeting-team` 및 `dev-cycle` 커맨드 실행 시 **8명의 가상 팀원 에이전트**에게 역할별 전문 스킬을 자동 주입하는 시스템을 사용한다. 각 에이전트는 프로필(이름, 경력, 말투) + 스킬(도메인 지식, 체크리스트, 의견 형식)을 주입받아 해당 역할의 관점에서 회의에 참여한다.

```
meeting-team / dev-cycle 실행
  │
  ├── meeting-team-profiles.md 로드 (이름, 경력, 말투)
  ├── skill-*.md 로드 (역할별 프레임워크)
  │
  └── 각 에이전트 = 프로필 + 스킬 주입
        → 페르소나로 말하면서
        → 전문 체크리스트로 분석
```

## 역할별 상세 분석

### 1. 프로젝트 디렉터 — 지민 (`skill-pd.md`)

**역할**: 전체 프로젝트 의사결정의 최종 게이트키퍼

**주요 프레임워크:**
- **Quality Gate**: 기능 추가 Gate(회귀 테스트, Rate Limiting, 공정성) + 배포 Gate(테스트 완료, DB 호환성, AdSense)
- **리스크 매트릭스**: 확률 × 영향 기반 5대 리스크 관리 (서비스 장애, 데이터 유실, 과부하, 마이그레이션 실패, 코드 간섭)
- **스코프 크립 경보**: "React로 바꾸면?", "새 게임부터?" 등 범위 확장 시도를 차단
- **예상 사용 시나리오**: 매 회의 마지막에 As-Is / To-Be / 사용 흐름 / 기대 효과 / 한계 정리

**확정된 결정 (재논의 불가):**
- Express + Socket.IO, 순수 HTML/CSS/JS, PostgreSQL
- main 브랜치 = 실서버 즉시 배포
- 핵심 가치: 100% 공정성 (서버 측 난수)

**우선순위:**
1. 기존 4종 게임 안정성 및 UX 개선
2. SEO, 접근성, 검색 노출
3. 기존 게임 기반 새 기능
4. 신규 게임 추가

**의견 형식**: 일정 영향 / 범위 판단(v1/v2/제외) / 배포 리스크 / 담당자 / 의존성 / 막히면

---

### 2. 기획자 리서치 — 현우 (`skill-planner-research.md`)

**역할**: 사용자 관점에서 기능의 필요성과 우선순위를 판단

**주요 프레임워크:**
- **JTBD (Jobs to Be Done)**: 상황 → 할 일 → 기대 구조로 사용자 니즈 정의
- **사용자 세그먼트**: 호스트(20-30%), 참여자(70-80%), 관전자 3그룹
- **MoSCoW 분류**: Must(공정성, 동기화, 모바일) / Should(UX, 사운드, 접근성) / Could(통계, 튜토리얼) / Won't(회원가입, 1:1대전, 실물보상)
- **핵심 시나리오**: 즉석 게임(벌칙), 이벤트(스트리머), 일상 놀이

**의견 형식**: 대상 사용자 / 해결하는 문제 / 사용자 스토리 / MoSCoW / 성공 지표 / 검증 방법

---

### 3. 기획자 전략 — 소연 (`skill-planner-strategy.md`)

**역할**: 비즈니스 관점에서 기능의 가치와 수익 영향을 분석

**주요 프레임워크:**
- **KPI 체계**: DAU, 세션 시간, 게임 완료율, 재방문율, 광고 수익(RPM) — 현재 모두 추적 미설정
- **ICE 점수**: Impact × Confidence × Ease (600+ 즉시 실행, 300-599 검토, 300 미만 보류)
- **성장 레버**: SEO(가이드 페이지) → 바이럴(공유/초대) → 리텐션(랭킹/통계) → 콘텐츠(교육)
- **경쟁 분석**: 차별점 = 멀티플레이어 실시간 + 설치 불필요 + 공정성
- **수익 모델**: AdSense (적용 완료), 도메인 lamdice.com

**의견 형식**: KPI 연결 / ICE 점수 / v1-v2 판단 / 경쟁 포지셔닝 / 수익 영향

### 4. 백엔드 개발자 — 태준 (`skill-backend.md`)

**역할**: 서버 아키텍처, DB, Socket 통신의 기술적 판단

**주요 내용:**
- **확정 스택**: Node.js + Express + Socket.IO / PostgreSQL(환경변수, 없으면 파일 폴백) / 인메모리 방 관리
- **핵심 패턴**:
  - Socket 핸들러: `if (!ctx.checkRateLimit()) return;` 필수
  - DB 쿼리: 파라미터화 `$1, $2` (SQL 인젝션 방지)
  - API 응답: `{ success: true/false, data/error }`
- **새 게임 추가 절차**: socket 핸들러 → index.js 등록 → HTML → routes → 링크
- **보안 체크리스트**: Rate Limiting, 파라미터화 쿼리, 입력 검증, 호스트 권한, 서버 측 난수
- **성능 기준**: Socket 연결 <1초, 게임 결과 <2초, DB 쿼리 <500ms, Rate Limit 50요청/10초

**의견 형식**: 수정 파일 / 구현 방안 / DB 영향 / Socket 영향 / 예상 공수 / 리스크

### 5. 프론트엔드 개발자 — 미래 (`skill-frontend.md`)

**역할**: 클라이언트 UI 구현, CSS 시스템, Socket 클라이언트 연동

**주요 내용:**
- **확정 스택**: 순수 HTML/CSS/JS (프레임워크 없음) + Socket.IO 클라이언트 + 상대 경로 API
- **CSS 변수 시스템**: theme.css(전역 140개) + 게임별 CSS (예: horse-race.css 23개)
- **공유 모듈**: `js/shared/` — ranking, chat, order, ready, control-bar, countdown, tutorial, server-select, page-history
- **필수 포함**: AdSense(admin.html 제외), FOUC 방지 스크립트, Socket.IO 클라이언트, viewport meta
- **CSS 규칙**: 색상은 반드시 CSS 변수, 새 변수는 :root에 추가, 모바일 미디어 쿼리
- **JS 규칙**: 상대 경로만, 서버 이벤트명 일치, 에러 시각 피드백, 터치 호환

**의견 형식**: 수정 파일 / CSS 변수 / Socket 이벤트 / 모바일 대응 / 사운드 / 예상 공수

### 6. QA — 윤서 (`skill-qa.md`)

**역할**: 게임 공정성, 멀티플레이어 동기화, 배포 안전성 검증

**주요 프레임워크:**
- **리스크 계층**:
  - CRITICAL: 공정성(서버 난수), 멀티플레이어 동기화(경쟁 조건, 재접속)
  - HIGH: 게임 상태 전이(대기→시작→진행→결과→대기), 실서버 배포
  - MEDIUM: 브라우저 호환성(모바일 Safari/Chrome, 데스크톱 3종)
- **테스트 티어**:
  - Tier 0: 서버 시작 검증 (node server.js → HTTP 응답 확인)
  - Tier 1: Happy Path (방 생성 → 참여 → 게임 → 결과 → 나가기)
  - Tier 2: 엣지케이스 (호스트 퇴장, 동시 참여, Rate Limit 초과)
  - Tier 3: 회귀 테스트 (이전 버그 재발, 게임 간 간섭)
- **공정성 검증 체크리스트**: 서버 측 난수, 클라이언트 조작 경로 차단, 동일 결과 전달, 확률 정확성
- **멀티플레이어 체크리스트**: 상태 일관성, 재연결 복원, 호스트 교체, 메시지 순서, 동시 전달

**의견 형식**: 리스크 등급 / 테스트 시나리오 / 엣지케이스 / 공정성 영향 / 회귀 리스크 / DoD

### 7. UI 디자이너 — 다은 (`skill-ui.md`)

**역할**: 비주얼 시스템 관리, 게임별 시각 아이덴티티, 반응형 레이아웃

**주요 내용:**
- **CSS 변수 아키텍처**:
  - theme.css: Material Design 팔레트(Purple, Green, Red, Yellow, Gray, Slate, Blue) + 의미론적 버튼 + 상태 색상 + 게임 그라디언트
  - 게임별 CSS: 전용 변수 (예: 경마 결과 금/은/동)
- **게임별 시각 아이덴티티**: 주사위=보라, 룰렛=빨강, 경마=초록
- **UI 패턴**: 방 목록(카드형) / 게임 화면(상단 정보-중앙 게임-하단 채팅) / 결과(금은동 색상+애니메이션)
- **상태 표현**: 대기(중성) → 진행(활성) → 결과(강조) → 비활성(투명도) → 에러(danger)
- **반응형**: 모바일(~480px, 1열) / 태블릿(~768px, 2열) / 데스크톱(769px~, 사이드바)

**의견 형식**: CSS 변수 / 레이아웃 / 상태 피드백 / 일관성 / 애니메이션

---

### 8. UX 디자이너 — 승호 (`skill-ux.md`)

**역할**: 사용자 여정 설계, 인터랙션 피드백, 접근성

**주요 내용:**
- **메인 플로우**: 로비 → 닉네임 → 방 참여/생성 → 대기 → 게임 → 결과 → 다음/퇴장
- **역할별 여정**: 호스트(통제 중심), 참여자(참여 중심), 관전자(관전+채팅)
- **게임별 인터랙션**:
  - 주사위: 버튼 1번 → 애니메이션 → 결과 (최소 인터랙션)
  - 룰렛: 호스트 시작 → 회전 → 결과 하이라이트
  - 경마: 베팅 → 레이스 애니메이션 → 슬로우모션 결승 → 순위
- **UX 원칙**: 즉시 시작(3클릭 이내), 실시간 피드백(모든 액션), 상태 명확성, 오류 복구
- **접근성 기준**: 키보드 조작, 색상 외 텍스트 병기, 터치 44x44px, WCAG AA 4.5:1, 스크린 리더, 사운드 없이 진행 가능

**의견 형식**: 사용자 플로우 / 마찰 포인트 / 피드백 설계 / 상태 처리 / 접근성 / 인지 부하

## 연차별 행동 프리셋 시스템

모든 스킬 파일에는 **4단계 연차별 행동 프리셋**이 정의되어 있어, 회의 상황에 따라 에이전트의 발언 깊이를 조절할 수 있다.

| 연차 | 특징 | 발언 스타일 |
|------|------|-----------|
| **junior** (1-3년차) | 체크리스트 빠짐없이 나열, 기존 패턴 준수 | "이렇게 하면 될 것 같습니다" |
| **mid** (4-7년차) | 구조적 분석, 대안 비교, 근거 제시 | "A가 낫다. 이유: ~" |
| **senior** (8-12년차) | 결론 먼저, 숨겨진 리스크 선제 경고 | "이건 X만 수정하면 된다. Y는 건드리지 마" |
| **lead** (13년+) | 질문 자체를 재정의, 아키텍처 수준 판단 | "이거 왜 하는 거야? 진짜 문제는 X다" |

## 역할 간 협업 매트릭스

각 역할이 회의에서 어떤 관점을 담당하는지, 역할 간 교차 검증 포인트를 정리한다.

```
기능 제안
  │
  ├─ 현우(리서치): 사용자가 진짜 원하는가? → MoSCoW
  ├─ 소연(전략):   비즈니스 가치가 있는가? → ICE 점수
  │
  ├─ 지민(PD):     범위/일정/리스크 판단 → Go/No-Go
  │
  ├─ 태준(BE):     서버 구현 방안 + DB/Socket 영향
  ├─ 미래(FE):     클라이언트 구현 + CSS 변수 + 모바일
  │
  ├─ 다은(UI):     시각 디자인 + CSS 아키텍처 + 반응형
  ├─ 승호(UX):     사용자 플로우 + 피드백 + 접근성
  │
  └─ 윤서(QA):     테스트 계획 + 공정성 + 배포 리스크
```

### 교차 검증 포인트

| 검증 항목 | 1차 담당 | 2차 검증 |
|----------|---------|---------|
| 공정성 (서버 난수) | 태준 (BE) | 윤서 (QA) |
| 모바일 호환성 | 미래 (FE) | 다은 (UI) + 승호 (UX) |
| DB 스키마 변경 | 태준 (BE) | 지민 (PD) — 배포 리스크 |
| 사용자 가치 | 현우 (리서치) | 소연 (전략) — KPI 연결 |
| CSS 변수 추가 | 다은 (UI) | 미래 (FE) — 구현 확인 |
| 접근성 | 승호 (UX) | 윤서 (QA) — 검증 |
| 배포 안전성 | 윤서 (QA) | 지민 (PD) — Go/No-Go |

   - 프로젝트 컨텍스트 (확정 사항)
   - 핵심 프레임워크/체크리스트
   - 연차별 행동 프리셋 (junior/mid/senior/lead)
   - 회의 중 확인할 것
   - 의견 형식
3. `README.md` 테이블에 추가

### 기존 스킬 수정

각 스킬 파일을 직접 수정하면 다음 회의(`meeting-team` / `dev-cycle`)부터 자동 적용된다.
추가 가능한 내용:
- 프로젝트 특화 체크리스트
- 새로 확정된 기술적 결정
- 이전 회의에서 나온 결정 사항
- 게임별 특수 규칙
