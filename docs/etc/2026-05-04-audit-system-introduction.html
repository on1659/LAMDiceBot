Audit 시스템 도입 — 작업 방식 변화 보고서

하네스(.claude/)와 메모리 폴더를 자동으로 점검하는 audit 시스템을 도입했습니다. 오늘 세션 전후로 작업 방식이 어떻게 달라졌는지 정리합니다.
2026-05-04 자동화 도입 감사 6개 룰 통과 한국어 보고서
TL;DR
손으로 가끔 메모리/하네스를 정리하던 방식 → SessionStart 카운터가 10세션마다 자동 알림하고, /memory-audit과 /harness-audit 두 슬래시 커맨드가 총 12개 룰로 후보를 진단하면 사용자 승인 후 일괄 정리하는 구조로 전환. 오늘 정리 결과는 메모리 -38%, 하네스 worktree 7→0, 토큰 누출 1→0건.

1. 핵심 변화 4가지

⏰

점검 트리거

Before: 사용자가 가끔 떠올렸을 때 손으로
After: SessionStart마다 카운터 +1, 10세션마다 자동 알림

🔍

진단 룰

Before: 명문화 없음, 매번 즉흥
After: 메모리 6개 + 하네스 6개 = 12개 룰 표준화

🛡️

안전 가드

Before: 자동 삭제 위험 노출
After: destructive 액션은 전부 사용자 승인 필요, dirty worktree 보존 우선

🔒

보안

Before: 텔레그램 토큰 prefix가 메모리에 노출
After: 정규식 기반 secret 누출 검사 + .env 참조로 전환

2. 작업 방식 — Before vs After

Before
  • 점검을 시작할 트리거가 없어서 잊기 쉬움
  • 점검 룰이 매번 새로 정의됨 — 일관성 없음
  • 고아 파일, 깨진 참조 같은 건 사실상 점검 못 함
  • worktree dirty 처리 절차가 즉흥적
  • secret 누출 검사 없음
  • 점검 결과를 외부 보고로 남기지 않음
→
After
  • SessionStart 카운터 — 10세션마다 자동 알림
  • 12개 룰이 코드/문서로 명문화
  • orphan / broken-ref / hook 무결성 등 자동 검사
  • dirty worktree는 WIP commit으로 보존 후 정리
  • 토큰 패턴 정규식 검사 (룰 6)
  • HTML 가이드 + 슬래시 커맨드 + 카운터 hook

  1. Stale (30일+ / 60일+)

  2. Stale Lock (24h+)

9. Hermes Agent와의 비교 (참고)

오늘 세션 초반에 Nous Research의 Hermes Agent 도입 검토했으나, LAMDiceBot의 도메인 특화 하네스(공정성 / 멀티플레이어 / 14곳 등록 체크리스트)가 범용 self-improving 에이전트보다 적합도 우위로 판단. 다만 운영 layer(텔레그램 보고 / 크론 retro)는 LAMDice-Pilot 위 Hermes PoC로 검토 가치 있음.

LAMDiceBot 하네스 (오늘 도입 후) Hermes Agent
스코프 프로젝트 별 글로벌 (사용자 단위)
메모리 한계 한계 없음 (인덱스 28줄 권장) MEMORY 2,200자 / USER 1,375자
자기개선 사용자 승인 후 룰 수정 5+ tool call 후 자동 patch
도메인 특화 공정성 / 모바일 / 멀티플레이어 없음 (범용)
외부 통합 현재 없음 텔레그램 / Discord / 크론 내장
결론
본체 코드 작업은 그대로 LAMDiceBot 하네스 + 오늘 도입한 audit 시스템으로 운영. Hermes는 LAMDice-Pilot에 한정해 텔레그램 보고 / 24h VPS 모니터링 PoC 후보로 별도 검토.
