# 03. 새로운 게임 & 확장 제안

> **작성일**: 2026-01-31
> **주제**: LAMDiceBot 플랫폼의 새로운 게임 추가 및 확장 기능
> **참여자**: 기획자 🎯 vs 프로그래머 💻

---

## 1. 카드 뽑기 게임 (가챠 스타일)

**기획자 🎯**: 카드 뽑기 게임 어때? 요즘 가챠 시스템 인기 많잖아. 사용자들이 랜덤으로 카드 뽑고, 등급별로 희귀도 있고... 팀 빌딩할 때도 재미있을 것 같은데?

**프로그래머 💻**: 음... 카드 데이터는 어떻게 관리할 건데? 이미지 에셋도 필요하고, 확률 테이블도 만들어야 하고. DB 없이 메모리로만 관리하기엔 좀 부담스러운데?

**기획자 🎯**: 아, 처음엔 간단하게 시작하자. 텍스트 기반으로 카드 이름이랑 등급만 표시하고, 이미지는 나중에 추가하는 거지. 확률은 JSON 파일로 관리하면 되잖아. 그리고 Gemini AI 활용해서 카드 설명 자동 생성하면 재미있을 것 같은데?

**프로그래머 💻**: 오 그거 괜찮은데? Gemini로 카드 플레이버 텍스트 생성하는 건 기존 인프라 활용하는 거니까 개발 비용이 적네. 그럼 기본 구조는 `card-draw-multiplayer.html` 만들고, `assets/cards/card-config.json`에 카드 풀 정의하고... Socket.IO로 실시간 뽑기 결과 공유하는 식으로 가면 될 것 같아.

**기획자 🎯**: 맞아! 그리고 "연속 10회 뽑기"나 "천장 시스템" 같은 거 넣으면 사용자 engagement 확 올라갈 거야. 방장이 뽑기 설정 커스터마이징할 수 있게 하고.

**프로그래머 💻**: 천장 시스템은 세션별로 카운터 관리해야 하니까 `server.js`에 roomData 확장이 필요하겠네. 근데 구현 난이도는 중간 정도? 기존 게임 구조 재활용하면 2주 정도면 MVP 나올 것 같아.

---

## 2. OX 퀴즈 게임 (AI 생성 문제)

**프로그래머 💻**: OX 퀴즈는 사실 구현이 제일 쉬울 것 같은데? 버튼 두 개만 있으면 되고, 정답 체크도 간단하고.

**기획자 🎯**: 그건 좀... 너무 단순하잖아. 차별화가 필요해. 내 생각엔 Gemini AI로 주제별 퀴즈를 실시간 생성하는 거야. 예를 들어 방장이 "영화", "역사", "과학" 같은 카테고리 선택하면 AI가 그 주제로 OX 문제 10개 만들어주는 거지.

**프로그래머 💻**: 오... 그거 재미있긴 한데, API 비용이 좀 걱정되는데? 매번 문제 생성하면 Gemini API 호출 많이 늘어나잖아.

**기획자 🎯**: 그럼 캐싱하자! 한 번 생성된 문제는 `quiz-cache.json`에 저장해두고 재활용하는 거야. 그리고 사용자들이 "좋아요" 누른 문제는 DB처럼 축적해서 나중에 다른 방에서도 쓸 수 있게 하고.

**프로그래머 💻**: 괜찮네. 그럼 AI 생성 모드랑 캐시된 문제 모드 두 가지로 나누면 되겠다. 실시간 점수 계산이랑 순위 표시는 기존 주사위 게임 로직 거의 그대로 쓸 수 있고... 개발 기간은 1주일? 제일 빠르게 출시할 수 있을 것 같아.

**기획자 🎯**: 완벽해! 그리고 "스피드 퀴즈 모드"도 넣자. 5초 안에 답 안 하면 자동 탈락. 긴장감 있잖아.

**프로그래머 💻**: Timer 추가는 어렵지 않아. `setTimeout` 쓰면 되니까. OK, 이거 우선순위 높게 가져가자.

## 3. 미니게임 모음 (파티 게임 컬렉션)

**기획자 🎯**: 지금 게임들이 다 좀 무겁잖아. 5분 안에 끝나는 가벼운 미니게임 모음 어때? "라이어 게임", "초성 퀴즈", "그림 추리" 이런 거.

**프로그래머 💻**: 그건 좀... 범위가 너무 넓은데? 미니게임 하나하나가 다 별도 프로젝트 수준인데, "모음"이라고 하면 최소 5~6개는 만들어야 할 것 같은데?

**기획자 🎯**: 아니, 처음엔 3개만! "초성 퀴즈", "숫자 야구", "끝말잇기" 이 정도면 개발 부담 적잖아? 텍스트 기반이니까 UI도 간단하고.

**프로그래머 💻**: 초성 퀴즈랑 끝말잇기는 한글 처리가 필요해서 좀 까다로운데... 숫자 야구는 쉽겠네. 그나저나 이거 별도 HTML 파일로 만들 거야, 아니면 하나의 `minigames-multiplayer.html` 안에서 게임 전환하는 식으로 갈 거야?

**기획자 🎯**: 후자! 하나의 플랫폼에서 게임 선택하는 거지. 로비 화면에서 "게임 1: 초성 퀴즈", "게임 2: 숫자 야구" 이런 식으로 고르고. 그래야 유저가 여러 게임 번갈아가면서 할 수 있잖아.

**프로그래머 💻**: 그럼 게임 전환 로직이랑 상태 관리가 복잡해지는데... 각 게임마다 별도 모듈로 분리하고, `minigame-loader.js` 같은 거 만들어서 동적으로 로드하는 식으로 가야겠네. 개발 기간은... 3주 정도?

**기획자 🎯**: 오 그거 좋다! 그리고 나중에 게임 추가하기도 쉽겠네. 모듈형으로 만들면.

**프로그래머 💻**: 맞아. 확장성은 좋을 거야. 근데 초기 개발 비용이 좀 높은 편이니까 우선순위는 중간 정도로 두자.

## 4. PWA 모바일 앱 (Progressive Web App)

**프로그래머 💻**: PWA는 사실 기술적으론 어렵지 않아. `manifest.json` 만들고, Service Worker 등록하고, HTTPS 적용하면 끝인데... Railway가 이미 HTTPS 지원하니까 절반은 된 거지.

**기획자 🎯**: 오! 그럼 모바일 앱으로 배포할 수 있다는 거야? 사용자들이 홈 화면에 추가하면 네이티브 앱처럼 쓸 수 있고?

**프로그래머 💻**: 응. 근데 완전한 네이티브 앱은 아니야. 오프라인 지원, 푸시 알림, 백그라운드 동기화 이런 기능 넣으려면 Service Worker 로직 제대로 짜야 하는데, 그게 좀 까다로워.

**기획자 🎯**: 일단 기본 기능만이라도 빨리 나가자. 홈 화면 추가, 스플래시 화면, 모바일 최적화된 UI 이 정도면 충분할 것 같은데? 오프라인 기능은 Phase 2로 미루고.

**프로그래머 💻**: 그럼 개발 난이도가 확 줄어드네. `manifest.json` 작성하고, 아이콘 여러 사이즈로 준비하고, CSS 미디어 쿼리로 모바일 레이아웃 조정하고... 1주일이면 될 것 같아.

**기획자 🎯**: 완벽해! 그리고 이거 홍보 효과 엄청 좋을 거야. "모바일 앱으로도 사용 가능!" 이러면 사용자 유입 확 늘어날 텐데?

**프로그래머 💻**: 맞아. 비용 대비 효과가 제일 좋을 것 같아. 우선순위 높게 잡자.

## 5. 커스텀 게임 빌더 (사용자 게임 제작)

**기획자 🎯**: 이건 진짜 킬러 피처인데... 사용자가 직접 게임 규칙을 만들 수 있게 하는 거야. "주사위 3개 굴려서 합이 10 이상이면 승리" 이런 식으로 룰을 GUI로 조합하는 거지. No-code 게임 빌더!

**프로그래머 💻**: 와... 그건 좀 규모가 엄청난데? 룰 엔진 설계부터 시작해야 하고, 조건 파싱, 실행 로직, UI 빌더... 이거 거의 새 프로젝트 수준인데?

**기획자 🎯**: 알아, 근데 이거 되면 플랫폼이 완전히 달라지는 거야. 사용자들이 무한히 새로운 게임 만들어내고, 공유하고... UGC(User Generated Content) 생태계가 만들어지는 거지.

**프로그래머 💻**: 비전은 좋은데... 현실적으로 개발 기간이 최소 2달은 걸릴 것 같은데? 그리고 사용자가 만든 게임이 버그 투성이면 어떡하고? 보안 이슈는? 무한 루프 같은 거 만들면?

**기획자 🎯**: 그럼 제약을 두는 거지. 블록 기반 빌더로 만들어서 허용된 액션만 조합할 수 있게 하고. "주사위 굴리기", "카드 뽑기", "점수 비교" 같은 프리셋 블록들만 제공하면 안전하잖아?

**프로그래머 💻**: 음... 그래도 여전히 복잡한데. 그리고 이거 제대로 만들려면 프론트엔드에 드래그앤드롭 UI 라이브러리 도입해야 하고, 백엔드에 룰 검증 로직 필요하고... 기존 코드베이스랑 결이 많이 달라.

**기획자 🎯**: 알겠어, 이건 장기 프로젝트로 가자. 하지만 로드맵엔 꼭 넣어두자. 이거야말로 진짜 차별화 포인트니까.

**프로그래머 💻**: 동의. 우선순위는 낮지만, 미래 비전으로는 최고야.

## 6. API 공개 / 봇 연동 (Discord, Slack 통합)

**기획자 🎯**: API 공개하면 어때? Discord 봇, Slack 봇에서 우리 게임 바로 실행할 수 있게. "/dice roll" 치면 LAMDiceBot 서버랑 연결돼서 게임 시작하는 거지.

**프로그래머 💻**: 오 그거 괜찮은데? 근데 현재 구조가 완전히 웹 기반이잖아. Socket.IO로 실시간 통신하고, HTML로 UI 렌더링하고... 봇 연동하려면 API 레이어를 새로 만들어야 해.

**기획자 🎯**: RESTful API랑 WebSocket API 둘 다 제공하면 되지 않아? `/api/game/dice/start`, `/api/game/dice/roll` 이런 엔드포인트 만들고.

**프로그래머 💻**: 가능은 한데, 기존 게임 로직이 다 HTML 렌더링이랑 결합돼 있어서 분리 작업이 필요해. 게임 로직을 순수 JS 모듈로 뽑아내고, API 응답은 JSON으로 주고... 리팩토링이 좀 필요하겠네.

**기획자 🎯**: 그럼 이참에 코드 정리하는 셈 치자! 게임 로직 분리하면 나중에 다른 기능 추가할 때도 편하잖아. 그리고 API 문서 Swagger로 자동 생성하고.

**프로그래머 💻**: Swagger 좋지. Express에 `swagger-jsdoc` 붙이면 되고... 인증은 API 키 방식으로 가고, Rate limiting 넣어서 남용 방지하고. 개발 기간은 3주 정도?

**기획자 🎯**: 완벽해! 이거 되면 개발자 커뮤니티에서 입소문 날 거야. "LAMDiceBot API 써서 우리 Discord 서버에 미니게임 추가했어요!" 이런 식으로.

**프로그래머 💻**: 맞아. 확장성 측면에서 엄청 중요해. 우선순위 중상 정도로 가자.
