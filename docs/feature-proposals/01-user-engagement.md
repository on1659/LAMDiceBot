## 1. 토너먼트 모드

**🎯 기획자**: 토너먼트 모드 어때요? 8명, 16명 이렇게 브래킷 짜서 승자전으로 올라가는 거죠. 사람들이 진짜 경쟁심 생겨서 재미있을 것 같은데요.

**💻 프로그래머**: 아이디어는 좋은데... 일단 현재 게임들이 각각 다른 승패 로직을 가지고 있잖아요. 주사위는 점수 합계, 경마는 순위, 룰렛은 당첨 여부. 이걸 통일된 승패 시스템으로 만들어야 하는데 꽤 복잡할 것 같아요.

**🎯 기획자**: 그럼 게임별로 토너먼트 규칙을 다르게 가져가면 안 될까요? 경마는 1등만 진출, 주사위는 높은 점수 진출 이런 식으로요.

**💻 프로그래머**: 가능은 한데, 그러면 브래킷 관리 시스템이 복잡해지겠네요. Socket.IO로 실시간 브래킷 업데이트도 해야 하고... 대기실 관리도 문제예요. 8명이 다 모일 때까지 기다려야 하는데, 유저가 적으면 토너먼트가 시작도 안 될 수 있어요.

**🎯 기획자**: 아, 그건 좋은 지적이네요. 그럼 최소 인원 미달 시 봇을 채워 넣거나, 4명 미니 토너먼트로 유연하게 시작하는 건 어때요?

**💻 프로그래머**: 봇 추가는... Gemini API 비용이 들텐데요. 근데 4명 미니 토너먼트는 괜찮을 것 같아요. 2라운드만 하면 되니까 구현도 상대적으로 단순하고. 서버에 tournament-manager.js 모듈 하나 추가하고, 브래킷 상태를 메모리나 Redis에 저장하면 될 것 같네요.

**🎯 기획자**: 좋아요! 그럼 MVP는 4명 토너먼트로 시작하고, 나중에 확장하는 걸로 하죠.

**💻 프로그래머**: 네, 그게 현실적일 것 같습니다. 예상 공수는 2-3주 정도?

**결론**: 4명 미니 토너먼트 형태로 MVP 구현. 각 게임별 승패 규칙 정의 필요. 브래킷 UI는 간단한 트리 구조로 시작.

## 2. 업적/뱃지 시스템

**🎯 기획자**: 업적 시스템은 필수라고 봐요. "10연승 달성", "경마 100판 플레이", "첫 승리" 이런 거 모으는 재미가 쏠쏠하거든요. 컬렉션 욕구를 자극하는 거죠.

**💻 프로그래머**: 업적 자체는 구현이 어렵지 않은데, 문제는 데이터베이스예요. 지금 우리 서버 메모리 기반이잖아요. 업적은 영구 저장되어야 하니까 DB가 필요한데, Railway에 PostgreSQL 추가하면 비용이...

**🎯 기획자**: 음... 그럼 일단 로컬스토리지로 클라이언트에 저장하면 안 될까요? 브라우저 쿠키나 localStorage 말이에요.

**💻 프로그래머**: 그건 너무 쉽게 조작당해요. F12 열어서 업적 다 해금할 수 있거든요. 진정성이 떨어지죠. 차라리 Railway의 무료 티어 PostgreSQL을 쓰거나, 아니면 JSON 파일 기반 간단한 DB를 만드는 게 나을 것 같은데요.

**🎯 기획자**: JSON 파일이요? 그게 안정적인가요?

**💻 프로그래머**: 유저가 수백 명 이하면 괜찮아요. 파일 읽기/쓰기에 락을 걸어서 동시성 문제만 해결하면 됩니다. 나중에 확장하면 그때 PostgreSQL로 마이그레이션하는 거죠. 업적 종류는 얼마나 기획하셨어요?

**🎯 기획자**: 게임당 5-7개씩, 총 30개 정도 생각했어요. 난이도별로 브론즈, 실버, 골드 이렇게.

**💻 프로그래머**: 30개면 적당하네요. 업적 체크 로직은 각 게임 종료 시점에 훅을 걸어서 achievement-checker.js가 검증하는 식으로 하면 될 것 같아요. UI는 모달 팝업으로 간단하게 구현하고요.

**🎯 기획자**: 완벽해요! 업적 해금할 때 애니메이션 효과도 넣어주시면 좋겠어요. 성취감 느끼게요.

**💻 프로그래머**: 애니메이션은 CSS로 금방 만들 수 있으니 문제없습니다. 예상 공수 1.5주 정도?

**결론**: JSON 파일 기반 영구 저장소 사용. 30개 업적 구현, 게임별 훅으로 실시간 체크. 해금 애니메이션 포함.

## 3. 관전 모드

**🎯 기획자**: 관전 모드 정말 필요한 것 같아요. 친구들이 게임하는 거 보면서 채팅으로 응원하고, 특히 토너먼트 있으면 더 몰입감 있잖아요.

**💻 프로그래머**: 관전 모드... 기술적으론 그렇게 어렵지 않아요. Socket.IO로 이미 실시간 통신하고 있으니까, 관전자용 room만 추가하면 되거든요. 근데 문제는 대역폭이에요.

**🎯 기획자**: 대역폭이요?

**💻 프로그래머**: 네. 지금은 플레이어끼리만 데이터를 주고받는데, 관전자가 10명, 20명 붙으면 같은 게임 상태를 그만큼 더 브로드캐스트해야 해요. Railway 무료 티어는 대역폭 제한이 있어서, 관전자 수를 제한하거나 별도 요금제로 올려야 할 수도 있어요.

**🎯 기획자**: 아... 그럼 일단 관전자는 방당 5명으로 제한하는 건 어때요? 나중에 인기 많아지면 그때 서버 업그레이드하고요.

**💻 프로그래머**: 그게 현실적이네요. 그리고 관전자는 읽기 전용이니까, 게임 상태만 받고 입력은 못 하게 해야죠. 채팅은 가능하게 하고요. 근데 "라이브 해설"은 어떻게 구현하려고 하셨어요?

**🎯 기획자**: 아, 그건 Gemini AI한테 현재 게임 상황 던져주면 자동으로 해설 생성하게 하려고 했어요. "3번 말이 선두를 달리고 있습니다!" 이런 식으로요.

**💻 프로그래머**: 오... 그거 괜찮은데요! Gemini API 호출 비용은 들지만, 매 턴마다 호출하지 말고 중요한 순간에만 호출하면 괜찮을 것 같아요. 경마에서 1등 바뀔 때, 주사위에서 역전 상황 발생할 때 같은 타이밍이요.

**🎯 기획자**: 완전 좋아요! 그럼 AI 해설은 선택적으로 켤 수 있게 하고, 관전자 수는 5명 제한으로 가죠.

**💻 프로그래머**: 네, 그럼 spectator-room.js 모듈 만들고, UI에 관전 모드 버튼 추가하면 될 것 같네요. 공수는... AI 해설 포함해서 2주 정도?

**결론**: 방당 관전자 5명 제한. Gemini AI 활용한 자동 해설 기능 (선택적 활성화). 관전자는 읽기 전용, 채팅 가능.

## 4. 리플레이 시스템

**🎯 기획자**: 리플레이 시스템이 있으면 대박 순간을 다시 보거나, SNS에 공유할 수 있잖아요. "내가 20연승했을 때 영상" 이런 거 저장해두고 싶어 하는 사람들 많을 거예요.

**💻 프로그래머**: 리플레이가 두 가지 방식이 있는데요. 하나는 비디오 녹화, 하나는 게임 상태 기록이에요. 비디오는 용량이 크고, 상태 기록은 나중에 재생하는 게 복잡해요. 어떤 걸 원하세요?

**🎯 기획자**: 음... 유저 입장에서는 비디오가 편하긴 한데, 용량 문제가 크다면 상태 기록도 괜찮아요. 어차피 우리 게임이 2D 애니메이션이니까 재생도 쉽지 않을까요?

**💻 프로그래머**: 맞아요. 게임 상태를 JSON 배열로 저장하고, 재생할 때 그대로 다시 렌더링하면 되거든요. 근데 문제는 또 저장소예요. 리플레이 데이터도 영구 보관해야 하는데, 한 게임당 몇 KB씩은 될 거고, 유저당 10개씩만 저장해도 금방 쌓여요.

**🎯 기획자**: 그럼 최근 5개만 저장하고, 나머지는 자동 삭제하는 건 어때요? "하이라이트 고정" 기능으로 중요한 건 따로 보관할 수 있게 하고요.

**💻 프로그래머**: 오, 그거 좋네요! 최근 5개 + 고정 3개 이렇게 하면 용량 관리가 되겠어요. 그럼 replay-storage.json에 유저별로 저장하면 될 것 같고... 재생 UI는 기존 게임 화면을 그대로 쓰되, 컨트롤러만 추가하면 되니까 개발이 크게 어렵진 않을 것 같아요.

**🎯 기획자**: 재생 속도 조절이랑 되감기도 되나요?

**💻 프로그래머**: 재생 속도는 쉬워요. setTimeout 간격만 조절하면 되거든요. 되감기는... 상태 배열을 역순으로 재생하면 되니까 가능은 한데, UI가 좀 복잡해질 수 있어요. 일단 정방향 재생이랑 일시정지만 구현하고, 되감기는 나중에 추가하는 게 어떨까요?

**🎯 기획자**: 네, 그게 나을 것 같아요. SNS 공유는요?

**💻 프로그래머**: 리플레이 URL을 생성해서 공유하면 될 것 같은데... 이건 좀 복잡해요. 나중에 2차 개발로 미루는 게 좋겠어요.

**🎯 기획자**: 알겠습니다! 일단 기본 재생 기능만 먼저 만들어봐요.

**결론**: 게임 상태 기록 방식 사용. 유저당 최근 5개 + 고정 3개 저장. 정방향 재생, 속도 조절, 일시정지 기능. SNS 공유는 2차 개발.

## 5. 커스텀 테마/스킨

**🎯 기획자**: 사용자들이 자기만의 개성을 표현할 수 있게 테마를 바꿀 수 있으면 좋을 것 같아요. 다크 모드, 네온 모드, 레트로 모드 이런 식으로요.

**💻 프로그래머**: 테마는 CSS 변수로 구현하면 되니까 기술적으론 간단해요. :root에 색상 변수 정의하고, 테마 전환 시 className만 바꿔주면 되거든요. 근데 스킨은 얘기가 달라요.

**🎯 기획자**: 스킨이요? 테마랑 다른 건가요?

**💻 프로그래머**: 네. 테마는 색상 팔레트만 바꾸는 거고, 스킨은 그래픽 에셋 자체를 바꾸는 거예요. 예를 들어 주사위 모양을 기본 큐브에서 크리스탈 다이아몬드로 바꾼다거나, 경마 말 디자인을 바꾼다거나 하는 거죠. 이건 에셋을 새로 만들어야 해서 디자인 리소스가 엄청 필요해요.

**🎯 기획자**: 아... 그럼 일단 테마만 먼저 구현하는 게 낫겠네요. 스킨은 나중에 디자이너 리소스 확보되면 추가하고요.

**💻 프로그래머**: 그게 좋을 것 같아요. 테마는 몇 개 정도 준비할까요?

**🎯 기획자**: 기본, 다크, 네온, 파스텔 이렇게 4개 정도면 적당할 것 같은데요. 사용자가 선택한 테마는 저장되어야 하니까, 아까 업적이랑 같이 user-settings.json에 넣으면 되겠죠?

**💻 프로그래머**: 네, 맞아요. 테마 정보는 용량도 작으니까 localStorage에 저장해도 되고요. 테마 전환 애니메이션도 CSS transition으로 부드럽게 만들 수 있어요.

**🎯 기획자**: 좋아요! 그럼 나중에 여유 있을 때 계절 테마도 추가하면 재미있을 것 같아요. 크리스마스, 할로윈 이런 거요.

**💻 프로그래머**: 계절 테마는 기존 시스템에 그냥 테마 옵션만 추가하면 되니까 나중에 쉽게 확장 가능해요. 공수는 1주일이면 충분할 것 같네요.

**결론**: 4가지 색상 테마 구현 (기본, 다크, 네온, 파스텔). CSS 변수 기반 테마 시스템. 스킨은 2차 개발로 연기. localStorage 저장.

## 6. 일일 챌린지

**🎯 기획자**: 일일 챌린지는 유저 유지율을 높이는 데 정말 효과적이에요. 매일 새로운 미션 주고, 완료하면 보상 주는 거죠. "오늘의 미션: 경마에서 3연승하기" 이런 식으로요.

**💻 프로그래머**: 일일 챌린지... 이건 생각보다 복잡한데요?

**🎯 기획자**: 왜요? 그냥 미션 목록 만들고, 매일 자정에 랜덤으로 하나 선택하면 되는 거 아닌가요?

**💻 프로그래머**: 미션 생성은 쉬운데, 문제가 세 가지예요. 첫째, 서버 시간 기준으로 자정에 리셋해야 하는데, 시간대가 다른 유저들은 어떻게 할 건가요? 둘째, 보상 시스템이 아직 없잖아요. 뭘 보상으로 줄 건가요? 셋째, 챌린지 진행도를 실시간으로 추적해야 하는데, 이것도 DB에 저장해야 해요.

**🎯 기획자**: 아... 그렇네요. 일단 시간대는 UTC 기준으로 통일하면 안 될까요? 그리고 보상은 아까 말한 업적 포인트 같은 걸 만들어서 쌓이면 특별 뱃지 주는 식으로요.

**💻 프로그래머**: UTC 기준은 가능한데, 한국 유저 기준으론 오전 9시에 리셋되는 거라 좀 애매할 수 있어요. 보상 시스템은 새로 만들어야 하니까 추가 공수가 들고요. 진행도 추적은 achievement 시스템이랑 비슷하게 만들면 될 것 같아요.
