# 소셜 & 커뮤니티 기능 제안서

**작성일**: 2026-01-31
**프로젝트**: LAMDiceBot 멀티플레이어 게임 플랫폼
**테마**: 소셜 & 커뮤니티 기능 강화

---

## 📋 문서 개요

이 문서는 LAMDiceBot의 사용자 참여도와 커뮤니티 형성을 위한 소셜 기능들을 논의합니다. 기획자와 프로그래머의 대화를 통해 각 기능의 사용자 가치와 기술적 구현 가능성을 검토합니다.

**현재 기술 스택**:
- Backend: Node.js + Express + Socket.IO
- Frontend: Vanilla JavaScript
- AI: Gemini Chatbot
- 배포: Railway
- 공통 모듈: ChatModule, order-shared.js, ready-shared.js

---

## 💬 기능 논의

### 1. 프로필 시스템

**🎯 기획자**: 사용자들이 자신만의 정체성을 가질 수 있는 프로필 시스템이 필요해요. 아바타, 닉네임, 게임 통계 같은 거요. 지금은 그냥 익명으로 들어와서 게임하고 나가잖아요. 재방문율을 높이려면 "내 프로필"이라는 개념이 있어야 한다고 봐요.

**💻 프로그래머**: 프로필 시스템은 사실 제일 기본이긴 한데... 지금 우리 로그인 시스템 자체가 없어요. Socket.IO 세션 기반으로만 돌아가는데, 이걸 구현하려면 먼저 사용자 인증 시스템을 만들어야 해요. DB도 필요하고요.

**🎯 기획자**: 꼭 복잡한 회원가입이 필요할까요? 간단하게 localStorage에 닉네임이랑 선택한 아바타만 저장해서 시작할 수는 없나요? 일단 가볍게 시작해서 나중에 계정 시스템으로 확장하는 게 어때요?

**💻 프로그래머**: 아, 그건 좋네요! "게스트 프로필"로 시작하는 거죠. localStorage에 닉네임, 아바타 ID, 간단한 통계(승수, 참여 게임 수 정도)만 저장하고, 서버에는 세션 동안만 유지하는 방식이면 금방 만들 수 있어요. 나중에 DB 붙이면 "프로필 저장하기" 기능으로 업그레이드하면 되겠네요. 아바타는 이모지나 간단한 아이콘 세트로 시작하면 될 것 같아요.

**🎯 기획자**: 완벽해요! 그럼 첫 접속 시 "닉네임 설정 & 아바타 선택" 모달을 띄우고, 게임 로비에서 내 프로필 카드를 볼 수 있게 하는 거죠. 통계는 "이번 세션에서" 같은 문구를 넣어서 임시라는 걸 알려주고요.

---

### 2. 리더보드/랭킹

**💻 프로그래머**: 리더보드는 사용자 프로필보다 더 복잡해요. 전체 사용자의 점수를 저장하고 정렬해야 하는데, 이건 DB 없이는 불가능합니다. MongoDB나 PostgreSQL 같은 걸 도입해야 해요.

**🎯 기획자**: 그럼 세션 기반 리더보드는 어때요? "지금 접속 중인 사용자들 중" 또는 "오늘의 리더보드" 이런 식으로요. 게임마다 다른 랭킹 기준도 있으면 재밌을 것 같아요. 주사위는 총 승수, 경마는 최고 연승, 룰렛은 최고 배당 같은 거요.

**💻 프로그래머**: 오늘의 리더보드는 가능하긴 한데... Railway 무료 플랜은 인스턴스가 재시작되면 메모리가 날아가요. 그래도 Redis 같은 인메모리 DB를 쓰면 하루 단위 캐싱은 할 수 있을 것 같네요. 근데 Redis도 설정이 필요하긴 하죠.

**🎯 기획자**: 일단은 "현재 방 안에서" 리더보드는 어때요? 같은 게임방에 있는 사람들끼리만 순위를 보여주는 거요. 이건 서버 메모리만으로도 가능하잖아요? 사용자들이 경쟁심을 느끼게 하는 게 목적이니까, 작게 시작해도 효과는 있을 거예요.

**💻 프로그래머**: 그건 완전 심플하네요! 각 room 객체에 플레이어별 통계를 tracking하고, 게임 종료 시 소켓으로 리더보드 emit하면 끝이에요. HTML/CSS로 간단한 리더보드 UI만 만들면 되고요. 좋습니다, 이건 당장 할 수 있어요.

---

### 3. 커스텀 이모지/스티커

**🎯 기획자**: 채팅 모듈이 이미 있으니까 여기에 이모지나 스티커를 추가하면 재밌을 것 같아요. 사용자들이 게임 중에 빠르게 반응할 수 있게요. "ㅋㅋㅋ", "ㅠㅠ", "축하해!" 같은 감정 표현이요.

**💻 프로그래머**: 이모지는 쉬워요. 유니코드 이모지 picker 라이브러리 갖다 쓰거나, 직접 만들어도 그리 어렵지 않아요. 근데 "커스텀" 스티커는... 이미지 업로드를 받아야 하나요? 그럼 파일 저장소랑 필터링 시스템이 필요한데요.

**🎯 기획자**: 아뇨, 커스텀은 "우리가 만든" 스티커 팩이에요! 각 게임마다 테마에 맞는 스티커들. 주사위 게임엔 주사위 캐릭터, 경마엔 말 캐릭터 이런 식으로요. 사용자 업로드는 나중 문제고요.

**💻 프로그래머**: 아하! 그럼 완전 간단하네요. assets/stickers/ 폴더에 PNG 파일 몇 개 넣고, 채팅 UI에 스티커 버튼만 추가하면 돼요. 서버는 그냥 스티커 ID만 전달하고 클라이언트에서 렌더링하면 되고요. 디자인만 있으면 바로 붙일 수 있어요.

**🎯 기획자**: 디자인은 Gemini AI한테 프롬프트 주고 생성해볼 수도 있겠네요. 아니면 무료 스티커 팩 찾아보거나요. 좋아요, 이건 빠른 승리감(quick win)을 줄 수 있는 기능이네요!

**💻 프로그래머**: 네, ChatModule.js에 sticker 타입 메시지만 추가하면 될 것 같아요. 기존 채팅 로직 재사용하면 되니까 하루면 충분해요.

---

### 4. 음성 채팅

**🎯 기획자**: 게임 중에 실시간으로 대화할 수 있으면 훨씬 몰입감이 높아질 거예요. 경마 게임 같은 경우는 다같이 응원하면서 보는 게 핵심 재미잖아요. 음성 채팅 기능을 넣으면 어떨까요?

**💻 프로그래머**: 음성 채팅은... 솔직히 난이도 상급이에요. WebRTC로 P2P 구현하거나 미디어 서버가 필요한데, Railway에서 WebRTC 시그널링 서버 돌리는 것도 복잡하고, 대역폭 비용도 엄청 나올 거예요. 외부 서비스(Agora, Twilio) 쓰면 월 요금제가 비쌉니다.

**🎯 기획자**: 흠... 그럼 간단한 "음성 메시지" 기능은 어때요? 카카오톡처럼 녹음해서 보내는 거요. 실시간은 아니지만 텍스트보다는 감정 전달이 잘 되잖아요.

**💻 프로그래머**: 음성 메시지도 파일 업로드/저장/재생 구조가 필요해서 간단하진 않아요. 그리고 녹음된 파일 크기 제한, 악용 방지, 저장소 용량 관리... 생각보다 고려할 게 많아요. 차라리 Discord나 Zoom 링크 공유 버튼을 넣는 건 어때요?

**🎯 기획자**: 그건 너무 UX가 끊기는데... 음, 알겠어요. 음성 채팅은 나중에 사용자가 많아지고 수익 모델이 생기면 그때 투자하는 걸로 하죠. 지금은 보류하겠습니다.

**💻 프로그래머**: 현실적인 판단이네요. 대신 텍스트 채팅을 더 강화하는 게 나을 것 같아요. 빠른 답장 버튼(quick reply)이나 채팅 이펙트 같은 걸로요.

---

### 5. 친구/즐겨찾기 시스템

**💻 프로그래머**: 친구 시스템은 계정 시스템이 전제되어야 해요. 지금은 세션 기반이라 같은 사람인지 알 방법이 없거든요. DB에 사용자 ID가 있어야 친구 관계를 저장할 수 있어요.

**🎯 기획자**: 그럼 "즐겨찾기"로 가볍게 시작하면 어떨까요? 같은 방에 있는 사람 중에 마음에 드는 사람을 즐겨찾기 하면, 그 사람이 다시 들어왔을 때 알림을 주는 거요. localStorage에 닉네임 리스트만 저장해도 되잖아요.

**💻 프로그래머**: 그건 가능하긴 한데... 닉네임은 중복될 수 있어서 신뢰성이 떨어져요. 대신 세션 ID 기반으로 "이 세션 동안만" 즐겨찾기하는 건 가능해요. 같은 방에 다시 들어오면 "즐겨찾기한 사용자가 있습니다" 표시하는 식으로요.

**🎯 기획자**: 흠, 그건 너무 제한적인데요. 차라리 간단한 계정 시스템을 먼저 만드는 게 나을 것 같아요. 닉네임 + 비밀번호 4자리 정도로 초간단 로그인을 구현하면 어때요? 그럼 친구 시스템도 만들 수 있고요.

**💻 프로그래머**: 비밀번호 저장하면 보안 이슈가 생겨요. 해싱, HTTPS, 세션 관리... 생각보다 복잡해요. 차라리 Discord/Google OAuth를 붙이는 게 안전하고 빨라요. passport.js 쓰면 하루 이틀이면 되고요.

**🎯 기획자**: 오, OAuth는 좋네요! 요즘 사람들 귀찮아하니까 소셜 로그인이 낫죠. 그럼 OAuth 인증 추가하고, 그 다음에 친구 시스템 만드는 걸로 단계를 나눠요. 우선순위는 중간 정도로 두고요.

**💻 프로그래머**: 좋습니다. 친구 시스템은 "Phase 2" 기능으로 분류하죠. DB 스키마에 users, friends 테이블 미리 설계해두면 나중에 빠르게 붙일 수 있어요.

### 6. 게임 내 투표/설문

**🎯 기획자**: 게임 시작 전에 간단한 투표를 할 수 있으면 재밌을 것 같아요. "다음 게임 뭐 할까?", "배팅 시간 늘릴까요?", "BGM 끌까요?" 같은 거요. 민주적으로 결정하는 느낌이 들어서 참여도가 높아질 것 같아요.

**💻 프로그래머**: 투표 기능은 구조는 간단해요. 투표 시작하면 모든 클라이언트에 poll 객체를 보내고, 사용자가 선택하면 서버로 vote 이벤트 받아서 집계하는 거죠. Socket.IO로 실시간 결과 업데이트도 쉽고요.

**🎯 기획자**: 그럼 쉬운 거네요? 왜 망설이세요?

**💻 프로그래머**: UI/UX가 문제예요. 투표 모달을 띄우면 게임이 중단되는데, 타이밍을 잘못 잡으면 게임 흐름을 방해해요. 그리고 누가 투표를 시작할 권한이 있는지, 악용하는 사람은 어떻게 막는지... 디테일한 룰이 필요해요.

**🎯 기획자**: 아, 그럼 "방장만" 투표를 시작할 수 있게 하고, 투표는 게임 시작 전이나 게임 종료 직후에만 가능하게 제한하면 어떨까요? 선택지도 3개까지만 제한하고요.

**💻 프로그래머**: 그럼 괜찮네요. ready-shared.js에 방장 권한 로직이 이미 있으니까 재사용하면 되겠어요. 투표 UI도 간단하게 버튼 3개 정도면 되고요. 구현 난이도는 중하 정도예요.

**🎯 기획자**: 좋아요! 그럼 간단한 버전으로 시작해서, 나중에 커스텀 질문이나 다양한 투표 타입을 추가하는 걸로 해요. 일단은 "다음 게임 선택" 투표만 만들어도 충분히 유용할 것 같아요.

**💻 프로그래머**: 네, 동의합니다. 빠르게 만들어서 사용자 반응 보고 개선하는 게 나을 것 같네요.

## 📊 우선순위 합의표

| 우선순위 | 기능 | 난이도 | 개발 기간 | 사용자 가치 | 비고 |
|---------|------|--------|----------|-----------|------|
| **P0** | 프로필 시스템 (게스트) | 하 | 2-3일 | 높음 | localStorage 기반, 닉네임+아바타+통계 |
| **P0** | 커스텀 스티커 | 하 | 1-2일 | 중 | ChatModule 확장, 게임별 스티커 팩 |
| **P1** | 방 내 리더보드 | 하 | 1일 | 중 | 현재 방 참여자만 대상 |
| **P1** | 게임 내 투표 | 중하 | 2-3일 | 중 | 방장 권한, 게임 전후만 가능 |
| **P2** | OAuth 로그인 | 중 | 3-5일 | 높음 | Discord/Google 연동 |
| **P2** | 친구 시스템 | 중 | 4-5일 | 높음 | OAuth 이후 구현 가능, DB 필요 |
| **P2** | 전체 리더보드 | 중상 | 5-7일 | 중 | DB 또는 Redis 필요 |
| **보류** | 음성 채팅 | 상 | 2주+ | 높음 | 비용/복잡도 높음, Phase 3 검토 |

### 우선순위 기준

- **P0 (즉시 시작)**: 낮은 난이도 + 빠른 구현 + 기반 기술
- **P1 (다음 스프린트)**: 중간 난이도 + 명확한 가치
- **P2 (중기 계획)**: 인프라 투자 필요 + 장기적 가치
- **보류**: 현재 리소스로 부적합

### 1차 스프린트 추천 (1-2주)

1. **프로필 시스템** - 사용자 정체성 기반 마련
2. **커스텀 스티커** - 채팅 경험 개선
3. **방 내 리더보드** - 경쟁 요소 추가

→ **총 예상 기간**: 4-6일
→ **누적 효과**: 사용자 재방문율 ↑, 세션 시간 ↑
