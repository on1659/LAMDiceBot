# 사다리타기 — 막대기 배치하며 레인도 같이 고르기 — Goal

사다리타기(ladder)에서 **레인(출발 위치) 선택을 별도 단계로 두지 말고, 막대기를 놓는 빌드 단계에서 함께 고르도록** 합쳐라. feature/goal-game-mode 브랜치에서 작업한다.

> 전제: 직전에 완료된 "유저 빌드 맵"(준비 → 막대기 직접 배치 → reveal) 위에 얹는 변경이다.
> baseline 명세: `docs/meeting/applied/`로 이동될 `2026-06-03-ladder-user-built-map-impl.md` 참고.

## 왜 (문제)

- 현재 흐름: ① 준비 → 막대기 배치(빌드) → ② 호스트 시작 → **별도 레인 선택 단계**(`selecting`) → ③ 공개.
- 빌드를 다 끝낸 뒤에야 "내가 어느 레인을 쓸지"를 따로 고르는 흐름이 어색하다. 막대기를 놓는 그 순간에 내 출발 레인도 같이 정하고 싶다.

## 핵심 규칙 (바뀌는 것)

- **레인 선택을 빌드 단계(phase `idle`)로 통합한다.** 막대기를 놓는 같은 화면에서, 준비한 사람이 자기 출발 레인(맨 위 칸)을 직접 1개 고른다.
- 레인은 **중복 불가**(1인 1레인). 점유한 레인에는 본인 이름/색이 표시되고, 모두에게 실시간으로 보인다(막대기와 동일한 가시성).
- 막대기 배치와 레인 선택 **둘 다 선택 요소**다. 막대기는 기존대로 안 놓아도 되고, 레인도 빌드 도중엔 비워둘 수 있다.
- **별도 레인 선택 단계(`selecting`)를 제거한다.** 호스트가 시작을 누르면 곧바로 공개(reveal)·추적으로 간다. 시작 후 따로 레인을 고르는 화면은 없다.
- **시작 조건**: 호스트는 준비 인원이 충분하면, 막대기를 안 놓았거나 레인을 안 고른 참가자가 있어도 시작할 수 있다(강제 안 함). 미선택 레인 처리 방식은 아래 "결정/기본값" 참조.
- **소유권/취소**: 준비 취소·퇴장·연결해제 시, 그 사람이 놓은 막대기뿐 아니라 **그 사람이 점유한 레인도 함께 해제**된다.
- **N 동적 변동 대응**: 컬럼 수 N = 준비 인원. 인원이 줄어 N이 작아지면 범위를 벗어난 막대기(c > N-2)와 **범위를 벗어난 레인 점유(lane ≥ N)도 함께 정리(트림)**한다.

## 결정/기본값 (구현자가 근거와 함께 바꿔도 됨 — 막힘 기준 위임분)

- **미선택 레인 자동 배정**: 빌드 중 레인을 안 고른 준비자는, 호스트 시작 시 남은 빈 레인 중 하나를 서버가 자동 배정한다(막대기 "미배치 허용" 철학과 동일하게, 레인 미선택도 시작을 막지 않는다). — 이 기본값이 어색하면 "전원 레인 선택 완료 시에만 시작 가능"으로 바꿔도 된다. 단 둘 중 무엇이든 시작이 영구히 막히지 않게 하라.
- **자동 배정 공정성**: 자동 배정은 결과를 사람이 예측·악용할 수 없어야 한다(서버 RNG로 남은 레인 셔플 배정). 클라 Math.random은 deviceId/tabId 외 0회 유지.
- **UI**: 빌드 화면 상단에 N개 레인 헤더(클릭해 내 레인 점유/해제), 그 아래 기존 막대기 격자. 한 화면에서 레인·막대기를 모두 조작. 모바일/PC 모두 대응.

## 가시성/공정성 (유지)

- 빌드 중 가시: 유저 막대기 + 유저 레인 점유. 둘 다 모두에게 실시간 공개.
- 숨김 유지: 서버 숨은 기본 막대기·바닥 꽝 위치·매핑은 **reveal 전까지 클라 전송 금지**. 재진입 응답의 server-only 마스킹(`ladder: undefined`)도 그대로 유지.
- 결과는 서버에서만 결정, 클라는 시각화만. 클라 추적 알고리즘과 서버 매핑 일치 유지.

## 작업 방식

- 먼저 현재 `socket/ladder.js`(`ladder:start`/`selectStart`/`ladder:pick`/`userLanes`/`reveal`), `js/ladder.js`(빌드 그리드·`ladder:rungsUpdated`·레인 선택 UI·`buildPath`), `css/ladder.css`, `ladder-multiplayer.html`(빌드/선택 영역), `socket/shared.js`(준비취소 ladder 분기), `socket/rooms.js`(leaveRoom/재진입 정리·마스킹), `utils/room-helpers.js`(ladder gameState)를 조사해 "어디를 합치고 어디를 지울지" 파악해라.
- 신규/변경 소켓 계약(제안 — 구현자가 더 단순한 안 채택 가능):
  - `ladder:pickLane { lane }` — phase `idle`, 준비자만, 범위·중복 검증, 1인 1레인(재선택 = 이동). 비점유 클릭 시 해제.
  - 브로드캐스트: `ladder:rungsUpdated`에 `userLanes` 포함(또는 별도 `ladder:lanesUpdated`)로 레인 점유를 실시간 동기화.
  - `ladder:start` → 미선택 레인 자동 배정 후 곧바로 reveal 흐름. **`selectStart` 단계/이벤트 제거**(또는 빈 패스스루).
- 준비취소·leaveRoom·disconnect·N 트림 경로에서 `userRungs`와 동일하게 `userLanes`(레인 점유)도 정리하고 동기화 emit 하라(`socket/shared.js`·`socket/rooms.js`·ladder 핸들러).

## 게임성 기준

- 한 판이 더 짧고 매끄럽다(별도 선택 단계 제거 → 클릭 흐름 단축). 막대기·레인을 같은 화면에서 정하는 "내 수를 한 번에 둔다"는 감각. 운(숨은 기본 막대기+꽝)과 선택(막대기 위치+레인) 모두 유지. 꽝=패자 즉시 이해. 매 판 새 기본 틀로 반복성 유지. 완료 보고에 재미/선택/긴장/반복 자체 평가 포함.

## 리소스 기준

- 기존 리소스/사운드 우선 재사용(레인 선택음은 기존 `ladder_*`/`common_*`). 신규 mp3/이미지는 꼭 필요할 때만 최소로, 기존 파일명 변경 금지. 새 리소스는 update-log에 기록.

## 기존 통합 유지/갱신 (스킵 금지)

- 통계(`db/stats`, `/api/statistics`), 랭킹(`db/ranking` getFullRanking + RankingModule ladder 탭) 계속 동작.
- 튜토리얼: 합쳐진 흐름에 맞게 `LADDER_TUTORIAL_STEPS` 갱신(레인 선택을 빌드 단계 설명으로 통합). 최초 1회 자동 노출 + 도움말 버튼 동작.
- 사운드: 레인 점유/막대기 배치/공개/결과 등 핵심 순간 재생 호출 연결, 로드 실패 없음.

## 테스트

- `tests/test-ladder.js`(Playwright)를 합쳐진 흐름에 맞게 갱신: 앱 실행 → 방 생성/입장(2탭) → 준비 → 빌드 화면에서 **각자 레인 점유 + 막대기 1개 배치(브로드캐스트 동기화)** → 준비 취소 시 본인 막대기·레인 모두 해제 확인 → (한 명은 레인/막대기 미선택) 호스트 시작 → **별도 선택 단계 없이** 공개/추적 → 결과/종료 도달 → 콘솔 에러 없음.
- 통계 집계, 랭킹 ladder 섹션, 튜토리얼 자동 노출 검증. 기존 대표 게임(경마) 선택/시작/진행 가능 확인.

## 완료 기준 (하나라도 미완이면 완료 아님)

- 빌드 단계에서 막대기 배치와 레인 선택을 **한 화면에서 함께** 할 수 있고, 별도 레인 선택 단계가 사라졌다.
- 레인 중복 불가, 준비 취소/퇴장 시 본인 막대기·레인이 모두 사라지고, N 축소 시 범위 초과 막대기·레인이 트림된다.
- 호스트 시작 → 공개/결과/종료 흐름이 실제로 동작한다(미선택자 자동 배정 또는 채택한 대안대로).
- 꽝/서버 숨은 막대기가 reveal 전 클라에 노출되지 않는다(재진입 마스킹 포함). 클라 Math.random 게임결과 0회.
- 통계/랭킹/튜토리얼/사운드가 모두 동작한다.
- 기존 대표 게임모드 1개 이상이 깨지지 않았음을 확인했다.
- update-log.md에 기존 형식대로 기록.
- 마지막 보고에 룰 요약, 변경 파일, 테스트 명령, 테스트 결과, 게임성 자체 평가, 새 리소스 여부, 남은 이슈 포함.
