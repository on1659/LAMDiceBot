# 회전 칼날(spin-arena) — 칼 수집 탈출 + 부활 메커니즘 재설계 (office-hours 우선) — Goal

방금 적용한 "탈출 리프레이밍"(생존=벌칙을 서사만 뒤집은 안)이 아직 **"이겼는데 지는 느낌"**을 완전히 해소하지 못했다. 사용자가 더 능동적인 승리 구조를 제안한다: **데미지 누적으로 칼이 늘고, 칼 N개를 모으면 탈출**. 죽음은 영구 탈락이 아니라 **비석 → 3초 후 부활**의 일시적 다운으로 바꾼다. 이건 시뮬레이션 핵심을 바꾸는 재설계이므로 **바로 코딩하지 말고 `/office-hours`(필요시 `/meeting-codex`)로 설계를 먼저 발전**시킨 뒤 구현한다. 브랜치: `feature/spin-arena-rework`.

> 전제/baseline: 직전 상태 = "탈출 리프레이밍" 적용·푸시 완료 (커밋 `a5d7a9f`, `docs/dev-cycle/2026-06-11-spin-arena-win-condition-review.md` / `-escape-reframe-instructions.md`). 현재 시뮬: HP 100 감소만(회복 없음), 죽으면 영구 탈락(부활 없음), 칼날은 **킬** 트리거로 +1(상한 5). 기준 파일: `socket/spin-arena.js`(simulate), `js/spin-arena.js`, `spin-arena-multiplayer.html`, `css/spin-arena.css`.

## 한 줄 요약

① 능동 승리 구조 = **데미지 누적 → 칼 +1 → 칼 N개 모으면 탈출**(생존/HP 기반 탈출을 칼 수집으로 교체) ② 죽음 = **비석 + 3초 후 부활**(영구 탈락 폐지, 일시 다운) ③ "HP가 다시 차는 느낌"·"이겼는데 지는 느낌" 제거 ④ **종료 압축** — 결판 나면 빈 시간 없이 끝낸다(고정 30초 dead air 제거) ⑤ **미션/달성률 HUD** — "지금 뭘 해야 하나(칼 N개)"와 진행도(예 3/5)를 화면에 표시 ⑥ **착수 전 office-hours로 설계 확정**.

## 0. 작업 순서 — 설계 먼저 (스킵 금지)

- **Phase 0 (필수): `/office-hours` 빌더 모드**로 아래 제안을 발전시킨다. 막판 결판 구조, 칼 수집 목표치 N, "데미지"의 정의(받은/가한), 부활 악용 방지, 30초 길이 적합성 등 열린 질문을 정리하고 **설계 결정서**를 `docs/dev-cycle/2026-06-12-spin-arena-blade-escape-design.md`로 남긴다. 필요하면 `/meeting-codex`로 보강.
- Phase 0에서 "이 방향이 현 게임 정체성을 깬다/30초 관전형과 안 맞는다"는 결론이 나오면, 구현하지 말고 대안과 함께 사용자에게 보고한다.
- Phase 1 (구현) 은 설계 결정서가 확정된 뒤에만 시작한다. 트리아지 **COMPLEX**(시뮬 변경 + 밸런스 재튜닝 + 공정성).

## 핵심 규칙 (제안 — Phase 0에서 확정)

### 1. HP가 "다시 차는 느낌" 제거
- 현재 시뮬엔 HP 회복이 없다(hp는 감소만). 사용자가 느낀 "HP가 다시 참"은 **탈출 게이지(100−HP) 표시 / 칩 잔상** 인상일 가능성이 크다 — Phase 0에서 원인을 먼저 확인한다.
- 새 메커니즘에선 게이지의 의미를 **칼 수집 진행도**로 재정의해 "체력이 오르락내리락하는" 인상을 없앤다(아래 2·3과 연동). 회복/충전 개념을 쓸지 말지 설계에서 명확히 결정.

### 2. "이겼는데 지는 느낌" 제거 — 능동 승리 목표
- 문제: 리프레이밍 후에도 "그냥 버티다 보니 탈출"이라 능동적 성취감이 약하다.
- 방향: **칼 N개를 모으면 탈출 성공**(명확한 목표 달성형). 끝까지 칼을 못 모은 최후 1인 = 당첨(벌칙). 당첨 1명을 뽑는 사이트 공통 목적은 유지.

### 3. 데미지 누적 → 칼 +1, 칼 N개 = 탈출 (핵심 신메커니즘)
- 현재 "킬당 칼 +1(상한 5)"을 **"데미지 누적 임계 N당 칼 +1"**로 교체 검토. 칼이 목표 개수에 도달하면 그 캐릭터는 **탈출**(아레나에서 빠져나감).
- **열린 질문(Phase 0 확정 대상)**:
  - "데미지"가 **받은 데미지**인가 **가한 데미지**인가? (현 탈출 게이지는 "맞을수록 참"이라 받은 데미지 쪽과 정합. 단 칼=공격 수단이라 가한 데미지 쪽이 직관적일 수도 — 설계에서 택1.)
  - 칼 임계 개수 N(예: 5), 데미지 임계량, 칼 상한과 탈출 트리거의 관계.
  - 탈출한 캐릭터는 시뮬에서 빠지는가(더 못 때림) → 남은 인원의 결판 가속 구조.
  - 전원이 거의 동시에 N 도달하는 교착/동시 탈출 엣지의 결정론 tie-break.

### 4. 죽음 = 비석 + 3초 후 부활 (영구 탈락 폐지)
- HP 0 → **그 자리에 비석 드롭 + 3초 다운 + 부활**(HP/위치 리셋 규칙은 설계에서). 부활 후 다시 칼 수집을 이어간다. 죽음은 일시 페널티이지 게임 아웃이 아니다.
- **열린 질문**: 부활 시 칼 수집 진행도 유지/감소 여부(악용 방지 — "일부러 죽어 이득" 금지), 부활 무적 시간, 30초 안에 결판나도록 부활 횟수/타이밍 한계. 다운 동안 칼 회전·판정 정지.

### 5. 종료 조건 압축 — 빈 시간 제거 (직전 "안 끝남" 피드백 흡수)
- 현재 문제(확인됨): 시뮬은 **항상 고정 30초**(`GAME_MS`)라 결판이 8~17초에 나도 남은 시간 동안 혼자 빙빙 도는 **dead air**가 길다. 서버 종료는 reveal + 카운트다운4s + 30s + 결과2.2s = **36.2초** 고정.
- 요구: **결판이 나면 즉시(짧은 결판 연출 후) 끝낸다.** 칼 수집 구조에선 "나머지가 전부 탈출하고 최후 1인(당첨)만 남는 순간"이 자연 종료점 → 그 시점 + 결판 비트(예 1.5~2.5초)까지만 재생하고 결과로 넘어간다.
- **결정론·2탭 동기 핵심(설계 필수)**: 종료 시각을 클라가 임의 판단하면 탭마다 어긋난다. **서버 시뮬이 "결판 시각(decideMs)"을 계산해 reveal 페이로드에 넣고**, 클라 리플레이는 `decideMs + 결판비트`까지만 재생. 서버 `endTimeout`도 고정값 대신 **`COUNTDOWN_MS + decideMs + 결판비트 + RESULT_HOLD_MS`** 로 가변화. frames는 결판 시점까지만 잘라 보내도 됨(페이로드 축소 — additive 아님이므로 클라 길이 의존부 점검).
- 안전장치: 아무도 N칼을 못 모으는 교착이면 **상한 30초**에서 강제 종료(현재 길이를 캡으로). 부활 중 종료 엣지 처리.

### 6. 미션 표시 + 달성률 UI ("지금 뭘 해야 하나" + 진행도)
- 목적: 관전자가 "이 게임의 목표가 뭐고, 누가 얼마나 달성했나"를 한눈에 — 능동 승리 구조(②)를 화면으로 뒷받침.
- **미션 텍스트**: 상단/상태바에 "⚔️ 칼 N개 모아 탈출!" 고정 안내(카운트다운·시작 시점). 튜토리얼·결과와 문구 일치.
- **달성률 표기 후보(office-hours에서 택1~2 — 같이 고민 대상)**:
  - (a) **칼 핍(pips)**: 캐릭터 머리 위 `◇◇◆◆◆` 식으로 N칸 중 채운 수 표시(가장 직관, 모바일도 또렷). 현재 HP 게이지 자리 대체.
  - (b) **원형 진행 링**: 캐릭터 둘레를 progress ring이 한 바퀴 채워지며 = 탈출 임박. 회전 칼날 테마와 시각적 일치.
  - (c) **미션 패널(현 HP 패널 전환)**: 이름 + `칼 3/5` + 상태(전투중/부활중 3·2·1). 직전 작업의 HP 현황 패널을 "미션 진행 패널"로 재활용.
  - (d) **임박 강조**: N−1 도달 시 글로우/펄스 + 효과음으로 "곧 탈출" 텐션.
- **부활 상태 표시**: 비석 아이콘 + 머리 위 **3·2·1 카운트다운**, 부활 순간 플래시. 다운 캐릭터는 미션 진행 정지 표시.
- **당첨자(미달자) 표기**: 끝까지 칼 못 모은 1인은 진행도 막대가 안 찬 채 강조(적색) — "탈출 실패(당첨)" 결과와 일관.
- 전부 **리플레이 t·페이로드 파생**(2탭 동일). 칼 수/부활 상태는 서버 시뮬 출력에서 결정론 산출.

## 공정성 (절대 불변)
- 결과는 **서버 결정론 시뮬에서만** 결정(시드 PRNG, 동일 시드 = 동일 frames/eliminations/result). 칼 수집·부활·탈출 판정 전부 시뮬 내부.
- 클라 `Math.random` 결과 영향 0회(deviceId/tabId·cosmetic 외). 비석/부활/칼 카운트 연출은 t·페이로드 파생(2탭 동일).
- reveal 전 server-only(timeline/result/seed) 비노출·재진입 마스킹 유지. 페이로드 변경은 additive 우선.

## 기존 통합 유지 (스킵 금지)
- 통계(recordGamePlay)/서버 랭킹(recordServerGame·recordGameSession)/사운드/스킨/튜토리얼/채팅 오버레이/카운트다운/BGM/다시보기 계속 동작.
- selected(당첨=벌칙)=DB rank 2/isWinner=false 시맨틱 유지. 당첨 = "끝까지 탈출 못 한 1인"으로 의미만 갱신.
- 소켓 이벤트 이름 유지. 튜토리얼·메타·update-log 문구를 새 규칙(칼 수집/부활)에 맞게 갱신.

## 막힘 기준
- Phase 0에서 "30초 무입력 관전형에 칼수집/부활이 안 맞는다"는 결론 → 구현 말고 대안 보고.
- 데미지 정의(받은/가한)·칼 임계 N·부활 규칙 등 핵심 수치는 배치 분포가 권위 — 기준 충족하는 가장 단순한 조합 선택 후 근거 보고.
- 부활 악용(일부러 죽어 이득)이 설계로 막히지 않으면 멈추고 사용자 결정 요청.
- 종료 압축에서 클라/서버 종료 타이밍이 어긋나면(2탭 desync) **서버 decideMs를 페이로드에 실어 클라가 그대로 따르는** 가장 단순·안전한 쪽으로 정하고 보고. frames 절단이 클라 길이 의존부를 깨면 절단하지 말고 decideMs까지만 재생하는 방식으로 후퇴.
- 미션/달성률 표기 방식(핍/링/패널)은 모바일 가독성 우선으로 택1, 나머지는 보고에만 기록.
- 테스트 불가 시 구현은 완료하되 막힌 지점 + 수동 QA 체크리스트.
