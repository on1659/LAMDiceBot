# 경마 꾸미기 상점 — 구매 UX 폴리시 (Phase 2) — Goal

경마 꾸미기 상점(이미 구현·배포됨)의 **구매 경험을 다듬는다**. 브랜치 `feature/goal-game-mode`. 세 가지: ① 구매 확인/성공 팝업이 상점 모달 뒤에 가려 안 보이는 문제 + 확인 단계 추가, ② 코인 차감 연출, ③ 트랙테마/결승연출 미리보기 설계. **전부 클라이언트 UI(`js/horse-shop.js` + `css/horse-shop.css`)만 손대고, DB·소켓·공정성·게임 결과와 무관**한 순수 시각/UX 작업이다.

> 전제/baseline: 상점+코인지갑(인증/구매 `shop:buy`/장착 `shop:equip`/방연출 broadcast)은 직전 세션에서 구현 완료. 서버 계약은 그대로 두고 그 위에 UX만 얹는다. 구현 명세(참조): `docs/meeting/impl/2026-06-07-horse-shop-impl.md`. 상점 모듈: `js/horse-shop.js`(전역 `HorseShop`).

## 한 줄 요약

① **구매 확인+성공 팝업을 상점 위 레이어로** (현재 뒤에 가려짐) → ② **코인 차감 연출**(빨간 −금액 플로팅 + 잔고 감소) → ③ **트랙테마/결승연출 미리보기** 설계·구현.

## 핵심 규칙 (번호 절)

### 1. 구매 확인 + 성공 팝업 (레이어 수정 — 가장 먼저)

- **문제(확인됨):** 지금 `shop:buy`는 확인 없이 바로 구매되고, 결과 알림 `showCustomAlert`/`showCustomConfirm`의 오버레이 z-index가 **10000**(`js/horse-race.js:5738`, `:5781`, `:5812`)인데 상점 `.hshop-overlay`는 **z-index 12000**(`css/horse-shop.css`) → **팝업이 상점 모달 뒤에 떠서 안 보인다.**
- **원하는 동작:**
  - 구매 버튼 클릭 → **"정말로 구매하실래요?"** 확인 팝업(아이템명·가격 표시) → 확인 시에만 `shop:buy` emit.
  - 구매 성공 → **"구매했습니다"** 알림(또는 가벼운 토스트). 실패(잔고부족/소유/네트워크)도 상점 위에서 명확히 보이게.
  - 이 확인/알림이 **상점 모달보다 위 레이어**로 보여야 한다.
- **결정 위임(막힘 기준 참조):** 기존 전역 `showCustomConfirm`/`showCustomAlert` 오버레이 z-index를 상점 위(예: 12500~13000)로 올릴지, 아니면 **상점 전용 인앱 확인/토스트**(`.hshop-overlay` 안에 더 높은 z-index 레이어)를 새로 만들지는 구현자가 판단. 전역 z-index를 올리면 다른 게임 팝업에도 영향이 가니 **회귀 주의** — 상점 전용 레이어가 더 안전할 수 있음(이 경우 horse-shop.js 자체에 확인/토스트 구현).
- 모바일/PC 양쪽에서 확인 버튼 누르기 쉬운 크기.

### 2. 코인 차감 연출

- 구매로 잔고가 줄 때 코인 UI(`#hshopBalance`)에서 **빨간색 "−50"** 같은 사용 금액이 떠오르며 사라지고, **잔고 숫자가 줄어드는** 연출(플로팅 −금액 + 숫자 카운트다운 또는 플래시).
- 현재 `updateBalanceLabel()`은 텍스트만 즉시 교체 → 차감 시 전/후 값 차이를 받아 애니메이션.
- **순수 시각 연출.** 잔고 값 자체는 서버 응답(`shop:buy` 콜백 `balance`)이 권위 — 연출은 그 값으로 수렴만 한다(연출이 잔고를 임의로 바꾸지 않음).
- 적립(`wallet:updated`)으로 잔고가 **늘 때**의 연출(초록 +금액)도 같이 넣을지는 선택(여유 되면).

### 3. 트랙테마 / 결승연출 미리보기 설계

- **현재:** 트랙테마 카드는 그라데이션 배경(`item.bg`)만, 결승연출은 이모지(🎆)만. 직전에 추가한 탈것 미리보기(`buildItemPreview`)는 도색/트레일/액세서리/마번 같은 **탈것에 입히는** 슬롯 전용이라 방 연출엔 안 맞는다.
- **원하는 것:** 트랙테마가 **방 전체 배경으로 적용된 모습**을 카드에서 직관적으로 보여주기. 예시 방향(구현자가 택1·혼합):
  - 미니 트랙 썸네일: 그라데이션 배경 + 바닥선/지평선 + 작게 달리는 탈것 실루엣(샘플 스프라이트 축소) 1~2개.
  - 카드 호버/탭 시 더 큰 미리보기 패널에 트랙테마를 깔고 샘플 탈것이 지나가는 모습.
- **결승연출(finish_fx) 미리보기:** 카드에 폭죽/색종이가 작게 터지는 루프 또는 정지 프레임.
- 새 이미지 에셋 없이 기존 `getVehicleSVG`(전역, `js/horse-race-sprites.js`) + CSS/이모지로. 무겁지 않게(카드 19개 렌더 성능 고려).

## 공정성 (UI 작업이지만 명시)

- 이 작업은 **게임 결과·서버 계약과 완전히 무관**한 상점 UI다. `socket/horse.js`(결과 계산/적립)·`shop:buy`/`shop:equip` 서버 핸들러는 **건드리지 않는다**.
- 잔고/소유/장착의 **권위는 서버 응답**. 연출은 서버가 준 값으로 수렴만 하고, 클라가 잔고·소유를 임의 변경 금지.
- 연출용 파티클 지터 등에 `Math.random`은 외관 효과 한정 허용(프로젝트 관례). 게임 결과 경로엔 0.

## 기존 통합 유지 (스킵 금지)

- 기존 구매/장착/방연출/적립 흐름, 탈것 미리보기(`buildItemPreview`), 게스트 차단, `wallet:updated` 잔고 갱신 모두 그대로 동작.
- 전역 `showCustomConfirm`/`showCustomAlert`를 수정한다면 **다른 게임(주사위/룰렛/사다리/다리건너기)의 팝업 회귀 확인** 필수 — 그래서 상점 전용 레이어가 더 안전할 수 있음(절 1 참조).
- `.hshop-*` 클래스 외 공유 CSS·다른 게임 z-index 레이어 침범 금지.

## 작업 방식

- 먼저 읽기: `js/horse-shop.js`(`renderCard`/`buildItemPreview`/`doBuy`/`updateBalanceLabel`/`openShop`), `css/horse-shop.css`(`.hshop-overlay` z-index 12000, `.hshop-balance`, `.hshop-thumb`, `.hshop-preview*`), `js/horse-race.js`의 `showCustomAlert`(:5735~)·confirm(:5781/:5812).
- 모바일·PC 양쪽 대응을 계획 단계부터.
- 클라이언트 파일만 바꾸므로 **서버 재시작 불필요**, 페이지 새로고침으로 검증.

## 테스트

- 수동: 상점에서 구매 클릭 → 확인 팝업이 **상점 위**에 보임 → 확인 → 코인 차감 연출(빨간 −금액 + 잔고 감소) → "구매했습니다" 알림. 취소 시 미구매. 잔고부족/소유 시 알림 보임.
- 트랙테마/결승연출 카드 미리보기가 의도대로 보이는지(모바일/PC).
- 회귀: 다른 게임(경마 본 게임 포함) 팝업·레이어 정상. 구매/장착/적립 서버 동작 미파손.

## 완료 기준 (하나라도 미완이면 완료 아님)

- 구매 시 확인 팝업이 **상점 모달 위**에 보이고, 확인해야 구매됨. 성공/실패 알림도 상점 위에 보임.
- 코인 차감 시 빨간 −금액 + 잔고 감소 연출 동작(잔고 최종값은 서버 응답과 일치).
- 트랙테마(및 결승연출) 카드 미리보기가 그라데이션만 있던 것보다 직관적으로 개선.
- 전역 팝업을 손댔다면 크로스게임 회귀 없음(상점 전용 레이어면 해당 없음).
- `update-log.md` 기록. 새 리소스(이미지/사운드) 여부 명시(없을 예정).
- 마지막 보고에 변경 요약·파일·테스트 명령/결과·자체 평가·남은 이슈 포함.

## 막힘 기준

- 팝업 레이어 방식(전역 z-index 상향 vs 상점 전용 레이어)이 애매하면 **회귀 위험이 작은 상점 전용 레이어**를 기본 채택하되 근거와 함께 보고.
- 차감 연출의 구체 수치(지속시간/이징/카운트다운 속도)는 기존 `css/horse-shop.css` 톤에 맞춰 합리적으로 정하고 보고.
- 트랙테마 미리보기 형태가 여러 안으로 갈리면 **가장 가벼운 안(미니 트랙 썸네일)**부터 구현하고, 더 큰 패널은 후속으로 분리 제안.
