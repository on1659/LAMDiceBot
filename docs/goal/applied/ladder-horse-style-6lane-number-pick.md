# 사다리타기 경마화 — 6레인(1~6) + 번호 고르기 + 동시 출발/도착 — Goal

사다리타기(ladder)를 "경마 느낌"으로 다듬어라. 레인을 **항상 번호 1~6 고정**으로 두고, 참가자는 경마의 탈것 선택처럼 **자기 번호(1~6 중 하나)를 고른다.** 공개(reveal)는 기존의 "한 명씩 순차 하강" 대신 **모든 토큰이 동시에 출발해서 동시에 도착하는 느낌**으로 바꾼다. 현재 `feature/ladder-rung-color-speed` 브랜치(또는 새 작업 브랜치)에서 이어서 작업한다.

> 전제/baseline: 현재 사다리는 최대 8명·레인 선택·"순차 하강 reveal"(`socket/ladder.js` `LADDER_DESCENT_*` / `ladderRevealDelay`)·유저가 막대기를 직접 놓는 **빌드 맵** 메커니즘(docs/goal/applied/ladder-user-built-map.md)을 쓴다. 이 baseline 위에 아래 3가지만 얹는다 — 빌드 맵·레인 선택 등 기존 메커니즘은 **그대로 유지**한다.

## 한 줄 요약

① 레인을 **항상 1~6 6개 고정**(인원이 6 미만이면 일부 번호는 주인 없는 빈 레인) → ② 참가자는 **경마식 번호 고르기 UI**로 1~6 중 자기 번호를 선택(중복 불가) → ③ reveal을 **순차 하강에서 "동시 출발 → 동시 도착 느낌"**으로 교체. 유저 빌드 막대기는 유지.

## 핵심 규칙

### 1. 레인 항상 6개(번호 1~6) 고정
- 레인 수를 참가자 수와 무관하게 **항상 6 고정**으로 한다. 각 레인은 번호 1~6으로 표기한다.
- 인원이 6 미만이면 고르지 않은 번호는 **주인 없는 빈 레인**으로 남는다(봇/제외 처리 없이 시각적으로 비어 있음). 빈 레인도 사다리/막대기/도착부 매핑에는 정상 포함된다(번호 고르기·결과 추적은 6열 기준).
- 최소 시작 인원은 기존 정책 유지(빈 레인 허용). 최대는 6으로 내린다(현재 `LADDER_MAX_PLAYERS=8` → 6). 관련 상수는 `socket/ladder.js` 상단 상수 블록 + 클라 동기 지점에서 함께 조정.

### 2. 경마식 번호 고르기 (1~6)
- 참가자는 경마의 탈것 선택과 **동일한 UX 흐름**으로 1~6 중 자기 번호를 1개 고른다(중복 불가, 이미 선택된 번호는 비활성). 기존 "레인 선택"을 이 번호 고르기로 표현/연결한다 — 내부적으로 레인=번호.
- 막대기 배치(빌드 맵)와 번호 고르기는 **둘 다 유지**한다. 둘의 순서/타이밍은 기존 ladder 흐름(준비 → 막대기 배치 → 레인/번호 선택 → 공개)을 따른다.
- 호스트 시작 조건은 기존 유지(번호 미선택자/막대기 미배치자 있어도 시작 가능 여부는 기존 정책 그대로).

### 3. 동시 출발 → 동시 도착 느낌 reveal
- 현재의 "한 명씩 차례로 내려가는" 순차 하강(`LADDER_DESCENT_BUDGET`/`_MIN`/`_MAX`/`_GAP`, `ladderRevealDelay`)을 **모든 토큰이 동시에 출발하는** 연출로 바꾼다.
- 경로 길이(막대기 수·곡선 vtravel)가 레인마다 달라 물리적으로는 도착 시간이 다르다. 사용자가 원하는 건 **"동시에 도착하는 느낌"** — 모든 토큰이 거의 같은 시각에 결승선(바닥)에 닿도록 **레인별 진행 속도를 정규화**한다(경로가 긴 레인은 빠르게, 짧은 레인은 느리게 → 동시 피니시). 정확한 정규화 방식(총 reveal 시간 고정 + 레인별 경로길이로 속도 스케일)은 구현자가 기존 곡선/속도 코드를 보고 결정.
- 서버의 reveal 종료 타이밍(자동 결과 오버레이) 계산을 새 "동시 도착" 모델에 맞춰 갱신한다. **서버 타이밍과 클라 애니 길이는 반드시 동기화**(기존 주석 "js/ladder.js와 동일 유지" 규약 준수)해, 애니가 끝나기 전에 결과로 넘어가지 않게 한다.
- 결과 매핑(어느 번호가 어느 바닥칸=꽝/순위에 도착하는지)은 **기존 `computeLaneToBottom` 로직 그대로** — reveal 연출 변경은 시각일 뿐 매핑·공정성에 영향 0이어야 한다.

## 공정성
- 결과는 서버에서만 결정, 클라는 시각화. 클라 `Math.random`은 deviceId/tabId 외 0회.
- reveal 전 server-only 정보(숨은 막대기/꽝/번호→바닥 매핑) 비노출, 재진입 응답 마스킹 유지.
- "동시 출발/도착"은 연출 변경일 뿐 → 매핑·승패 결정에 어떤 영향도 주지 않음을 코드/테스트로 보장(곡선 무관성 회귀 테스트가 있으면 함께 통과).

## 기존 통합 유지 (스킵 금지)
- 통계(db/stats, /api/statistics), 랭킹(db/ranking + RankingModule ladder 탭) 계속 동작.
- 튜토리얼(LADDER_TUTORIAL_STEPS), 사운드(ladder_*/common_* 재생 호출) 계속 동작. 동시 출발 reveal에 맞춰 사운드 트리거(출발/도착/결과)가 어색하지 않게 연결.
- 유저 빌드 막대기 메커니즘(배치/제거/소유권/준비취소·퇴장 시 정리)은 그대로 유지.

## 작업 방식
- 먼저 조사: `socket/ladder.js`(상수·`ladderRevealDelay`·매핑), `js/ladder.js`(reveal 애니·토큰 하강·레인 선택 UI), `css/ladder.css`, `ladder-multiplayer.html`(번호/레인 선택 마크업), `utils/room-helpers.js`(ladder gameState), 그리고 비교 대상으로 경마의 번호/탈것 선택 흐름(`js/horse-race.js` 또는 `js/horse.js`, `horse-race-multiplayer.html`).
- `.claude/rules/new-game.md`는 새 게임 절차이므로 신규 등록 14곳은 불필요(기존 게임 수정). 단 게임별 함정은 `docs/GameGuide/lessons/_common.md`와 `docs/GameGuide/lessons/horse-race.md`(번호 선택 UX 참고)·해당 ladder lesson을 읽고 시작.
- 모바일·PC 양쪽 대응(번호 고르기 버튼 크기/배치, 6레인 가로 폭, 동시 하강 가독성)을 계획 단계부터 포함.
- 상수는 `config/` 또는 파일 상단 const 블록에 정의(매직넘버 금지). 서버↔클라 동기 상수는 한쪽 변경 시 반대쪽도.

## 테스트
- `tests/test-ladder.js`(Playwright)를 갱신: 앱 실행 → 2탭 방 생성/입장 → 준비 → (유지된) 막대기 배치 → **번호 1~6 중 선택(중복 비활성 확인)** → 공개 시 **모든 토큰 동시 출발·거의 동시 도착** 관찰 → 결과/종료 도달 → 콘솔 에러 없음.
- 6레인 고정(인원 2~3이어도 1~6 표시 + 빈 레인) 확인. 결과 매핑이 reveal 변경 전후로 동일함(공정성 불변) 확인.
- 통계 집계·랭킹 ladder 섹션·튜토리얼 동작 확인. 기존 대표 게임(경마) 선택/시작/진행 미파손 확인.
- 소켓 변경 후 dev 서버 재시작 후 테스트(node server.js 자동 리로드 없음).

## 완료 기준 (하나라도 미완이면 완료 아님)
- 레인이 항상 1~6 6개로 고정 표시되고, 인원 6 미만이면 빈 레인이 정상 존재한다.
- 참가자가 경마식 UI로 1~6 중 자기 번호를 중복 없이 고를 수 있다(빌드 막대기 메커니즘도 그대로 동작).
- 공개가 "동시 출발 → 동시 도착 느낌"으로 보이고, 서버 종료 타이밍이 애니와 동기화되어 결과로 정확히 넘어간다.
- 결과 매핑/승패가 reveal 변경과 무관하게 기존과 동일(공정성 불변). 클라 Math.random 게임결과 0회, reveal 전 마스킹 유지.
- 통계/랭킹/튜토리얼/사운드 모두 동작. 경마 등 기존 게임 1개 이상 미파손 확인.
- update-log.md에 기존 형식대로 기록(새 리소스 여부 명시 — 기존 사운드 재사용 우선).
- 마지막 보고에 룰 요약, 변경 파일, 테스트 명령/결과, 게임성 자체 평가, 새 리소스 여부, 남은 이슈 포함.

## 막힘 기준
- "동시 도착" 정규화의 정확한 속도 모델(총 reveal 시간 값, 레인별 경로길이 산정·속도 스케일 식)·번호 고르기 UI의 정확한 마크업/스타일·6레인 모바일 폭 등 세부가 불명확하면, 기존 ladder/경마 구조를 조사해 근거와 함께 합리적으로 선택한다.
- 테스트 환경 문제로 검증 불가 시 구현은 완료하되 어떤 명령/단계에서 막혔는지 구체 보고.
