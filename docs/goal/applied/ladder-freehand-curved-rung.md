# 사다리타기 — 막대기 "그림판처럼" 자유 곡선 그리기 (A안) — Goal

사다리타기(ladder) 빌드 단계의 막대기 배치를 **그림판처럼 자유 곡선으로 긋는** 방식으로 확장해라. `feature/goal-game-mode` 브랜치에서 작업한다.

> 전제: 직전까지 완료된 "연속 좌표 막대기(`{c, y, slant}`) + 드래그 자유 배치 + 막대기 위치 고정(이웃에 영향 없음)"이 적용된 상태 위에 얹는다.
> baseline 파일: `socket/ladder.js`, `js/ladder.js`, `ladder-multiplayer.html`, `css/ladder.css`, `tests/test-ladder.js`, `AutoTest/ladder/ladder-multitab-bot.js`.

## 한 줄 요약

막대기를 직선이 아니라 **드래그 궤적 그대로의 자유 곡선**으로 그릴 수 있게 한다. **양끝은 두 인접 기둥에 자동으로 붙고**(사다리 성립), 공개(추적) 화면에서 **토큰이 그 곡선을 따라 구불구불 내려간다**. 곡선 모양은 **순수 시각 장식 — 결과(누가 꽝)는 불변**이다.

---

## 1. 빌드 단계 — 자유 곡선 입력

- **무엇**: 지금은 두 기둥 사이를 드래그하면 직선 막대기 1개가 놓인다. 이걸 **드래그 궤적(포인터가 지나간 경로)을 그대로 곡선으로** 그려지게 한다(그림판 느낌).
- **양끝 스냅(중요)**: 곡선이 아무리 구불거려도 **시작점은 한 기둥, 끝점은 그와 인접한 다른 기둥**에 자동으로 붙는다. 즉 막대기는 항상 "인접한 두 기둥을 잇는 다리"여야 사다리가 성립한다. 어느 두 기둥인지는 드래그가 가로지르는 기둥 구간으로 판정한다.
- **기대**: 준비한 사용자가 두 기둥 사이를 손가락/마우스로 자유롭게 그으면, 그 궤적 모양대로 곡선 막대기가 실시간으로 보이고(본인 진하게, 남은 옅게), 다른 탭에도 같은 모양으로 동기화된다. 내 막대기를 톡 누르면 제거(현 동작 유지).
- **결정/위임**: 곡선 표현 방식(드래그 점들을 정규화해 배열로 저장 vs 베지어 제어점), 점 개수 상한·다운샘플링, 곡선이 기둥 밖으로 튀지 않게 하는 정리(clamp/smoothing)는 가독성과 페이로드 크기를 보고 합리적으로.

## 2. 공개(추적) 화면 — 토큰이 곡선을 따라간다

- **무엇**: 공개 화면에서 토큰이 막대기를 건널 때, 직선이 아니라 **그 막대기의 곡선 궤적을 따라** 이동하게 한다.
- **현재**: `js/ladder.js`의 `buildPath`가 막대기를 만나면 두 점(양끝)으로 폴리라인을 만든다. 곡선 막대기는 그 구간에 **곡선 점들을 삽입**해 토큰이 따라가게 한다. `drawLadderFrame`도 곡선으로 렌더.
- **기대**: 토큰이 막대기 구간에서 곡선을 부드럽게 따라 내려간다. 곡선이 위아래로 출렁여도 전체적으로는 아래로 진행(연출이 자연스럽게). 마지막에 결과 캡션/오버레이는 현행 유지.

## 3. 데이터/동기화 — 모든 탭 동일

- 곡선 막대기 데이터(예: `points` = 정규화 좌표 배열)를 **서버 `ladder:rungsUpdated`(빌드) / `ladder:reveal`(공개) 페이로드에 포함**해 모든 탭이 같은 모양으로 본다. 클라가 곡선을 임의로 다르게 그리지 않는다.
- 서버는 곡선 데이터를 **신뢰경계로 검증**한다: 점 개수 상한(예 ≤ 24~32), 좌표 범위 clamp, 비정상 값 무시. 곡선은 매핑에 쓰지 않으므로 검증 실패 시 직선으로 폴백해도 된다.

---

## 공정성 (필수)

- **결과(패자/매핑/꽝)는 오직 막대기의 "두 기둥(c) + 중심 높이(y)"로만** 서버에서 결정한다. 곡선 모양(`points`)·기울기는 **시각 장식이며 결과에 0 영향**임을 보장해라. 곡선을 아무리 구불구불 그려도 누가 꽝인지는 바뀌지 않는다.
- 매핑은 현행 그대로 — 막대기를 중심 y 오름차순 정렬해 인접 스왑(`buildLadder`). 곡선 점들은 매핑 계산에서 제외.
- 충돌 규칙(같은 기둥 공유 + `|Δy| < LADDER_MIN_GAP_Y` 금지)은 **중심 y 기준**으로 유지.
- 클라 `Math.random`은 deviceId/tabId 외 0회 유지. 곡선 좌표는 사용자의 드래그 입력에서만 나온다.
- reveal 전 server-only(숨은 기본 막대기/꽝/매핑) 비노출·재진입 마스킹 유지. 기본(숨은) 막대기의 곡선은 서버가 생성(직선이어도 무방).
- 직전에 고친 **"막대기 위치가 이웃에 영향받지 않는다"**(자기 위치로 고정)는 회귀 없이 유지.

## 작업 방식

- 먼저 `js/ladder.js`(빌드 캔버스 드래그 핸들러 `bindBuildCanvas`/`computeDragRung`/`drawBuildCanvas`, 공개 `buildPath`/`drawLadderFrame`, 막대기 모델 `buildState.userRungs`/`ladderState.rungs`), `socket/ladder.js`(`addRung` 검증·`buildLadder`·`emitRungsUpdated`·reveal payload), `utils/room-helpers.js`(ladder gameState 모델)를 조사.
- 곡선은 `points`(정규화: x는 두 기둥 사이 0~1 비율, y는 0~1 높이) 배열로 막대기에 덧붙이고, 양끝 점을 두 기둥(0과 1)에 스냅한다. `c`/`y`/`slant`는 매핑·정렬·폴백용으로 유지(곡선 없으면 직선).
- 모바일/PC 양쪽: 터치 드래그로 곡선 입력(`touch-action: none` 유지), 캔버스 좌표 변환은 현 `toCanvas` 패턴 재사용.

## 테스트

- `tests/test-ladder.js`(Playwright) 갱신/확장: 빌드에서 곡선 막대기 배치(드래그 시뮬 또는 노출 헬퍼로 `points` 포함 emit) → 양쪽 동기화, 공개 화면에서 토큰이 곡선 따라 진행(캔버스 표시 유지), 결과/순위 도달, 콘솔 에러 0.
- `AutoTest/ladder/ladder-multitab-bot.js`가 곡선 막대기로 한 판을 끝까지 자동 플레이.
- 공정성 회귀: 같은 (c, y)에 대해 곡선이 달라도 패자/매핑이 동일한지 확인(곡선 무관성). 막대기가 이웃에 영향 없이 고정인지 유지.
- 기존 대표 모드(경마) 채팅·시작·진행 미파손 확인.

## 완료 기준 (하나라도 미완이면 완료 아님)

- 빌드 단계에서 두 기둥 사이를 드래그하면 **궤적 그대로의 곡선 막대기**가 그려지고, 양끝이 두 인접 기둥에 붙는다.
- 곡선 막대기가 모든 탭에 같은 모양으로 동기화된다(서버 페이로드 경유).
- 공개 화면에서 **토큰이 곡선을 따라** 내려간다.
- 곡선 모양과 무관하게 **결과(누가 꽝)는 서버 확정값과 일치**한다(곡선은 시각 장식).
- 막대기는 이웃에 영향받지 않고 제자리에 고정된다(직전 수정 유지).
- 클라 `Math.random` 게임결과 0회, reveal 전 마스킹 유지, 서버가 곡선 데이터를 검증(점 수 상한·좌표 clamp).
- 테스트 봇/스모크 통과, 콘솔 에러 0, 경마 미파손.
- `update-log.md` 기록. 새 리소스가 있으면 함께 기록(가급적 신규 mp3/이미지 없이).
- 마지막 보고에 항목별 변경 요약, 변경 파일, 테스트 명령/결과, 게임성 자체 평가, 새 리소스 여부, 남은 이슈 포함.

## 막힘 기준

- 곡선 표현 방식·점 개수 상한·스무딩·양끝 스냅 판정 등 세부 불명확 시, 기존 구조(연속 좌표 막대기) 조사 후 근거와 함께 합리적으로 선택.
- 토큰의 곡선 추종이 부자연(역행 과다)스러우면, 곡선의 세로 출렁임을 적당히 제한하는 방향으로 최소 변경.
- 테스트 환경 문제로 검증 불가 시 구현은 완료하되 어떤 명령/단계에서 막혔는지 구체 보고.
