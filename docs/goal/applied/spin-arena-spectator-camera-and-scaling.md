# 회전 칼날(spin-arena) — 관전 카메라 + 인원 가변 스케일링(대인원 확장) (office-hours 우선) — Goal

회전 칼날의 참가 상한(현재 준비 선착 6명)을 인원 가변 수치 스케일링으로 확장(16명 검토)하고, 대인원에서 무너지는 모바일 가독성을 **관전 카메라 시스템**(자기 캐릭터 추적 / 이슈 이벤트 컷 / 랜덤 시점 + 미니맵)으로 해소한다. 둘은 한 몸의 기능이다 — 스케일링은 물리·밸런스를 풀고, 카메라는 가독성을 푼다. 시뮬 코어와 관전 모델을 동시에 건드리는 재설계이므로 **바로 코딩하지 말고 `/office-hours`(필요시 `/meeting-codex`)로 설계를 먼저 확정**한 뒤 구현한다. 브랜치: `feature/spin-arena-rework`(또는 후속 브랜치).

> 전제/baseline: 「칼 수집 탈출 + 부활 + 종료 압축」 재설계 적용 상태가 기준이다 (설계 결정서 `docs/dev-cycle/2026-06-12-spin-arena-blade-escape-design.md`, 워킹트리 — 이 goal 시작 전 커밋/배포돼 있어야 함). 현재 시뮬: 받은 데미지 35(BLADE_UP_DMG)당 칼 +1, 칼 5개(ESCAPE_BLADES) = 탈출, 비석+3초 부활, decideMs 종료 압축, frames `[x,y,cumDmg]`, 페이로드에 `arena/ring/escapes/downs/bladeUps/decideMs`. 기준 파일: `socket/spin-arena.js`, `js/spin-arena.js`, `css/spin-arena.css`, `spin-arena-multiplayer.html`, `AutoTest/spin-arena-*`.

## 한 줄 요약

① **인원 가변 스케일링** — 캐릭터/칼날/링/칼업 임계 등을 인원 n의 함수로 산출(예: s=√(6/n))하고 서버가 reveal 페이로드에 실어 클라가 그대로 따름(상수 미러 → 페이로드 단일 권위) ② **참가 상한 확대** — 16명 검토(최종 상한은 Phase 0에서 가독성·재미 기준으로 확정, 10명 후퇴안 포함) ③ **관전 카메라** — 자기 캐릭터 추적 / 이슈 이벤트 컷(다운·탈출·칼 4개 임박·결판) / 랜덤(로밍) 시점 + 전체 미니맵, 뷰어별 로컬 ④ **"2탭 동일" 불변조건 재정의** — 시뮬 상태·이벤트·결과는 모든 탭 동일, 카메라 시점만 뷰어 로컬 ⑤ **착수 전 office-hours로 설계 확정**.

## 0. 작업 순서 — 설계 먼저 (스킵 금지)

- **Phase 0 (필수): `/office-hours` 빌더 모드**로 아래 열린 질문을 결론내고 **설계 결정서**를 `docs/dev-cycle/YYYY-MM-DD-spin-arena-camera-scaling-design.md`로 남긴다. 필요하면 `/meeting-codex`로 보강. 직전 blade-escape 결정서처럼 적대적 스펙 리뷰를 거쳐 확정한다.
- Phase 0에서 "16명이 카메라로도 관전 재미가 안 산다"는 결론이 나오면 상한을 낮춰(예: 10명) 진행하고 근거를 보고한다 — 16은 목표가 아니라 검토 대상이다.
- Phase 1(구현)은 결정서 확정 후에만. 트리아지 **COMPLEX**(시뮬 수치 함수화 + 페이로드 + 렌더 파이프라인 + 밸런스 재검증).

## 1. 인원 가변 스케일링 (물리·밸런스)

- 수치를 상수가 아닌 **n의 함수**로: 축소 계수 s(n)=√(6/n) 제안(밀도 보존 — n≤6은 s=1로 현행 수치 동결, 검증된 밸런스 보존). 대상: `CHAR_RADIUS`/`BLADE_RADIUS`/`SWORD_LEN`/`BLADE_EDGE_R`(기하), 초기 배치 반경(스폰 즉시 피격 방지 — 간격 > 2×칼날+여유), `RING_R_END`(막판 과밀 방지), `BLADE_UP_DMG` T(n)(밀도 상승으로 데미지 적산이 빨라짐 — 결판률 게이트 유지용 곡선).
- **페이로드 단일 권위**: 산출된 기하·임계를 reveal 페이로드에 실어(기존 `arena`/`ring`/`bladeRadius` 확장) 클라가 페이로드 값만 사용 — "서버/클라 상단 상수 동일 값" 수동 동기 부담을 줄이는 방향. 클라 미러 함수(bladeCountAt, 진행 바 공식)도 페이로드 값 기준으로.
- 스킨: 상한만큼 프리셋 확장(16이면 16종 — 소형 스케일에서 색 구분 가능성을 Phase 0에서 검증, 색+무늬/테두리 조합 검토 위임).
- **열린 질문(Phase 0 확정)**: 최종 상한(10 vs 16), s(n) 곡선과 T(n) 곡선, n≤6 동결 여부, MAX_SLOTS 초과 시 관전 규칙(현행 유지) — 전부 200시드 배치 분포가 권위.

## 2. 관전 카메라 (가독성 — 뷰어별 로컬)

- **모드 3종 + 자동 디렉터**:
  - **내 캐릭터 추적**: 참가자 기본값. 내 슬롯 중심으로 줌(배율 Phase 0 확정, 예: 논리 480 중 ~240 영역). 내가 탈출/다운하면 디렉터 모드로 자동 전환.
  - **이슈 이벤트 컷(디렉터)**: 관전자·다시보기 기본값. 우선순위 컷 — 결판(decideMs) > 탈출 > 다운/부활 > 칼 4개 임박 > 최다 교전 클러스터. 이벤트는 전부 페이로드(escapes/downs/bladeUps/decideMs) t 파생이라 컷 타이밍은 결정론으로 산출 가능.
  - **랜덤(로밍)**: 이벤트가 없는 구간을 채우는 시점(t 해시 파생 또는 클러스터 추적 — Phase 0에서 택1).
  - 수동 전환 UI(탭/버튼: 내 캐릭터 ↔ 전체 보기 ↔ 디렉터)와 **전체 보기(현행 고정 뷰)** 폴백은 반드시 유지.
- **미니맵**: 구석에 전체 아레나 축소판(점 + 링) — 카메라 밖 상황 파악. 오프스크린 이벤트는 화면 가장자리 인디케이터(화살표/플래시) 검토.
- **HUD 연동**: 줌 상태에서 핍/이름표가 자연 확대되므로 16명 가독성 벽이 풀리는지 Phase 0에서 목업으로 검증. 미션 패널(칼 k/5)은 카메라와 무관하게 전원 표시 유지.
- 카운트다운 백드롭/결과 오버레이/결판 비트(적색 글로우)와 카메라의 상호작용(결판 시 selected로 강제 컷 등) 정의.
- **열린 질문(Phase 0 확정)**: 줌 배율·전환(컷 vs 팬) 연출, 컷 최소 유지 시간(어지러움 방지), 모바일 제스처(핀치 줌?) 범위, 다시보기 기본 모드.

## 3. "2탭 동일" 불변조건 재정의 (중요)

- 기존 QA 불변조건 "2탭 동일 화면"을 **"2탭 동일 시뮬 상태"**로 재정의한다: frames/이벤트/결과/종료 시점은 모든 탭 동일(서버 결정론), **카메라 시점만 뷰어 로컬**(자기 캐릭터 추적은 정의상 뷰어별).
- 같은 모드·같은 대상이면 카메라도 재현 가능해야 한다(랜덤 로밍 포함 — t 해시 파생 권장, `Math.random` 금지). 디렉터 컷은 전 탭 동일 타임라인.
- 테스트 단언 갱신: 결정론 테스트는 시뮬 산출물 동일성(기존 유지), 렌더 하네스는 "같은 모드+같은 t = 같은 픽셀"로.

## 공정성 (절대 불변)

- 결과는 서버 결정론 시뮬에서만 결정. 카메라·스케일링은 **결과 영향 0** — 스케일 수치는 시드 시뮬 입력(n)의 결정론 함수, 카메라는 순수 시각화.
- 클라 `Math.random` 결과 영향 0(deviceId/tabId 외 0회 — 카메라 로밍도 t 해시 파생).
- reveal 전 server-only(timeline/result/seed/이벤트) 비노출·재진입 마스킹 유지. 페이로드 변경은 additive 우선(기하 필드 추가).

## 기존 통합 유지 (스킵 금지)

- 통계(recordGamePlay)/서버 랭킹/사운드/스킨/튜토리얼/채팅 오버레이/카운트다운/BGM/다시보기/주문 계속 동작. selected=DB rank 2/isWinner=false 시맨틱 유지.
- 직전 재설계 산출물(칼 수집 탈출/부활/decideMs 종료 압축/핍 HUD/미션 패널/스포일러 가드 decidedNow) **무파손** — 카메라 줌 하에서도 스포일러 가드(결판 전 selected 비식별) 유지.
- 소켓 이벤트 이름 유지. 튜토리얼·메타·update-log 문구를 새 규칙(인원 확대/카메라)에 맞게 갱신.

- 서버·클라 공유 수치는 페이로드 전달로 일원화(1절). 캐시버스트(`js/spin-arena.js?v=`) 상향. **소켓 변경 → dev 서버(5173) 재시작 필수.**
- 모바일·PC 양쪽: 카메라/미니맵/수동 전환 UI의 모바일 레이아웃을 계획 단계부터.

## 완료 기준 (하나라도 미완이면 완료 아님)

- Phase 0 설계 결정서 존재(상한·s(n)/T(n) 곡선·카메라 모드 정책·줌 배율·미니맵 형태 전부 결론 + 근거). 채택 시에만 Phase 1.
- 확정 상한 인원으로 한 판이 정상 동작(스폰 즉시 피격 없음, 결판률 게이트 충족, 스킨 전원 식별).
- 카메라 3모드 + 수동 전환 + 전체 보기 폴백 + 미니맵 동작. 참가자 기본 = 내 캐릭터 추적, 관전자/다시보기 기본 = 디렉터.
- "2탭 동일 시뮬 상태" 불변조건 충족(시뮬·결과·종료 동일, 시점 로컬). 스포일러 가드(결판 전 selected 비식별) 카메라 하에서도 유지.
- 직전 재설계 기능(칼 수집/부활/종료 압축/핍 HUD) 회귀 0. 결정론·공정성·마스킹 불변.
- 확장 200시드 배치 게이트 충족(결정서가 정한 수치). 모바일(375px)에서 카메라·미니맵·패널 가독.
- 튜토리얼/메타/문구 갱신(평이한 한국어). update-log.md 기록. 새 리소스(스킨 확장 이미지·사운드) 여부 명시.
- 마지막 보고: 설계 결론·변경 파일·테스트 명령/결과·밸런스 수치·자체 평가·남은 이슈.

## 막힘 기준

- Phase 0에서 "16명은 카메라로도 관전 재미/식별이 안 산다" → 상한 하향(10 권장)으로 진행하고 근거 보고. 상한·곡선 수치는 배치 분포가 권위 — 기준 충족하는 가장 단순한 조합 채택.
- 카메라 뷰포트 변환이 기존 fx/줌/흔들림 구조와 충돌해 과대 작업이 되면: 1차로 "전체 보기 + 미니맵 + 이벤트 인디케이터"까지로 축소하고 카메라 추적은 후퇴 보고.
- 스킨 16종 식별이 색만으로 불가하면 색+테두리/무늬 조합으로, 그래도 불가하면 상한 하향 근거로 보고.
- 페이로드 기하 일원화가 기존 클라 의존부를 광범위하게 깨면: 신규 수치만 페이로드로, 기존 상수는 유지하는 절충으로 후퇴하고 보고.
- 테스트 불가 시 구현은 완료하되 막힌 지점 + 수동 QA 체크리스트.
