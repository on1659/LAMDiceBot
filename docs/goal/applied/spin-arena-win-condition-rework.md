# 회전 칼날(spin-arena) 승리 조건 구조 — 플레이테스트 피드백 검토 + 채택분 구현 — Goal

플레이테스트 피드백("마지막 생존자가 벌칙 = 역인센티브 구조" + 개선안 3개)을 받아, `feature/spin-arena-rework` 브랜치에서 **먼저 검토·평가하고, 타당하다고 판단된 범위만 구현**한다. 피드백을 100% 그대로 적용하는 작업이 아니다 — 검토 결과 기각/변형/보류가 모두 가능하다.

> 전제/baseline: `feature/spin-arena-rework` 브랜치. 현재 spin-arena는 커밋 `4938fc5`(결정론 생존전) 기반 + 워킹트리에 미커밋 수정 있음. 기준 파일: `socket/spin-arena.js`, `js/spin-arena.js`, `spin-arena-multiplayer.html`, `css/spin-arena.css`.

## 한 줄 요약

① 피드백의 전제·제안 3개를 현 구조 대비 검토하고 결정서 작성 → ② 채택분만 구현 (High 우선순위 = 승리 조건 구조) → ③ 결정론·공정성 불변조건은 유지.

## 원본 피드백 (검토 대상)

- **핵심 문제 제기**: 마지막 생존자가 벌칙 → "살아남는 게 불리한" 역인센티브 → 일부러 빨리 죽는 전략이 최적해 → 긴장감 붕괴. 실제 테스트에서 최후 생존자가 "억울하다" 반응.
- **안 1 (High)**: 탈출봉 수집 — 아이템 5개 이상 획득 시 탈출 성공, 마지막까지 탈출 못 한 플레이어가 벌칙.
- **안 2 (Mid)**: 아이템 강탈 + 수집 — 전원 3개 보유로 시작, 빼앗아 5개 먼저 채우면 탈출 (수집 + PvP).
- **안 3 (Low)**: 물리 기반 충돌 이동 — 초기 방향 랜덤, 벽 튕기기, 플레이어 충돌 반사, 이동속도 랜덤.
- **우선순위 제안**: 🔴 승리 조건 구조 변경(탈출 방식) > 🟡 아이템 수집/강탈 > 🟢 속도 랜덤/물리 충돌.

## 1. 검토 단계 (구현보다 먼저, 스킵 금지)

- **전제 검증부터**: 현재 spin-arena는 **플레이어 입력이 없는 결정론 관전형**이다 — 서버가 시드 PRNG로 30초를 사전 시뮬레이션하고 클라는 리플레이만 한다 (`socket/spin-arena.js` 상단 주석, `simulate()`). 플레이어 소켓 이벤트는 스킨 선택/시작뿐. 따라서 피드백의 "일부러 빨리 죽는 전략이 최적해"는 **메커니즘상 성립하지 않는다**. 검토에서 두 층위를 구분할 것:
  - (a) 전략적 역인센티브 — 현 구조에선 부존재 (조작 불가)
  - (b) 감정적 프레이밍 역전 — "내 캐릭터가 끝까지 버텼는데 벌칙"이라는 억울함은 실재하는 UX 문제. 안 1의 "탈출 성공 = 안전, 탈출 실패 = 벌칙" 프레이밍이 이를 해소하는지 평가.
- **/office-hours 사용 허용**: 빌더 모드로 대안 브레인스토밍에 활용해도 된다 (필수는 아님).
- 안 1~3 각각을 **채택 / 변형 채택 / 기각 / 보류**로 분류하고 근거를 적은 **검토 결정서**를 `docs/dev-cycle/2026-06-11-spin-arena-win-condition-review.md`로 남긴다.
- 평가 시 고려: 안 1·2·3 모두 "무입력 결정론 시뮬" 안에서 관전형으로 구현 가능한지(아이템 자동 수집/강탈도 시뮬 연출로 가능), 라운드 길이(현 30초)와 긴장감, 기존 칼날 메커니즘과의 공존 여부.
- **bridge-cross 교훈 적용**: 메커니즘을 통째로 갈아엎지 말 것. 칼날 생존전이라는 핵심 정체성은 유지하면서 승리 조건/프레이밍을 손보는 방향을 우선 탐색한다.

## 2. 구현 단계 (검토에서 채택된 범위만)

- High(승리 조건 구조)가 채택되면 그것부터. Mid/Low는 검토 결과에 따라 이번 범위에 포함하거나 보류로 명시.
- **스코프 가드**: 채택안이 "플레이어 실시간 입력 추가"를 필수로 요구한다는 결론이면 구현하지 말고 막힘 보고로 전환한다 (관전형 → 조작형 전환은 이 goal의 범위를 벗어나는 구조 변경).
- 세부 수치(아이템 개수, 스폰 규칙, 탈출 연출 등)는 검토 결정서에서 정한 값을 따르되, 미정 항목은 구현자가 기존 상수 패턴(`socket/spin-arena.js` 상단 const 블록)에 맞춰 합리적으로 정하고 근거를 남긴다.

## 공정성

- 결과는 서버에서만 결정(시드 PRNG 결정론 시뮬), 클라는 시각화/리플레이만. 클라 Math.random은 deviceId/tabId·외관 효과 외 0회.
- reveal 전 server-only 정보(timeline/result/seed) 비노출 유지 — 현재 skins만 보내는 패턴(`socket/spin-arena.js`의 reveal 전 emit) 보존. 재진입 마스킹 유지.
- 승리 조건이 바뀌어도 결정론 보장(같은 시드 → 같은 결과)은 깨지 않는다 — `AutoTest/spin-arena-determinism-test.js`로 검증.

## 기존 통합 유지 (스킵 금지)

- 통계(recordGamePlay)/서버 랭킹(recordServerGame, recordGameSession)/사운드/스킨 선택이 계속 동작.
- gameEnd의 `selected`(당첨 = 벌칙) 의미가 바뀌면 DB 기록부(`socket/spin-arena.js` 후반)와 클라 결과 오버레이·랭킹 표시를 함께 정합시킨다.

## 작업 방식

- 먼저 읽을 것: `socket/spin-arena.js`(simulate + 결판 규칙 + rankings), `js/spin-arena.js`(리플레이/연출), `docs/goal/applied/spin-arena*.md` 4건(직전 결정 맥락), `docs/GameGuide/lessons/_common.md`.
- 모바일·PC 양쪽 화면 대응을 계획 단계부터 포함 (아이템/탈출 연출 추가 시 캔버스 스케일 확인).
- `socket/*` 수정이므로 트리아지 최소 STANDARD, 사실상 COMPLEX 예상.

## 테스트

- `node -c socket/spin-arena.js js/spin-arena.js server.js` 문법 체크.
- `AutoTest/spin-arena-determinism-test.js` (결정론), `AutoTest/spin-arena-2tab-test.js` (2탭 동기화) 통과 — 룰 변경에 맞게 테스트 기대값 갱신 허용.
- 소켓 변경 후 dev 서버 재기동 필수 (자동 리로드 없음).
- 수동 체크: 2탭에서 게임 시작 → 새 승리 조건대로 결과/랭킹/히스토리 표시 → 재진입 마스킹.
- 대표 게임(경마) 미파손 확인 — 공유 모듈을 건드린 경우에만.

## 완료 기준 (하나라도 미완이면 완료 아님)

- 검토 결정서(`docs/dev-cycle/2026-06-11-spin-arena-win-condition-review.md`)가 존재하고, 안 1~3 + 우선순위 각각에 채택/변형/기각/보류 + 근거가 적혀 있다.
- 채택분이 구현되어 위 테스트를 통과한다 (전부 기각이면 결정서 + 사용자 보고로 완료).
- 결정론·공정성·reveal 마스킹 불변조건이 유지된다.
- update-log.md 기록. 새 리소스(사운드/이미지) 필요 여부 명시.
- 마지막 보고에 검토 결론 요약·변경 파일·테스트 명령/결과·자체 평가·남은 이슈(보류 항목 포함) 포함.

## 막힘 기준

- 검토 결과 "관전형 유지로는 피드백의 핵심을 해소할 수 없고 실시간 조작 전환이 필요하다"는 결론 → 구현하지 말고 결정서 + 선택지를 사용자에게 보고.
- 채택안 사이에서 게임 정체성이 갈리는 판단(예: 칼날 제거 수준의 변형) → 사용자 결정 요청.
- 세부 수치 불명확 → 기존 상수/유사 게임 조사 후 근거와 함께 합리적으로 선택.
- 테스트 불가 시 구현은 완료하되 어디서 막혔는지 구체 보고.
