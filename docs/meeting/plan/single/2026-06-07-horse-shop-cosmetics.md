## 문제 정의 (Problem Statement)

경마 게임에 **상점 버튼 → 상점에서 물건 구매** 기능을 넣고 싶다.
단, 구매 품목은 **게임 플레이에 0 영향**, 오직 꾸미기/스킨이다.
핵심 질문은 "뭘 팔지"였고, 이 세션에서 카탈로그와 화폐 모델을 확정했다.

## 무엇이 쿨한가 (What Makes This Cool)

경마는 이미 **개인 자산(탈것 16종)**과 **방 전체가 보는 무대(레이스 연출)**를 동시에 가진다.
그래서 꾸미기가 두 방향의 욕구를 다 건드린다:

- **개인 표현** — 내 탈것을 황금 말·네온 로켓으로. "내 거"라는 소유감.
- **방 플렉스** — 방장이 산 트랙 테마·결승 폭죽을 방 전체가 체험. "와, 이거 뭐야?" → 바이럴·구매 동기.

이미 깔린 인프라가 상점을 싸게 만든다:
- `users.prefs` JSONB + 크로스서버 영구화 → 장착 상태 저장 그릇이 이미 있음
- `assets/backgrounds/`(배경), SoundManager(사운드), win/lose 스프라이트, commentary 시스템 → 연출 꾸미기 표면이 이미 존재
- 탈것 PNG 16종 → CSS 필터(hue-rotate)만으로 도색 무한 변형 가능 (새 아트 0)

## 선택한 꾸미기 표면 (확정)

1. **탈것 꾸미기** — 개인이 자기 탈것에 적용
2. **레이스 연출 꾸미기** — 방 전체가 보는 무대 연출

(플레이어 닉네임 태그 꾸미기는 이번 범위에서 제외 — 경마 외 게임까지 번지므로 후순위)

## 화폐 모델 (확정)

- **단일 소프트 화폐 "코인" 하나.** 이중화폐(하드/소프트) 안 만든다 — 과설계.
- **1차: 인게임 코인** — 경마 참여/승리로 적립 → 상점에서 소비.
- **미래: 코인 현금 충전(top-up)** — 결제 모듈은 "코인 잔고에 +N" 만 꽂으면 되도록 지금부터 구조를 잡는다.

> ⚠️ **돈은 JSONB에 숫자로 박지 않는다.** 코인 잔고는 감사 추적(거래 로그)이 필요하고,
> 미래 현금 결제는 환불·분쟁 대응을 위해 원장(ledger)이 필수다. 전용 테이블로 간다.

## 전제 (Premises — 사용자 동의 완료)

1. **코인 지갑은 처음부터 현금 충전 가능 구조로.** 화폐는 코인 하나, 잔고는 계정(서버 유저) 단위 영구.
2. **상점 UI는 경마 전용, 코인 지갑은 계정 글로벌.** 코인은 모든 게임에서 벌고, 미래 다른 게임 상점도 같은 지갑을 쓴다.
3. **모든 꾸미기는 공정성 0 영향.** 결과는 서버에서만 결정, 꾸미기는 시각/청각 렌더링만. 현금 충전이 들어와도 **pay-to-win 절대 금지** (must-preserve invariant).
4. **첫 배치는 새 아트 최소.** CSS/canvas로 만드는 ★ 품목부터. 프리미엄 새 탈것 스킨은 2차.

## 검토한 접근 (Approaches Considered)

### 접근 A: MVP — 도색+트레일+연출 1차
- 요약: 코인 지갑 + 경마 상점 버튼 + ★ 품목 4~6개. 새 아트 거의 0.
- 노력: M · 위험: Low
- 장점: 가장 빠른 출시, 구매 루프·코인 적립을 실제로 검증, 개인+방 플렉스 둘 다 맛봄
- 단점: 첫인상이 가벼움(프리미엄 욕망 자극 약함)

### 접근 B: 연출 중심 "방장 플렉스" 상점
- 요약: 방 전체가 보는 연출에 집중 → 바이럴 강함
- 노력: M~L · 위험: Med
- 장점: 구매 동기(노출) 강함
- 단점: "누가 적용하나(방장 vs 개인)" 권한 설계가 추가로 필요

### 접근 C: 풀 카탈로그 상점 (사용자 선택)
- 요약: 위 10개 카테고리 전부 + 프리미엄 새 탈것 + 등급(레어리티) + 미리보기
- 노력: XL · 위험: Med (아트 물량, 경제 밸런싱)
- 장점: 완성형, 처음부터 깊이 있는 수집 욕구
- 단점: 첫 출시를 통째로 늦출 위험, 경제 밸런싱 한 번에 잡기 어려움

## 추천 접근 (Recommended Approach) — 전부 한번에 출시 (사용자 확정)

**풀 카탈로그 10종 + 등급 + 미리보기를 한 번의 작업으로 전부 낸다.** 단계 분할 없음.

핵심 통찰: **"한번에"의 병목은 코드가 아니라 아트 에셋이다.**

- **코드로 한 번에 다 나오는 것 (아트 의존 없음):** `paint`(CSS 필터), `trail`(canvas), `track_theme`(배경 교체), `finish_fx`(canvas), `bib`(CSS), `win_sound`(mp3 키), `caster`(텍스트팩). → 상점 UI·코인 지갑·인벤토리·등급·미리보기까지 포함해 한 작업으로 구현 가능.
- **그림이 준비돼야 출시되는 것 (아트 병목):** `skin_premium`(새 탈것 스킨), `win_emote`(새 포즈/애니). 코딩이 늦는 게 아니라 **에셋 제작이 선행돼야** 함.

→ **"한번에 출시" 성립 조건 = 구현 착수 전에 프리미엄 스킨·이모트의 아트를 먼저 확보하거나, 출시 시점에 들어갈 정확한 품목 수를 아트 제작 가능 범위로 고정한다.**

스키마·UI·코인·인벤토리·등급·미리보기는 처음부터 풀 카탈로그를 수용하게 만든다 (나중에 마이그레이션 없음).
현금 충전(top-up)만 별개 트랙 — 결제 인프라(PG·영수증·환불)는 꾸미기 출시와 분리해서 이후 작업.

## 기술 메모 (구현 시 참조 — 명세 아님)

### 코인 지갑 (계정 글로벌)
- 전용 테이블 권장: `user_coins`(잔고) + `coin_ledger`(거래 로그: 적립/소비/충전, 사유, 시각).
  돈은 JSONB 숫자 금지 — 미래 결제 환불·분쟁 대응 위해 원장 필수.
- **서버 권위:** 모든 적립/소비는 서버에서만. 클라이언트는 잔고를 설정하지 못한다.

### 인벤토리 (소유/장착)
- 소유 목록: `user_cosmetics` 테이블 (현금 결제 대비 정식 테이블 안전).
- 장착 상태: `users.prefs` JSONB의 `equipped` 맵으로 충분 (이미 크로스서버 영구).

### 카탈로그 정의
- `config/horse/cosmetics.json` — id·category·name·price·asset·rarity. (프로젝트 관례: 상수는 config/)

### 공정성 불변조건 (must-preserve)
- 꾸미기 적용은 **순수 클라이언트 렌더**. `calculateHorseRaceResult`는 꾸미기 데이터를 절대 읽지 않는다.
- 정적 검증: cosmetic ID가 결과 계산 경로에 유입되는지 grep 가드.

### 소켓 (구현 시 신규)
- `wallet:get`, `shop:buy`, `shop:equip`. 게임 종료 시 서버가 코인 적립 후 갱신 잔고 emit.
- `socket/*` 변경 + DB + 크로스게임 지갑 + 공정성 → **구현 트리아지 COMPLEX**.

## 과제 (The Assignment)

**코인 1개의 가치를 정하라.** 숫자 3개만 정하면 경제 전체가 따라온다:

- 경마 1판 **참여** 시: **+___ 코인**
- 경마 **승리** 시: **+___ 코인**
- **도색 1개** 가격: **___ 코인**

예시 감각: 참여 +10, 승리 +30, 도색 50 → "두세 판이면 도색 하나" (부담 없는 첫 구매).
이 3개를 정하면 나머지 품목 가격은 비례로 깔린다. 현금 충전가도 이 기준 위에 얹는다.

## 세션에서 관찰한 것

- "게임 플레이에는 연관이 없고, 꾸미기나 스킨 정도" — 처음부터 **공정성 경계를 스스로 그었다.** pay-to-win을 본능적으로 피한 것. 멀티플레이 게임에서 이 감각이 신뢰를 지킨다.
- "추후에 코인을 현금으로도 구매 가능하게 할 계획" — 1차 기능을 짜면서 **3차 수익화까지 내다봤다.** 그래서 "코인 잔고를 JSONB에 박지 말고 원장으로" 같은 구조 결정이 지금 가능해졌다.
- 표면 선택에서 **탈것 + 연출 둘 다** 골랐다 — 개인 소유감과 방 플렉스를 동시에 노린 것. 카탈로그가 풍성해진 이유.
