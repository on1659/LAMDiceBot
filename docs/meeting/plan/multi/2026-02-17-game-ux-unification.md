# LAMDiceBot 팀 회의록

**일시**: 2026-02-17
**주제**: 게임 내 UX 통일 - 주사위와 룰렛/경마 게임의 UX 차이 개선
**참석자**: 기획자, 개발자, QA/UX
**회의 방식**: 3인 멀티에이전트 독립 분석 → 교차 검토 → 합의 도출

---

## 1. 현황 요약

### 현재 상태
- **주사위 게임**: 로비 → 방 생성/입장 → 게임 화면 (3단계 흐름)
- **룰렛/경마 게임**: URL 파라미터로 직접 입장 (로비 없음, 2단계 흐름)
- **UI 구조 차이**: 주사위는 `lobby-section`, `create-room-section`, `game-section` 분리, 룰렛/경마는 단일 `game-section`
- **결과**: 게임 간 사용자 경험이 일관되지 않아 학습 곡선 발생

### 문제점
- 게임마다 다른 진입 방식으로 사용자 혼란
- 방 생성 프로세스 불일치 (주사위: UI 폼, 룰렛/경마: URL 파라미터)
- 준비 시스템, 히스토리 패널, 채팅 UI 등 세부 요소 위치/동작 상이

---

## 2. 독립 분석 결과

### 2-1. 기획자 (A)

#### 제안 1: 통합 로비 시스템 구축 - 우선순위: 상
- **전략적 의의**: 모든 게임에 일관된 진입 경험 제공 → 플랫폼 정체성 강화
- **사용자 가치**: 학습 곡선 제거, 방 목록 조회로 빈 방 생성 방지, 친구 초대 용이
- **이유**: 현재 룰렛/경마는 URL 파라미터 직접 입력으로 일반 사용자 접근 어려움

#### 제안 2: 게임 타입 선택 UI 분리 - 우선순위: 중
- **전략적 의의**: 게임 선택과 모드 선택을 명확히 분리 → 미래 확장성
- **사용자 가치**: 명확한 정보 계층, 새 모드 추가 시 혼란 최소화
- **이유**: 현재는 단일 모드이지만 향후 확장 시 필요

#### 제안 3: URL 기반 직접 입장 유지 (옵션) - 우선순위: 중
- **전략적 의의**: 파워 유저와 외부 링크 공유 시나리오 보존
- **사용자 가치**: 디스코드/카톡에서 링크로 즉시 입장, 로비 로딩 시간 제거
- **이유**: 현재 사용 중인 방식 보존, 기존 사용자 경험 유지

#### 제안 4: 방 생성 프로세스 통일 - 우선순위: 상
- **전략적 의의**: 서버 부하 관리(중복 방 생성 방지), 호스트 vs 게스트 역할 명확화
- **사용자 가치**: 실수로 여러 방 생성 오류 방지, 방 설정 사전 결정 가능
- **이유**: 룰렛/경마는 새로고침마다 신규 방 생성 가능, 호스트 개념 모호

#### 제안 5: 게임 시작 전 대기 화면 UX 통일 - 우선순위: 하
- **전략적 의의**: 대기 시간 동안 게임 규칙 설명, 팁, 광고 노출 기회
- **사용자 가치**: 대기 시간 긍정적 활용, 참가자 수 실시간 확인
- **이유**: 기능적 문제는 아니지만 사용자 참여도 향상 가능

### 2-2. 개발자 (B)

#### 제안 1: 통합 로비 시스템 구현 - 난이도: 중
- **구현 방법**: 룰렛/경마/크레인에 주사위와 동일한 3단계 흐름 추가, `lobby-section` HTML/CSS 공통 모듈 분리
- **영향 파일**: `roulette-game-multiplayer.html`, `horse-race-multiplayer.html`, `crane-game-multiplayer.html`, (선택) `public/js/lobby-shared.js`
- **이유**: HTML 구조는 복사 가능하지만 각 게임 초기화 로직 수정 필요

#### 제안 2: 게임 시작 전 준비 시스템 통일 - 난이도: 중
- **구현 방법**: `socket/shared.js`의 `toggleReady` 확장, 모든 게임에 준비 섹션 추가
- **영향 파일**: `roulette-game-multiplayer.html`, `socket/roulette.js`, `socket/shared.js`, `dice-game-multiplayer.html`
- **이유**: 준비 UI는 복사 가능하지만 룰렛의 즉시 진행 특성과 충돌 가능

#### 제안 3: 게임 기록(히스토리) 패널 표준화 - 난이도: 하
- **구현 방법**: 경마에도 우측 히스토리 패널 추가, CSS는 주사위 `.history-section` 재사용
- **영향 파일**: `horse-race-multiplayer.html`, `socket/horse.js`, (선택) `css/horse-race.css`
- **이유**: HTML/CSS 복사 + 소켓 이벤트 1개 추가로 구현 가능

#### 제안 4: 채팅/이모티콘 UI 일관성 개선 - 난이도: 하
- **구현 방법**: 채팅창 위치를 "게임 섹션 하단 고정" 또는 "우측 패널 통합" 중 하나로 통일
- **영향 파일**: 모든 게임 HTML 파일 (`dice-game-multiplayer.html`, `roulette-game-multiplayer.html`, `horse-race-multiplayer.html`, `crane-game-multiplayer.html`)
- **이유**: CSS 레이아웃 수정만으로 가능, 기능 로직 변경 불필요

#### 제안 5: 방 설정 옵션 표준화 - 난이도: 상
- **구현 방법**: 모든 게임의 방 생성 페이지에 동일한 설정 항목 제공, 게임별 추가 옵션은 별도 섹션
- **영향 파일**: 모든 게임 HTML + `socket/rooms.js`
- **이유**: 각 게임 초기화 로직 전면 수정 필요, DB 스키마 확장 가능성

### 2-3. QA/UX (C)

#### 제안 1: 룰렛/경마 게임에 로비 화면 추가 (3단계 흐름 통일) - 리스크: 중
- **테스트 방법**: 시나리오 테스트(브라우저 뒤로가기 → 로비 복귀), 크로스 브라우저 테스트, 세션 복원 테스트, A/B 테스트
- **UX 개선점**: "왜 주사위는 로비가 있는데 룰렛은 없지?" 혼란 제거, 예상 가능한 네비게이션
- **이유**: 기존 북마크/공유 링크 호환성 처리 필요

#### 제안 2: 주사위 게임의 로비 제거 (2단계 흐름 통일, 역방향) - 리스크: 상
- **테스트 방법**: 비교 사용성 테스트(작업 소요 시간 측정), 오류율 측정, 사용자 인터뷰
- **UX 개선점**: 클릭 횟수 감소, URL 공유만으로 친구 초대
- **이유**: 주사위의 "방 목록 선택" 기능 제거 시 기존 사용자 혼란, 공개 방 탐색 불가

#### 제안 3: 게임 입장 프로세스 시각화 (로딩 화면 통일) - 리스크: 하
- **테스트 방법**: 로딩 시간 시뮬레이션(3G 속도), 에러 시나리오 테스트, 다중 기기 테스트
- **UX 개선점**: "연결 중인지 에러인지" 불명확 해소, 느린 네트워크에서 중복 클릭 방지
- **이유**: CSS/HTML 추가만으로 구현 가능, 부작용 최소

#### 제안 4: 방 나가기 동작 통일 - 리스크: 하
- **테스트 방법**: 동작 일관성 검증(확인 다이얼로그, 이동 경로), 호스트 vs 일반 유저 테스트, 브라우저 뒤로가기 비교
- **UX 개선점**: 예측 가능한 이동 경로, sessionStorage 정리 일관성, 모바일 직관성 향상
- **이유**: `leaveRoom()` 함수 통일만 필요

#### 제안 5: 게임 타입별 일관된 세션 복원 메커니즘 - 리스크: 중
- **테스트 방법**: 새로고침 시나리오 테스트, 세션 스토리지 검증, 탭 복제 테스트, 멀티 디바이스 테스트
- **UX 개선점**: 통일된 네이밍, "새로고침 시 방 유지" 동작 일관성, URL 파라미터 vs sessionStorage 우선순위 충돌 해결
- **이유**: sessionStorage 키 변경 시 기존 세션 무효화, 각 게임 초기화 로직 수정 필요

- **B-5 방 설정 표준화**: **중립** - 장기 확장성 vs 단기 투자 부담

- **C-3 로딩 화면 통일**: **긍정** - 사용자 불안 해소, 품질 인식 개선

- **C-5 세션 복원 통일**: **긍정** - 기술 부채 해소, 안정성 개선

| **13** | 주사위 로비 제거 (2단계 통일) | ❌ 기각 | 부정 | 가능 | 부정(리스크 상) | 하향 평준화, 기존 사용자 혼란, 핵심 경험 손상 |

### 의견 충돌 사항

없음. 모든 채택 항목은 2개 이상 역할의 긍정 평가를 받았으며, 기각 항목(C-2)은 2개 역할(기획자, QA/UX)이 부정 평가.

## 6. 회의록 요약

### 핵심 결정
1. **방향성**: "더 나은 쪽으로 통일" 원칙 - 주사위의 로비 시스템을 다른 게임에 확장 (역방향 제거 기각)
2. **이중 진입 전략**: 로비 시스템 + URL 직접 입장 병행 → 초보자와 파워 유저 모두 만족
3. **단계별 접근**: 낮은 난이도 항목(로딩 화면, 히스토리, 채팅)부터 구현 → 통합 로비 → 복잡한 항목(준비 시스템, 세션 복원)

### 기대 효과
- 신규 사용자 학습 곡선 제거
- 게임 간 교차 플레이 증가 → 체류 시간 증가
- 플랫폼 브랜드 일관성 강화
- 유지보수 효율성 향상 (공통 컴포넌트 관리)
