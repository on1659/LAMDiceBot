### WebSocket Reconnect v2 — 메인 connect 핸들러 rejoin
- **문제**: v1에서 socket.off 제거로 page-load 핸들러를 영구화했지만, 이 핸들러는 페이지 로드 시 sessionStorage에 activeRoom이 있을 때만 등록됨
- **실제 시나리오**: 페이지 로드 → 로비 → 방 입장 → transport close → reconnect 시 page-load 핸들러가 등록되지 않은 상태 → joinRoom 미전송 → grace period 만료 → 방 삭제
- **해결**: 메인 connect 핸들러(항상 등록됨)에 rejoin 로직 추가
  - currentRoomId 존재 + sessionStorage activeRoom 존재 시 → setServerId + joinRoom 전송
  - dice: line 2446 connect 핸들러 끝에 추가 (기존 상태 업데이트 코드 유지)
  - roulette: line 2941 빈 핸들러를 rejoin 로직으로 교체
  - crane: line 2668 빈 핸들러를 rejoin 로직으로 교체
- **page-load 핸들러와 역할 분담**:
  - page-load 핸들러: F5 새로고침 대응 (currentRoomId null, activeRoom 존재)
  - 메인 핸들러: transport close 재연결 대응 (currentRoomId 설정됨, activeRoom 존재)
- horse-app: v1에서 이미 connect 핸들러에 rejoin 추가 완료, 추가 수정 불필요
- Railway ROOM_GRACE_PERIOD=300000 (5분) 설정 완료 (이전 세션)

## 변경 파일 (4개)

- js/horse-race.js: tabId 생성 + 모든 joinRoom/createRoom에 tabId 전송
- roulette-game-multiplayer.html: tabId 생성 + 모든 joinRoom/createRoom에 tabId 전송

## 상세 변경 내역

### 문제
- deviceId는 localStorage → 같은 브라우저의 모든 탭이 동일 값 공유
- 중복접속(_1) 기능이 동작하지 않음 (같은 deviceId → 재연결로 처리 → 구 소켓 강제 종료)
- 다른 탭에서 접속하면 첫 번째 탭이 끊어지는 버그

### 해결: sessionStorage 기반 tabId
- sessionStorage는 탭별 격리 → 같은 탭 새로고침: 유지, 새 탭: 다른 값
- 같은 탭 새로고침 → 같은 tabId → 재연결 (구 소켓 disconnect + 기존 유저 정보 인계)
- 같은 브라우저 새 탭 → 다른 tabId → 중복접속 (_1 접미사)
- 다른 브라우저 → 다른 tabId → 중복접속 (_1 접미사)

### 서버 변경 (rooms.js)
- joinRoom: `isSameDevice` → `isSameTab` (data.tabId vs connectedUserWithSameName.tabId)
- createRoom/joinRoom/getCurrentRoom: socket.tabId = data.tabId 저장
- 재연결 경로에서도 socket.tabId 저장

### 클라이언트 변경 (3개 게임)
- 페이지 로드 시 sessionStorage에 tabId 없으면 랜덤 생성
- getTabId() 함수 추가
- 모든 joinRoom/createRoom emit에 tabId: getTabId() 추가
