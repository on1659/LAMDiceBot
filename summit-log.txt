### [N등 찾기 투표 시스템 — 핵심]
- 게임 룰: 라운드 시작 전 1~N등(N=출전 말 수) 투표 → 호스트 시작 시 서버 가중 랜덤 룰렛 → targetRank 결정 → 슬로우모션·결과창 모두 그 등수 기준
- 공정성:
    * 서버 Math.random() 1회로 winningRank 결정, Fisher-Yates 셔플 1회로 rankOrder 결정
    * io.to(roomId).emit('horseRouletteStart', {winningRank, rankOrder, ...}) — 모든 클라이언트 동일 데이터 broadcast
    * 클라는 배열/값 시각화만 — Math.random() 추가 0회 (deviceId/jitter/comment 외)

이전: 왕복(outbound + return) 2단계, return에서 정확히 1명만 강제 통과. outbound는 정보 누적으로 후순위 색 통과율 ~50%, 1번 색 ~1.5% 편향.

변경:
1. 왕복 제거 → outbound 1회만 진행
2. 매 라운드 활성 색 중 random pick → 당첨 색(winnerColor) 결정
3. 시나리오: winnerColor는 buildPassPath(통과 보장), 다른 색은 buildRandomFailPath(실패)
4. 베팅 정산: winnerColor에 베팅한 모든 사용자 = winners (다수 가능)
5. "활성 색 중에서만" pick하므로 winners.length >= 1 항상 보장 (모두 허탕 없음)

검증: AutoTest/bridge-cross-fairness-test.js (M=2/3/4/6, 총 100,000 라운드)
- 색별 우승 분포 최대 편차 < 2% (균등)
- M=6 마지막 색 16.65% (이전 ~50% → 균등 16.7%)
- survivorPositions === [winnerPos] 정합성 100%
- 룰 위반 0건 (winner 통과 + 다른 색 fail 모두 보장)

## 상세 변경 내역

### 룰렛 현재 구조 문서화
- 서버가 `participants`, `winnerIndex`, `winner`, `spinDuration`, `totalRotation`, `effectType`, `effectParams`를 한 번 결정한다는 흐름 정리
- 클라이언트는 같은 서버 데이터를 받아 같은 최종 각도를 계산하므로 결과가 동일하다는 점 기록
- 현재 최종 확정이 호스트 클라이언트의 `rouletteResult` 전송에 의존한다는 리스크 정리

### 서버 주도 종료 개선 방향
- 후속 구현 필드로 `roundId`, `serverStartedAt`, `displayDurationMs`, `serverEndAt` 제안
- 서버 내부 타이머가 종료 예정 시각에 결과를 확정하고 `rouletteEnded`를 broadcast하는 구조 기록
- `rouletteResult`는 호환용으로 남기되 서버 pending result와 맞지 않는 값은 결과를 바꾸지 못하게 하는 방향 명시

### 구현 전제
- 이번 서밋은 문서 전용 변경이며 게임 코드 동작 변경 없음
- 서버 부팅/브라우저 QA는 생략, Markdown 구조와 diff 정합성만 확인 대상
