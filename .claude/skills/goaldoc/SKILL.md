# /goaldoc — goal 명세 문서 생성기

사용자가 말로 풀어놓은 요청(여러 개여도 됨)을 받아, `/goal`로 바로 실행할 수 있는 **goal 명세 md** 한 개를 `docs/goal/`에 만들고, **첫 실행 프롬프트**를 알려준다.

입력: `$ARGUMENTS` = 만들고 싶은 작업 요청(자연어). 비어 있으면 무엇을 goal로 만들지 한 줄로 물어라.

## 산출물 (이 두 가지를 반드시 낸다)

1. `docs/goal/<슬러그>.md` — 아래 템플릿을 채운 goal 명세 파일.
2. 사용자에게 안내하는 **첫 프롬프트**: `/goal docs/goal/<슬러그>.md 해줘`
   - 생성 파일의 풀 경로(`D:\Work\LAMDiceBot\docs\goal\<슬러그>.md`)도 함께 보여준다.

## 절차

1. **요청 분해**: $ARGUMENTS를 읽고 독립 항목으로 쪼갠다(예: "A 하고 B 하고 C 해줘" → 3개). 의도가 불명확하면 추측하지 말고 한 줄로 물어라 (CLAUDE.md: "Never guess intent").
2. **슬러그 결정**: 주제를 영문 kebab-case로 (예: `ladder-reveal-and-ux-polish`). 게임 관련이면 접두에 게임명을 붙인다(`ladder-...`, `horse-...`). `docs/goal/`에 같은 이름이 있으면 새로 만들지, 갱신할지 물어라.
3. **가벼운 정찰만**: 명세를 정확히 쓰기 위해 관련 실제 파일/경로를 grep·glob로 **짧게** 확인해 참조에 적는다(예: 경마 비교 대상, 공유 모듈, 사운드 키). goal 명세는 세부 결정을 구현자에게 위임하는 문서이므로 **과도하게 조사하지 말 것** — 코드를 고치는 단계가 아니다.
4. **md 작성**: 아래 템플릿으로 `docs/goal/<슬러그>.md`를 Write. 각 요청 항목을 "핵심 규칙"(또는 번호 절)로 옮기고, 애매한 세부는 "막힘 기준"에 위임으로 명시한다.
5. **공정성/멀티플레이어 점검 포함**: 대상이 게임(`*-multiplayer.html`, `js/{game}.js`, `socket/{game}.js`)이면 공정성 섹션(클라 Math.random 0회, 서버 결과 결정, reveal 마스킹)과 "기존 통합 유지(통계/랭킹/튜토리얼/사운드)"를 반드시 넣는다.
6. **첫 프롬프트 안내**: 생성 후 `/goal docs/goal/<슬러그>.md 해줘` 를 그대로 보여주고, 위임으로 남긴 항목이 있으면 "미리 정할 게 있으면 알려달라"고 한 줄 덧붙인다.

## 규칙

- 이 스킬은 **goal 문서 생성까지만** 한다. 절대 구현(코드 수정)으로 넘어가지 마라 — 실행은 사용자가 `/goal`로 한다.
- 경로는 풀(절대) 경로로 보여준다(전역 CLAUDE.md). 단 첫 프롬프트의 `/goal` 인자는 워크스페이스 상대경로(`docs/goal/...`)로 — `/goal`이 그 형식으로 동작한다.
- 회의록/impl 문서 규칙(`.claude/rules/docs.md`)과 충돌하지 않게: goal 문서는 `docs/goal/`에만 만든다(meeting/plan, impl과 별개).
- 새 게임/게임 파일이 대상이면 명세에 `.claude/rules/new-game.md`와 `docs/GameGuide/lessons/` 참조를 넣어라.

## goal 명세 템플릿

```markdown
# <제목> — Goal

<한 문단: 무엇을, 어디(브랜치)에서, 왜.>

> 전제/baseline: <이어가는 작업이면 직전 상태와 기준 파일>

## 한 줄 요약

<여러 항목이면: ① … ② … ③ … 으로 압축>

## 핵심 규칙 (또는 번호 절 1, 2, 3 …)

- <요청 항목별 의도 + 기대 동작 + (필요시) 결정/위임>

## 공정성  ← 게임 대상일 때만
- 결과는 서버에서만 결정, 클라는 시각화. 클라 Math.random은 deviceId/tabId 외 0회.
- reveal 전 server-only 정보 비노출, 재진입 마스킹 유지.

## 기존 통합 유지 (스킵 금지)  ← 게임 대상일 때만
- 통계/랭킹/튜토리얼/사운드가 계속 동작.

## 작업 방식
- 먼저 조사할 파일/비교 대상. 모바일·PC 양쪽 대응을 계획 단계부터.

## 테스트
- 검증 방법(Playwright/수동 체크리스트), 기존 대표 게임(경마) 미파손 확인.

## 완료 기준 (하나라도 미완이면 완료 아님)
- <검증 가능한 체크 항목들>
- update-log.md 기록. 새 리소스 여부 명시.
- 마지막 보고에 변경 요약·파일·테스트 명령/결과·자체 평가·남은 이슈 포함.

## 막힘 기준
- 세부 수치/형태 불명확 시 기존 구조 조사 후 근거와 함께 합리적으로 선택.
- 테스트 불가 시 구현은 완료하되 어디서 막혔는지 구체 보고.

## 참고
- baseline/비교/규칙 문서 경로.
```

## 출력 형식 (사용자에게 보여줄 마지막 메시지)

```
goal 문서를 만들었습니다: D:\Work\LAMDiceBot\docs\goal\<슬러그>.md

<핵심 항목 3~6줄 요약>

실행하려면:
/goal docs/goal/<슬러그>.md 해줘

(위임으로 둔 세부: <있으면 나열>. 미리 정하고 싶으면 알려주세요.)
```
