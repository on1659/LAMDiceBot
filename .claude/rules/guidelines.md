# Coding Guidelines

코드 작업 전 반드시 읽을 것. 모든 코드 수정에 적용.

## 1. Think Before Coding

- 가정을 명시적으로 밝힌다. 불확실하면 물어본다.
- 여러 해석이 가능하면 조용히 하나를 고르지 말고 제시한다.
- 더 간단한 방법이 있으면 말한다. 필요하면 반론을 제기한다.
- **명시적 지시만 실행한다.** 의도를 추측해서 추가 행동하지 않는다.
  - "확인해봐" → 읽고 분석만. 수정하지 않는다.
  - 파괴적 작업(git checkout, git reset, 파일 삭제)은 **절대** 추측으로 실행 금지 → 반드시 확인.

## 2. Simplicity First

- 요청된 것 이상의 기능 추가 금지.
- 한 번만 쓰이는 코드에 추상화 금지.
- 불가능한 시나리오에 대한 에러 핸들링 금지.
- **"나중에 쉽게 가자" 금지.** 분리가 필요하면 지금 분리, 구조가 잘못되면 지금 고친다.
- 200줄을 50줄로 줄일 수 있으면 줄인다.

"시니어 엔지니어가 이걸 보고 과하다고 할까?" 그렇다면 단순화.

## 3. Surgical Changes

- 주변 코드, 주석, 포매팅 "개선" 금지.
- 망가지지 않은 것을 리팩터링 금지.
- 기존 스타일에 맞춘다 (내 취향과 달라도).
- 관련 없는 데드 코드 발견 → 삭제 말고 언급만.
- 내 변경이 만든 미사용 import/변수/함수는 제거.

**테스트**: 변경된 모든 줄이 사용자 요청에 직접 연결되어야 한다.

## 4. Goal-Driven Execution

작업을 검증 가능한 목표로 변환:

- 정적 검증: Grep으로 호출부 시그니처 일치, 위험 패턴 검사
- 서버 검증: `node -c server.js` 문법 체크
- 브라우저 검증: 변경 완료 후 수동 QA 체크리스트 제시
- 공통 모듈 변경 시: 크로스게임 검증 (주사위/룰렛/경마 모두)

다단계 작업은 간략한 계획 제시:
```
1. [단계] → 검증: [확인사항]
2. [단계] → 검증: [확인사항]
```

## 5. Record Before Talk

다음 세션에서도 필요한 정보는 반드시 MEMORY.md에 먼저 기록 후 작업 계속.

기록 대상: 새 코드 패턴/관례, 디버깅 원인과 해결법, 아키텍처 결정, 자주 실수하는 부분.

## 6. Context-First Development

코드 작성 전 반드시:
1. 수정할 파일과 그 파일을 import하는 곳 읽기
2. 기존 게임 핸들러/DB 모듈 등 동일 패턴 참조
3. 이 변경이 다른 곳에 어떤 영향을 주는지 확인

확인 안 했으면 코드 작성 금지.
