# /meeting-team - 지속 팀원 기반 멀티에이전트 회의

## 언제 쓰나 (먼저 읽기)

`/meeting-team`은 **다관점 체크리스트** 용도다. 동일 Opus 모델을 여러 번
호출하므로 "다른 두뇌"가 아니라 "같은 두뇌의 여러 관점"이다. 통찰의
다양성보다는 빠뜨림 방지가 목적.

| 목적 | 쓸 도구 |
|------|--------|
| 진짜 다른 시각의 계획 토론 (Claude vs GPT 충돌) | `/meeting-codex` |
| 실제 분업 실행 (정찰→구현→리뷰→QA, 단계별 누적) | 하네스 (`/build`, `/dev-cycle`) |
| 신규 페이지·기능을 여러 직군 관점에서 빠뜨림 없이 점검 | `/meeting-team` (이 명령) |

**핵심 4명(지민/태준/승호/윤서) 중심으로 가볍게 돌리는 것을 기본으로 한다.**
8명 전원 호출은 사용자가 "전원"을 명시했을 때만.

## 개요
`.claude/meeting-team-profiles.md`에 정의된 팀원 프로필과
`.claude/skills/` 의 역할별 전문 스킬을 함께 로드해
각 팀원의 판단 기준·전문성을 가진 독립 서브에이전트로 병렬 실행.

`/meeting-multi`와 달리 역할이 아닌 **사람**으로 동작함.
2라운드 진행: 1라운드는 독립 의견, 2라운드는 상대 의견을 본 뒤 보강/반박/응답.

---

## Step 1: 프로필 + 스킬 로드 및 회의 설정

1. 다음 파일들을 Read 도구로 동시에 읽어오세요:
   - `.claude/meeting-team-profiles.md` (팀원 프로필 + 모델 등급 + 자동 선별 가이드)
   - `.claude/skills/skill-pd.md`
   - `.claude/skills/skill-planner-research.md`
   - `.claude/skills/skill-planner-strategy.md`
   - `.claude/skills/skill-backend.md`
   - `.claude/skills/skill-frontend.md`
   - `.claude/skills/skill-qa.md`
   - `.claude/skills/skill-ui.md`
   - `.claude/skills/skill-ux.md`

2. **회의 주제 확정**:
   - `$ARGUMENTS`가 있으면 그걸 주제로 사용
   - 비어있으면 AskUserQuestion으로 주제 입력 요청

3. **참여자 자동 선별**:
   - `meeting-team-profiles.md`의 "자동 선별 가이드" 표를 참고
   - 주제를 분석해서 핵심 참여자 + 옵션 참여자 선정
   - AskUserQuestion으로 사용자에게 확정 요청:
     ```
     주제: [주제]
     추천 참여자: [선별된 이름들] (사유: [핵심 키워드 매칭])
     [옵션] 추가 후보: [옵션 인원]
     ```
   - 옵션: "이대로 진행" / "조정 (이름 입력)" / "전원 호출"
   - 단, 사용자가 처음부터 명시한 인원이 있으면 그걸 우선

4. **모델 할당**:
   - 확정된 참여자에 대해 모델 등급 규칙 적용
   - 동일 직군 다수 → 대표만 Opus, 나머지 Sonnet
   - 결과를 1줄로 보고: `참여: 지민(Opus), 태준(Opus), 미래(Sonnet), 윤서(Opus)`

## Step 2: 1라운드 — 독립 의견 (병렬)

선택된 팀원 수만큼 **Agent 도구를 동시에 한 번에** 실행하세요.

각 팀원에 해당하는 스킬 파일 매핑:
- 지민 (PD) → `skill-pd.md`
- 현우 (기획자①) → `skill-planner-research.md`
- 소연 (기획자②) → `skill-planner-strategy.md`
- 태준 (개발자① BE) → `skill-backend.md`
- 미래 (개발자② FE) → `skill-frontend.md`
- 윤서 (QA) → `skill-qa.md`
- 다은 (UI) → `skill-ui.md`
- 승호 (UX) → `skill-ux.md`

Agent 호출 시 `model` 파라미터를 모델 등급 규칙대로 지정 (`opus` 또는 `sonnet`).

각 팀원 에이전트 프롬프트 구조:
```
당신은 [이름]입니다. 다음은 당신의 프로필입니다:
[해당 팀원의 프로필 전체 내용]

## 당신의 전문 스킬셋
[해당 역할의 스킬 파일 전체 내용]

## 프로젝트 컨텍스트
LAMDiceBot - Express + Socket.IO 기반 멀티플레이어 게임 플랫폼
게임 4종: 주사위, 룰렛, 경마, 다리건너기
핵심 가치: 100% 공정성 (서버 측 난수 생성)
기술: Node.js + Express + Socket.IO + PostgreSQL
프론트엔드: 순수 HTML/CSS/JS (프레임워크 없음)

## 회의 주제
[주제]

## ⚠️ 작업 순서 (반드시 이 순서로 실행)

### 1) 컨텍스트 로딩 (먼저 Read)
프로필에 명시된 "읽을 파일"을 Read 도구로 먼저 읽으세요.
주제 관련 게임이 명확하면 해당 게임 파일을 우선 읽고,
모호하면 가장 관련성 높은 1-2개를 골라 읽으세요.
파일을 읽지 않은 채 일반론으로 답하지 마세요.

### 2) 답변 작성

`meeting-team-profiles.md`의 "출력 형식 표준"을 따라 작성:

### 핵심 의견
(자신의 판단 기준에 따른 결론. 3~5줄. 위에서 읽은 파일의 구체적 근거를 인용)

### 리스크
- (구체적 리스크 + 영향 범위, 가능하면 파일 경로 또는 함수명 명시)

### 권장 방향
- (실행 가능한 수준으로 구체적으로)

### 다른 팀원에게 던질 질문 (필수 1개)
@[이름]: (질문 내용)
- 이름은 다음 중에서: [참여자 명단 나열]
- 자신에게 묻지 말 것
- 표면적 인사가 아니라 결정에 영향 주는 핵심 질문
```

## Step 2.5: 1라운드 결과 정리

모든 1라운드 결과 수집 후, 이더가 다음을 정리한 **공유 컨텍스트 블록**을 만듭니다:

```
## 1라운드 요약
[팀원별 핵심 의견 + 리스크 + 권장방향 1줄 요약]

## 1라운드 미해결 질문
@승호 → @태준: ...
@윤서 → @미래: ...
...

## 충돌 지점
- [팀원 A] vs [팀원 B]: ... (요지)
```

## Step 3: 2라운드 — 상대 의견 반영 (병렬)

같은 팀원들을 다시 병렬 호출. 같은 모델 등급 유지.

2라운드 프롬프트 구조:
```
1라운드에서 당신이 작성한 의견:
[1라운드 본인 답변 전체]

## 1라운드 전체 결과
[Step 2.5에서 정리한 공유 컨텍스트 블록 전체]

## 2라운드 임무
1) 자신에게 온 질문(@[당신이름])에 답변
2) 다른 팀원 의견을 보고 자신의 입장을 **수정/강화/철회** 중 하나로 명시
3) 1라운드와 달라진 점이 있다면 명시 (없으면 "1라운드 유지")
4) 충돌 지점에 대한 본인 판단

## 출력 형식

### 받은 질문 답변
@나에게 → @질문자: ...

### 입장 변화
[수정 / 강화 / 철회 / 유지] — 사유

### 최종 권장
- (1라운드와 통합한 최종 액션)

### 충돌에 대한 판단
- [충돌 지점]: 나는 [어느 쪽] — 사유
```

## Step 4: 결과 종합

모든 2라운드 결과를 받은 후 이더가 작성:

```
## 🏁 [주제] — 팀 회의 결과 (2라운드 종료)

### 참여자
[이름(모델) 나열]

### 1라운드 → 2라운드 변화 요약
[입장 변화 있는 팀원만]

### 합의된 방향
[2라운드에서 다수가 동의한 방향]

### 미해결 충돌
[2라운드 후에도 남은 충돌 — 사용자 결정 필요]

### 권장 다음 액션
[구체적 실행 항목 3가지 이내, 담당자 제안 포함]
```

## Step 5: 회의록 저장

`docs/meeting/plan/multi/YYYY-MM-DD-HHmm-{topic-summary}.md`에 저장.

회의록에는 1라운드 + 2라운드 전체 내용 포함.

### Git 인코딩 설정 (커밋 시)
```bash
git config --local i18n.commitEncoding utf-8
git config --local i18n.logOutputEncoding utf-8
```

## Step 6: 구현문서(impl) 생성

채택 항목을 **영문 구현문서**로 분리 생성.

**파일명**: `docs/meeting/impl/{회의록 파일명}-impl.md`

**규칙**:
- impl이 구현의 **source of truth**
- 회의록은 역사 기록이므로 변경하지 않음
- 피드백으로 구현 내용이 바뀌면 impl만 수정
- 구현 완료 시: impl → `docs/meeting/applied/`로 이동

## 팀원 수정 방법

- `.claude/meeting-team-profiles.md` — 팀원 프로필 (직군, 읽을 파일, 판단 기준, Critical Rules)
- `.claude/skills/skill-[역할].md` — 역할별 전문 방법론 (프레임워크, 체크리스트)

두 파일을 함께 수정해야 완전한 팀원 업그레이드가 됩니다.
