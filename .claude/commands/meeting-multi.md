Topic: $ARGUMENTS (default: "현재 단계에서 추가할 기능")

## 지시사항

LAMDiceBot 프로젝트의 7인 전문가를 **독립 에이전트로 병렬 실행**한다.
총 2라운드(독립 제안 + 교차검토) 진행.

---

## 7인 전문가 정의

### 기획팀
- **A1 기획 팀장**: 전략적 방향, 비즈니스 가치, 로드맵, 우선순위 판단
- **A2 기획 사원**: 사용자 경험, 게임 재미, 디테일한 시나리오, 유저 대변

### 개발팀
- **B1 개발 팀장**: 아키텍처, 기술 부채, 확장성, 기술 방향 결정
- **B2 개발 사원**: 실제 구현, Socket.IO 코드, 영향 파일, 빠른 프로토타입

### 디자인팀
- **C UI/UX 디자이너**: 사용성, 시각적 피드백, 반응형 디자인, 모바일 호환
- **D 사운드 디자이너**: 효과음, BGM, 오디오 피드백, 몰입감

### 품질팀
- **E QA 엔지니어**: 테스트 전략, **테스트 방법론**, 공정성 검증 방법

---

## 프로젝트 컨텍스트 (에이전트에게 전달)

```
LAMDiceBot - 멀티플레이어 게임 플랫폼
- 게임: 주사위, 룰렛, 경마, 팀 배정
- 핵심 가치: 100% 공정성 (서버 난수 생성)
- 기술: Node.js + Express + Socket.IO (실시간 WebSocket)
- 보안: Rate Limiting, 입력값 검증, Host 권한 관리
- 주요 파일:
  - server.js (서버)
  - dice-game-multiplayer.html (주사위)
  - roulette-game-multiplayer.html (룰렛)
  - horse-race-multiplayer.html (경마)
  - team-game-multiplayer.html (팀 배정)
  - assets/sounds/ (사운드 리소스)
```

## 1단계: 현황 파악

- README.md, CHANGELOG.md 읽고 현재 상태 확인
- 4개 게임 HTML 파일 상태 파악
- assets/sounds/ 사운드 현황 확인
- 현황 요약 작성 (`statusSummary`)

---

## 2단계: 7 에이전트 병렬 독립 제안

**CRITICAL: 7개 에이전트를 반드시 하나의 메시지에서 병렬 Task 호출로 실행할 것.**

각 에이전트에게 `statusSummary`, topic, 프로젝트 컨텍스트를 전달한다.

### Agent A1: 기획 팀장
- 전략적 관점에서 3-5개 제안
- 각 항목: 기능명, 전략적 의의, 비즈니스 가치(상/중/하), 우선순위

### Agent A2: 기획 사원
- 사용자 관점에서 3-5개 제안
- 각 항목: 기능명, 상세 설명, 게임 시나리오, 재미 요소(상/중/하)

### Agent B1: 개발 팀장
- 아키텍처 관점에서 3-5개 제안
- 각 항목: 기능명, 아키텍처 영향, 구현 난이도(상/중/하), 기술 부채 영향

### Agent B2: 개발 사원
- 구현 관점에서 3-5개 제안
- 각 항목: 기능명, 구체적 구현 방법, 영향 파일, 구현 가능성(상/중/하)

### Agent C: UI/UX 디자이너
- UX 관점에서 3-5개 제안
- 각 항목: 기능명, UX 개선점, 인터랙션 디자인, 모바일 호환성

### Agent D: 사운드 디자이너
- 사운드 관점에서 3-5개 제안
- 각 항목: 기능명, 사운드 효과 제안, 적용 시점, 몰입감 영향(상/중/하)

### Agent E: QA 엔지니어
- 품질 관점에서 3-5개 제안
- 각 항목: 테스트 항목명, **테스트 방법 제안**, 자동화 가능 여부, 리스크(상/중/하)

## 3단계: 7 에이전트 병렬 교차검토

**CRITICAL: 7개 에이전트를 반드시 하나의 메시지에서 병렬 Task 호출로 실행할 것.**

- Agent A1: 기획 팀장이 다른 팀 제안에 전략적 가치 평가
- Agent A2: 기획 사원이 다른 팀 제안에 사용자 경험 평가
- Agent B1: 개발 팀장이 다른 팀 제안에 아키텍처 영향 평가
- Agent B2: 개발 사원이 다른 팀 제안에 구현 가능성 평가
- Agent C: UI/UX가 다른 팀 제안에 UX 영향 평가
- Agent D: 사운드가 다른 팀 제안에 오디오 관점 평가
- Agent E: QA가 모든 제안에 **"어떻게 테스트할 것인가"** 방법론 제시

---

## 4단계: 합의 도출

7개 역할의 의견을 종합하여:

- 3개 이상 역할이 긍정 평가 → 우선순위 상승
- 2개 이상 역할이 부정 평가 → 보류 검토
- 의견 충돌 → 명시적 기록
- 최종 판정: 채택/보류/기각

---

## 규칙

1. 사용자에게 질문하지 않는다. README.md 기준으로 자율 판단.
2. 병렬 에이전트는 반드시 하나의 메시지에서 동시 Task 호출한다.
3. QA는 리스크 평가가 아닌 **"어떻게 테스트할 것인가"** 방법론에 집중한다.
4. 한국어로 응답한다.
