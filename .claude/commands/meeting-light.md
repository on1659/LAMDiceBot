# /meeting-light - 1인 순차 팀 회의 (경량 버전)

Topic: $ARGUMENTS (default: "현재 단계에서 추가할 기능")

## 용도

빠르게 아이디어를 정리할 때 쓰는 경량 회의 명령이다.

풀 하네스 리뷰, 역할별 스킬 검토, PD 게이트, impl 문서 생성을 원하면 `/meeting`을 사용한다.

## 지시사항

LAMDiceBot 프로젝트의 7인 전문가 관점을 **1명이 순차적으로 분석**한다.
Task 에이전트 호출 없이 직접 수행하는 경량 버전이다.

## 7인 전문가 관점

### 기획팀
- **A1 기획 팀장**: 전략적 방향, 비즈니스 가치, 로드맵, 우선순위 판단
- **A2 기획 사원**: 사용자 경험, 게임 재미, 디테일한 시나리오, 유저 대변

### 개발팀
- **B1 개발 팀장**: 아키텍처, 기술 부채, 확장성, 기술 방향 결정
- **B2 개발 사원**: 실제 구현, Socket.IO 코드, 영향 파일, 빠른 프로토타입

### 디자인팀
- **C UI/UX 디자이너**: 사용성, 시각적 피드백, 반응형 디자인, 모바일 호환
- **D 사운드 디자이너**: 효과음, BGM, 오디오 피드백, 몰입감

### 품질팀
- **E QA 엔지니어**: 테스트 전략, 테스트 방법론, 공정성 검증 방법

## 작업 흐름

1. 관련 README, GameGuide, 최근 meeting 문서를 필요한 만큼만 확인한다.
2. 7인 관점에서 관점당 2-3개 제안을 작성한다.
3. 채택/보류/기각을 명시한다.
4. `docs/meeting/plan/single/YYYY-MM-DD-HHmm-{topic-summary}.md`에 저장한다.
5. 구현 가능한 채택 항목이 있으면 `docs/meeting/impl/{회의록 파일명}-impl.md`를 별도로 생성한다.

## 출력 형식

```markdown
# LAMDiceBot 팀 회의록 (Single)

**일시**: {날짜}
**주제**: {topic}
**참석자 (관점)**:
- 기획팀: 기획 팀장, 기획 사원
- 개발팀: 개발 팀장, 개발 사원
- 디자인팀: UI/UX 디자이너, 사운드 디자이너
- 품질팀: QA 엔지니어
**회의 방식**: 1인 순차 분석 (경량 버전)

---

## 1. 현황 요약

## 2. 관점별 분석 결과

## 3. 테스트 전략 (QA 주도)

## 4. 합의 도출

## 5. 다음 단계 (Action Items)
```

## 규칙

1. 사용자에게 질문하지 않는다. 로컬 문서와 코드 기준으로 자율 판단한다.
2. 코드 수정은 하지 않는다. 문서 산출물만 만든다.
3. QA는 "어떻게 테스트할 것인가" 방법론에 집중한다.
4. 한국어로 응답한다.
5. 자동 커밋/푸시는 하지 않는다. 사용자가 요청하면 `/summitdocs`로 넘긴다.
