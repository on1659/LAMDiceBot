# Review - Iterative Code Review Until Clean

변경된 코드를 **문제가 없을 때까지** 여러 관점에서 반복 검토한다. 문제 발견 시 즉시 수정하고 처음부터 다시 검토한다. 최대 5사이클.

## Step 0: 변경사항 파악

```bash
git diff --name-only
git diff --cached --name-only
```

변경된 파일 목록과 diff 내용을 읽어 **무엇이 바뀌었는지** 파악한다. 변경 내용의 의도를 요약한다.

## Step 1: 5가지 관점 순차 검토

아래 5개 관점을 순서대로 검토한다. 각 관점에서 문제가 발견되면 **즉시 수정**하고 Step 1을 처음(관점 1)부터 다시 시작한다.

### 관점 1: 정확성 (Correctness)

변경 코드가 의도대로 동작하는가?

- **로직 흐름**: 조건문, 분기, 반환값이 올바른지
- **타이밍/순서**: async/await, Promise 체인에서 실행 순서가 보장되는지
- **데이터 흐름**: 변수가 올바른 값을 갖고 올바른 곳에 전달되는지
- **엣지 케이스**: null/undefined, 빈 배열, 동시 호출 등

### 관점 2: 스코프 (Scope)

변경 코드가 올바른 컨텍스트 안에 있는가?

- **async/await 스코프**: `await`가 반드시 `async` 함수 안에 있는지
- **변수 스코프**: 클로저, 블록 스코프가 의도대로인지
- **this 바인딩**: 콜백/화살표 함수에서 this가 올바른지

검증 방법: 변경된 파일에서 `await` 키워드가 있는 줄을 찾고, 해당 줄이 async 함수 내부인지 확인.

### 관점 3: 동일 패턴 누락 (Missing Same Pattern)

같은 문제/패턴이 다른 파일에도 존재하는가?

- 변경의 핵심 패턴을 식별 (예: "recordServerGame에 await 추가")
- Grep으로 동일 패턴이 다른 파일에도 있는지 검색
- 같은 수정이 필요한 곳이 있으면 함께 수정

검증 방법: 변경에서 수정한 함수명/패턴을 프로젝트 전체에서 grep하여 미수정 지점 확인.

### 관점 4: 잔여 참조 (Stale References)

삭제/변경한 것을 아직 참조하는 곳이 없는가?

- **삭제한 함수/변수**: 다른 파일에서 아직 호출하는 곳이 없는지
- **이름 변경**: 이전 이름을 사용하는 곳이 남아있지 않은지
- **삭제한 HTML 요소**: JS에서 getElementById 등으로 접근하는 코드가 남아있지 않은지

검증 방법: 삭제/변경한 식별자를 프로젝트 전체에서 grep.

### 관점 5: 부작용 (Side Effects)

변경이 다른 기능에 영향을 주는가?

- **호출 체인 추적**: 변경한 함수를 호출하는 곳 → 그 함수를 호출하는 곳 (2단계)
- **공유 상태**: 변경한 코드가 수정하는 전역/공유 변수가 다른 곳에서도 읽히는지
- **이벤트 순서**: emit 순서 변경이 클라이언트에 영향을 주는지

## Step 2: 문제 발견 시 처리

문제를 발견하면:
1. 문제 내용과 영향 범위를 보고
2. **즉시 수정**
3. 수정 내용을 보고
4. **Step 1을 관점 1부터 다시 시작** (새 수정이 새 문제를 만들 수 있으므로)

최대 5사이클까지 반복. 5사이클 후에도 문제가 남으면 해결 못한 이슈 목록과 함께 종료.

## Step 3: 최종 보고

모든 관점을 통과하면 다음 형식으로 보고:

```
## Review Results

### Round N: [관점명]
- 검토 대상: (파일 목록)
- 결과: ✅ PASS / ❌ FAIL
- (FAIL 시) 발견: [문제 설명]
- (FAIL 시) 수정: [수정 내용]

### Final: ✅ PASS (N사이클, M건 수정) / ❌ UNRESOLVED (미해결 N건)
```
